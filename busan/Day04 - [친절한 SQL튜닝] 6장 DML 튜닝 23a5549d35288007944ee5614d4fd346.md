# Day04 -  [친절한 SQL튜닝] 6장 DML 튜닝

<aside>
🔖

대량 DML은 **“묶음 처리 + Direct Path 또는 파티션 교체”** 로 I/O를 확 줄이고, 동시 사용자 환경에서는 **락 충돌**을 최소화하도록 트랜잭션 범위와 채번 방식을 설계하라.

## **📌** 공통 준비 - 간단한 실습 환경

```sql
-- ① 학습용 스키마 생성(권한이 있을 때)
-- CREATE USER DMLTEST IDENTIFIED BY pw;
-- GRANT CONNECT, RESOURCE TO DMLTEST;

ALTER SESSION SET CURRENT_SCHEMA = DMLTEST;

-- ② 학생/점수 테이블
CREATE TABLE students (
    student_id NUMBER CONSTRAINT pk_students PRIMARY KEY,
    name       VARCHAR2(50),
    dept       VARCHAR2(30)
);

CREATE TABLE scores (
    student_id NUMBER CONSTRAINT fk_scores_students
                REFERENCES students(student_id),
    subject    VARCHAR2(20),
    score      NUMBER,
    CONSTRAINT pk_scores PRIMARY KEY (student_id, subject)
);

-- ③ 기초 데이터 5명 × 2과목
INSERT INTO students VALUES (1, '김하나', '컴퓨터');
INSERT INTO students VALUES (2, '박둘',   '경영');
INSERT INTO students VALUES (3, '이셋',   '컴퓨터');
INSERT INTO students VALUES (4, '최넷',   '디자인');
INSERT INTO students VALUES (5, '오다섯', '경영');

INSERT INTO scores VALUES (1, 'SQL', 88);  INSERT INTO scores VALUES (1, 'C',   95);
INSERT INTO scores VALUES (2, 'SQL', 77);  INSERT INTO scores VALUES (2, 'C',   83);
INSERT INTO scores VALUES (3, 'SQL', 92);  INSERT INTO scores VALUES (3, 'C',   81);
INSERT INTO scores VALUES (4, 'SQL', 80);  INSERT INTO scores VALUES (4, 'C',   76);
INSERT INTO scores VALUES (5, 'SQL', 85);  INSERT INTO scores VALUES (5, 'C',   89);

COMMIT;
```

### 오류 원인

현재 로그인한 계정(예: `SCOTT`, `HR` 등)에게 **테이블 생성·제약 생성 권한** 또는 **다른 스키마로 전환할 권한**이 없습니다.

Oracle은 보안을 위해 계정마다 DDL(객체 생성) 권한을 따로 부여하도록 설계돼 있으므로, 처음 환경을 잡을 때 한 번만 권한을 열어 주어야 합니다.

---

### ① 전용 실습 스키마를 새로 만드는 방법

*전제*: `SYSDBA` 또는 `SYSTEM` 등 DBA 권한으로 접속할 수 있을 때

1. **사용자 생성**
    
    관리자 계정으로 로그인한 뒤 다음처럼 새 스키마(예: `DMLTEST`)를 만듭니다.
    
    ```sql
    sql
    복사편집
    CREATE USER dmltest IDENTIFIED BY pw
      DEFAULT TABLESPACE users
      QUOTA UNLIMITED ON users;   -- 테이블스페이스 사용 한도 해제
    
    ```
    
2. **필수 권한 부여**
    
    실습 동안 테이블·시퀀스·트리거 등을 마음껏 만들 수 있도록 DDL 권한을 명시적으로 부여합니다.
    
    ```sql
    sql
    복사편집
    GRANT CREATE SESSION,        -- 로그인
          CREATE TABLE,          -- 테이블 생성
          CREATE VIEW,           -- 뷰 생성
          CREATE SEQUENCE,       -- 시퀀스 생성
          CREATE TRIGGER,        -- 트리거 생성
          CREATE PROCEDURE       -- PL/SQL 객체 생성
    TO dmltest;
    
    ```
    
3. **세션 전환 후 실습**
    
    수강용 계정에서 접속한 뒤
    
    ```sql
    sql
    복사편집
    ALTER SESSION SET CURRENT_SCHEMA = dmltest;
    
    ```
    
    를 실행하면, 이후 모든 객체가 새 스키마에 생성됩니다.
    

이 방식은 연습용 객체가 기존 업무 데이터베이스와 완전히 분리되므로, 실험·삭제를 자유롭게 할 수 있다는 장점이 있습니다.

---

### ② 현재 계정 그대로 사용하는 방법

*전제*: 새 사용자를 만들 권한이 없을 때(예: 사내 교육용 계정)

1. **스키마 전환·사용자 생성 구문을 삭제**
    
    `CREATE USER …`, `ALTER SESSION SET CURRENT_SCHEMA …` 부분을 통째로 지웁니다.
    
2. **테이블과 제약을 현 계정에 직접 생성**
    
    예시 스크립트의 `students`, `scores` 테이블 등을 지금 로그인해 있는 계정에 바로 만들면 됩니다.
    
3. **필요 DDL 권한이 없다면**
    
    테이블이나 트리거를 만들 때 *권한 부족* 오류가 발생하면, 관리자에게 다음 권한만 요청합니다.
    
    ```sql
    GRANT CREATE TABLE, CREATE SEQUENCE, CREATE TRIGGER
      TO <현재계정>;
    ```
    

이 두 가지 조치만으로도 대부분의 실습을 무리 없이 진행할 수 있습니다.

## 스크립트 수정 예시 (방법 B — 현 계정 사용)

```sql
/* ①·② 단계 통합 : 바로 테이블·제약 생성 */
CREATE TABLE students (
    student_id NUMBER CONSTRAINT pk_students PRIMARY KEY,
    name       VARCHAR2(50),
    dept       VARCHAR2(30)
);

CREATE TABLE scores (
    student_id NUMBER REFERENCES students(student_id),
    subject    VARCHAR2(20),
    score      NUMBER,
    CONSTRAINT pk_scores PRIMARY KEY (student_id, subject)
);

/* ③ 기본 데이터 */
INSERT INTO students VALUES (1,'김하나','컴퓨터');
... 생략 ...
COMMIT;
```

이와 같이 **계정 전환 / 새 사용자 생성 단계**를 삭제하면, 일반 학습 계정에서도 오류 없이 실습을 진행할 수 있습니다.

---

### 요약

- **DBA 권한이 있으면** — 새 스키마를 만들고 충분한 DDL 권한을 부여한 뒤, 세션을 그 스키마로 전환하여 실습한다.
- **DBA 권한이 없으면** — 사용자·스키마 전환 단계는 생략하고, 현 계정에서 직접 객체를 만든다. 필요한 최소 DDL 권한은 DBA에게 별도로 요청한다.
- 부족한 권한(테이블·트리거 생성 등)은 DBA에게 `GRANT CREATE TABLE, …` 형태로 별도 요청

위 방법 중 상황에 맞는 방안을 택해 스크립트를 다시 실행하시면 됩니다.

# 6.1 기본 DML 튜닝

### 6.1.1 DML 성능에 영향을 미치는 요소

> DML을 빠르게 하려면 **왕복(Call)·인덱스 작업 · 락**을 줄이는 전략(배치 전송, 인덱스 해제, MERGE 등)을 우선 적용하라.
> 

| 요소 | 설명(쉬운 풀이) |
| --- | --- |
| **Redo/Undo** | 변경 내용을 되살리기 위해 기록하는 로그. 쓰기가 느리면 전체 DML이 지연됩니다. |
| **인덱스 유지** | 행 하나를 넣어도 관련 인덱스마다 “책갈피”를 갱신해야 합니다. |
| **락(LOCK)** | 동시에 같은 행을 고치면 기다림 발생. 긴 트랜잭션일수록 충돌 시간이 길어집니다. |

### **✅** 실습 목표

*인덱스·트리거·제약* 개수가 많을수록 단일 DML이 느려짐을 체감한다.

```sql
-- ① 동일 테이블에 불필요한 인덱스 4개 추가
CREATE INDEX idx_students_name   ON students(name);
CREATE INDEX idx_students_dept   ON students(dept);
CREATE INDEX idx_students_dept2  ON students(dept);
CREATE INDEX idx_students_dummy  ON students(name, dept);

-- ② 부가 트리거 정의(이벤트 로깅용)
CREATE OR REPLACE TRIGGER trg_students_log
AFTER INSERT OR UPDATE OR DELETE ON students
BEGIN
   NULL; -- 여기서는 로깅하지 않아도 트리거 자체가 부하
END;
/

-- ③ 동일 행 INSERT 5,000건 비교
SET TIMING ON
DECLARE
    v_start NUMBER := DBMS_UTILITY.get_time;
BEGIN
    FOR i IN 1001..6000 LOOP
        INSERT INTO students VALUES (i, '테스트'||i, '테스트부');
    END LOOP;
    COMMIT;
    DBMS_OUTPUT.put_line( (DBMS_UTILITY.get_time - v_start)/100 || '초 경과' );
END;
/

-- ④ 인덱스·트리거 제거 후 동일 루프 재실행 → 속도 차이 확인
DROP TRIGGER trg_students_log;
DROP INDEX idx_students_name;
DROP INDEX idx_students_dept;
DROP INDEX idx_students_dept2;
DROP INDEX idx_students_dummy;

-- 다시 5,000건 INSERT+COMMIT 실행
```

**원인 — ORA-01408: such column list already indexed**

Oracle DB는 **동일 테이블에, 같은 컬럼 조합(순서까지 동일)** 을 가진 B-Tree 인덱스를 두 개 이상 생성하는 것을 허용하지 않습니다.

이미 `idx_students_dept  ON students(dept)` 가 존재하므로, 즉시 이어지는

```sql
CREATE INDEX idx_students_dept2 ON students(dept);
```

명령은 중복 정의로 간주되어 `ORA-01408` 오류가 발생합니다.

## 해결·대안

| 방법 | 설명 | 예시 |
| --- | --- | --- |
| **1. 중복 인덱스 생성 생략** | 시나리오 목적이 “인덱스가 많을수록 DML이 느려진다”는 것을 보여주는 데 있다면, 굳이 같은 컬럼으로 두 번 만들 필요가 없습니다. | `idx_students_dept2` 구문을 삭제 |
| **2. 컬럼 구성을 다르게** | 서로 다른 컬럼 조합이면 여러 개 만들어도 오류가 없습니다(그만큼 DML 부하는 커짐). | `sql CREATE INDEX idx_students_dept2 ON students(dept, name);` |
| **3. 인덱스 유형 차별화** | B-Tree ↔ Bitmap 이나 Function-Based Index라면 같은 컬럼이라도 공존 가능. | `sql CREATE BITMAP INDEX idx_students_dept2 ON students(dept);` (OLAP 전용·DML 적음 환경에서만 권장) |

---

## 수정된 예제 스크립트

```sql
-- ① (중복 제거) 이름·부서 컬럼에 다양한 인덱스 부여
CREATE INDEX idx_students_name   ON students(name);
CREATE INDEX idx_students_dept   ON students(dept);
CREATE INDEX idx_students_dept2  ON students(dept, name);   -- ← 조합 변경
CREATE INDEX idx_students_dummy  ON students(name, dept);

-- ② 부가 트리거

AFTER INSERT OR UPDATE OR DELETE ON students
BEGIN
   NULL;
END;
/

-- ③ 5,000건 INSERT 속도 측정

BEGIN

        ROUND((DBMS_UTILITY.get_time - v_start)/100,2)||'초');
```

### 6.1.2 데이터베이스 Call과 성능

- **Call**은 ‟클라이언트 ↔ 서버” 한 번 왕복입니다.
    - 1건씩 1,000번 보내면 **round-trip 1,000회**.
    - 100건씩 10번 보내면 **round-trip 10회** → 네트워크 지연이 1/100.

### **✅** 실습 목표

**왕복 SQL**(1건씩)과 **배치 전송**(100건씩)의 차이를 측정한다.

```sql
-- ① 1건씩 1,000회 루프
SET TIMING ON
BEGIN
    FOR i IN 1..1000 LOOP
        UPDATE scores SET score = score + 1 WHERE student_id = MOD(i,5)+1 AND subject='SQL';
    END LOOP;
END;
/

-- ② Array 방식(예: PL/SQL FORALL) : 100건씩 10배치
DECLARE
    TYPE t_id  IS TABLE OF NUMBER;
    TYPE t_sub IS TABLE OF VARCHAR2(20);
    l_id  t_id  := t_id();        -- 바인드 배열 준비
    l_sub t_sub := t_sub();
BEGIN
    FOR b IN 1..10 LOOP
        l_id.DELETE; l_sub.DELETE;
        FOR i IN 1..100 LOOP
            l_id.EXTEND;  l_sub.EXTEND;
            l_id(i)  := MOD((b-1)*100 + i, 5) + 1;
            l_sub(i) := 'SQL';
        END LOOP;
        FORALL i IN 1..l_id.COUNT
            UPDATE scores
               SET score = score + 1
             WHERE student_id = l_id(i)
               AND subject    = l_sub(i);
    END LOOP;
END;
/
```

---

### 6.1.3 Array Processing

- PL/SQL: `FORALL`, JDBC: `batch`, Python cx_Oracle: `executemany` 사용.
- 서버 관점에서는 “1000건을 한꺼번에 받아 한 번에 처리”하므로 **파싱·락** 횟수도 1회.

### **✅** Array Processing (JDBC 예시)

```java
// Java 8+ 예 : addBatch() 활용
try (Connection con = DriverManager.getConnection(url, id, pwd);
     PreparedStatement ps = con.prepareStatement(
           "INSERT INTO students(student_id, name, dept) VALUES(?, ?, ?)")) {
    for (int i = 7001; i <= 8000; i++) {
        ps.setInt(1, i);
        ps.setString(2, "배치_" + i);
        ps.setString(3, "테스트부");
        ps.addBatch();                 // 배열에 담음
        if (i % 100 == 0) ps.executeBatch();  // 100건마다 전송
    }
    ps.executeBatch();                 // 남은 건 전송
    con.commit();
}
```

### 6.1.4 인덱스·제약 해제 후 대량 DML

1. `ALTER INDEX ... UNUSABLE` 로 사용 중지
2. 대량 INSERT 또는 DELETE 진행
3. `ALTER INDEX ... REBUILD` 재생성
    - 대량 작업 시 **로그 양이 줄어** redo 부담도 낮아집니다.

```sql
-- ① 점수 테이블 인덱스 일시 해제
ALTER INDEX pk_scores          UNUSABLE;
ALTER INDEX fk_scores_students UNUSABLE;

-- ② 다량 INSERT (예시 100만 행)
INSERT /*+ APPEND */ INTO scores
SELECT ROWNUM + 100000 AS student_id,
       DECODE(MOD(ROWNUM,2),0,'SQL','C') AS subject,
       TRUNC(DBMS_RANDOM.value(60,100))
FROM dual CONNECT BY LEVEL <= 1_000_000;

COMMIT;

-- ③ 인덱스 재구축
ALTER INDEX pk_scores          REBUILD;
ALTER INDEX fk_scores_students REBUILD;
```

### 6.1.5 수정가능 조인 뷰

- 아래 조건을 모두 만족하면 뷰에서 곧바로 INSERT/UPDATE/DELETE가 가능
    1. 조인된 테이블 중 **한 테이블의 PK**가 뷰에 포함
    2. `GROUP BY`, `DISTINCT`, 집계 함수 같은 변형이 없음
- 개발자는 “조인된 결과”를 보고도 바로 UPDATE할 수 있어 코딩이 단순해집니다.

```sql
-- ① 뷰 생성
CREATE OR REPLACE VIEW stu_score_v AS
SELECT s.student_id,
       s.name,
       sc.subject,
       sc.score
  FROM students s
  JOIN scores   sc
    ON s.student_id = sc.student_id;

-- ② 뷰를 통해 점수 수정 (PK 포함 → 수정가능)
UPDATE stu_score_v
   SET score = 100
 WHERE student_id = 1
   AND subject    = 'SQL';

COMMIT;
```

### 6.1.6 MERGE 문

- 구조
    
    ```sql
    MERGE INTO <대상> t
    USING (<소스>) s
    ON (매칭조건)
    WHEN MATCHED THEN UPDATE ...
    WHEN NOT MATCHED THEN INSERT ...
    ```
    
- 장점
    - **두 번 락** 걸 필요 없이 한 번에 끝.
    - 차등 로깅이 가능해 redo 양이 **최솟값**.

```sql
-- ① 갱신·삽입 대상 테이블 및 데이터
CREATE TABLE new_scores AS
SELECT 1 AS student_id, 'SQL' AS subject, 95 AS score FROM dual UNION ALL
SELECT 6, 'SQL', 88 FROM dual;   -- 새 학생

-- ② MERGE 수행
MERGE INTO scores tgt
USING new_scores src
   ON (tgt.student_id = src.student_id AND tgt.subject = src.subject)
 WHEN MATCHED THEN
      UPDATE SET tgt.score = src.score
 WHEN NOT MATCHED THEN
      INSERT (student_id, subject, score)
      VALUES (src.student_id, src.subject, src.score);

COMMIT;
```

### **✅** 직접 확인 체크리스트

| 체크 항목 | 확인 방법 |
| --- | --- |
| 인덱스·트리거 증가 → DML 지연 | 6.1.1 실습 전후 경과 시간 비교 |
| Array 처리 효과 | `SET TIMING` 또는 애플리케이션 로그 비교 |
| 인덱스 해제-재빌드 후 Redo 양 감소 | `V$LOG_HIST` 또는 AWR Redo Size 비교 |
| MERGE 문 한 번에 행 수·락 수 감소 | `V$SESSION_EVENT`(row lock)·`V$SYSSTAT`(user calls) |

# 6.2 Direct Path I/O 활용

<aside>
🔖

**대량 적재 · 갱신**은 *“Direct Path + (필요 시) 병렬 DML”* 로 캐시 오염과 로그 쓰기를 줄이고, 대신 롤백 · 공간 관리 리스크를 미리 대비하자.

</aside>

| 소절 | 핵심 개념 | 실무 팁 | 간단 실습 예제 |
| --- | --- | --- | --- |
| **6.2.1 Direct Path I/O** | 버퍼 캐시를 거치지 않고 **디스크 ↔ PGA** 로 곧장 읽기·쓰기 | • 대량 작업은 캐시 오염*¹*을 줄여 성능이 안정됨.
• Undo·Redo가 최소화되므로 로그 폭주 방지. | 읽기·쓰기 예제는 6.2.2·6.2.3에서 상세 실습 |
| **6.2.2 Direct Path Insert** | `INSERT /*+ APPEND */` → 기존 블록을 건너뛰고 **새 공간**을 한 번에 할당 | • 대상 테이블은 **NOLOGGING** 옵션과 궁합이 좋음.• 롤백 시에는 일반 Insert보다 느릴 수 있음. | ▶ *실습 A* |
| **6.2.3 병렬 DML** | `ALTER SESSION ENABLE PARALLEL DML` 후 `/*+ PARALLEL */` 힌트 → 여러 프로세스가 **동시에 기록** | • Direct Path와 함께 쓰면 I/O 대역폭 극대화.
• 파티션 키가 없으면 **락 경합**에 유의. | ▶ *실습 B* |
- 캐시 오염(cache pollution) : 대용량 데이터를 읽어오면서 기존 핫 블록이 버퍼 캐시에서 밀려나는 현상.

---

### 6.2.1 Direct Path I/O ― 작동 방식

| 구분 | 일반 Path I/O | Direct Path I/O |
| --- | --- | --- |
| 데이터 흐름 | 디스크 → **버퍼 캐시** → PGA | 디스크 ↔ **PGA 직접** |
| 버퍼 캐시 영향 | 블록이 남아 캐시 히트율이 변동 | 캐시 사용량 변하지 않음 |
| 로그(Redo) | 모든 블록이 자세히 기록 | 최소 정보만 기록(대량 Insert 시) |

## 실습 A : Direct Path Insert

> 목표 : 버퍼 캐시 사용량, Redo 양 차이를 체감한다.
> 

```sql
ALTER SESSION SET CURRENT_SCHEMA = DMLTEST;

-- 1) 비교 대상 테이블 준비
CREATE TABLE big_scores AS SELECT * FROM scores WHERE 1=0; -- 빈 테이블
ALTER TABLE big_scores NOLOGGING;                          -- 대량 적재용 설정

-- 2) 일반 Insert (버퍼 캐시 경유)
SET TIMING ON
INSERT INTO big_scores
SELECT * FROM scores
CONNECT BY LEVEL <= 200_000;   -- 약 20만 행
COMMIT;
SET TIMING OFF

-- 3) Direct Path Insert
TRUNCATE TABLE big_scores;    -- 다시 비움
SET TIMING ON
INSERT /*+ APPEND */ INTO big_scores
SELECT * FROM scores
CONNECT BY LEVEL <= 200_000;
COMMIT;
SET TIMING OFF

-- 4) 결과 비교 : Redo 양(바이트)
SELECT name, value
  FROM v$statname sn JOIN v$mystat ms ON sn.statistic# = ms.statistic#
 WHERE name = 'redo size';
```

**관찰 포인트**

1. `APPEND` 없이 실행한 2)번이 더 오래 걸리고 Redo Size가 크다.
2. Direct Path Insert(3)번은 버퍼 캐시 hit 율도 변하지 않음을 V$BH(버퍼 헤드)로 확인 가능.

---

## 실습 B : 병렬 DML + Direct Path Insert

> 목표 : CPU 여러 코어를 활용해 Insert 속도를 끌어올린다.
> 

```sql
-- 1) 세션에 병렬 DML 허용
ALTER SESSION ENABLE PARALLEL DML;

-- 2) 테이블을 파티션 없이 단순 생성
DROP TABLE big_scores PURGE;
CREATE TABLE big_scores NOLOGGING
PARALLEL 4                 -- DDL 단계 힌트
AS SELECT * FROM scores WHERE 1=0;

-- 3) 병렬 Insert + Direct Path
SET TIMING ON
INSERT /*+ APPEND PARALLEL(big_scores 4) */
INTO big_scores
SELECT * FROM scores
CONNECT BY LEVEL <= 1_000_000;     -- 100만 행
COMMIT;
SET TIMING OFF

-- 4) 사용된 병렬 프로세스 확인
SELECT degree, dop, requested_dop FROM v$pq_sesstat
WHERE statistic = 'Server Threads';
```

**유의사항**

| 항목 | 설명 |
| --- | --- |
| Undo, Redo | 병렬 + Direct Path라도 최소 메타 정보는 로그에 남음. NOLOGGING → 백업 정책 유의. |
| 공간 할당 | 테이블 하이워터마크(HWM)가 한 번에 크게 상승. → 이후 일반 Insert는 새 블록부터 쌓이므로 공간이 넓게 들 수 있음. |
| 롤백 | 실패 시 병렬 프로세스가 모두 취소해야 하므로 단일 Insert보다 시간이 더 걸린다. |

# 6.3  파티션을 활용한 DML 튜닝

### 6.3.1 테이블 파티션

<aside>
🔖

**파티션은 “대량 DML의 안전벨트”**

· *범위가 뚜렷*  할수록 효과가 크며,

· “**프루닝 · 교환 · 자르기(TRUNCATE)**” 세 가지 기술로 **UPDATE · DELETE · INSERT** 를 빠르고 안정적으로 수행한다.

</aside>

## ✅ 공통 전제 – 대용량 샘플 테이블 준비

```sql
ALTER SESSION SET CURRENT_SCHEMA = DMLTEST;

-- (1) 5백만 행짜리 원본 테이블 생성 : 1년치 일(yyyy-mm-dd) + 점수
DROP TABLE big_scores_raw PURGE;
CREATE TABLE big_scores_raw
AS
SELECT
       ROWNUM                     AS id,
       TRUNC(SYSDATE) - MOD(ROWNUM, 365) AS score_dt,
       MOD(ROWNUM, 5)  + 1        AS student_id,    -- 1~5
       DECODE(MOD(ROWNUM,2),0,'SQL','C') AS subject,
       TRUNC(DBMS_RANDOM.VALUE(60, 100)) AS score
FROM dual
CONNECT BY LEVEL <= 5_000_000;

EXEC DBMS_STATS.gather_table_stats(user,'BIG_SCORES_RAW');
```

| 항목 | 설명 |
| --- | --- |
| 핵심 개념 | 한 테이블을 **작은 조각(파티션)** 으로 쪼개 저장하면, 검색·수정·삭제 때 **필요한 조각만** 읽어 더 빠르다. |
| 실무 팁 | • “날짜·번호 범위” → **Range 파티션** / “고객ID 해시” → **Hash 파티션**• 대량 로드 전 **NOLOGGING + PARALLEL** 로 속도↑ |

### 실습 ① Range 파티션 생성·프루닝

```sql
DROP TABLE big_scores PURGE;
CREATE TABLE big_scores
PARTITION BY RANGE (score_dt)   -- 날짜별 월 단위
(
  PARTITION p2024_07 VALUES LESS THAN (DATE '2024-08-01'),
  PARTITION p2024_08 VALUES LESS THAN (DATE '2024-09-01'),
  PARTITION p2024_09 VALUES LESS THAN (DATE '2024-10-01'),
  PARTITION pmax     VALUES LESS THAN (MAXVALUE)
)
NOLOGGING PARALLEL 4
AS SELECT * FROM big_scores_raw;   -- 5백만 행 삽입

ALTER TABLE big_scores PARALLEL 1; -- 이후는 일반 모드
EXEC DBMS_STATS.gather_table_stats(user,'BIG_SCORES',granularity=>'ALL');

-- ▶ 파티션 프루닝 확인
SET AUTOT TRACE ONLY EXPLAIN
SELECT COUNT(*) FROM big_scores
 WHERE score_dt = DATE '2024-07-15';
SET AUTOT OFF
```

> 실행 계획의 PARTITION RANGE SINGLE 을 확인하면 특정 파티션만 스캔하는 것을 알 수 있다.
>

## 6.3.2 인덱스 파티션

| 항목 | 설명 |
| --- | --- |
| 핵심 개념 | 테이블만 쪼개면 인덱스가 병목이 될 수 있다. **Local 인덱스**는 파티션과 1:1로 매치되어 함께 잘린다. |
| 실무 팁 | • **Global 인덱스**는 파티션 교체 시 재빌드 필요.• **Local 인덱스 + 해당 파티션만 통계 분석** → 유지비↓ |

### 실습 ② Local vs Global

```sql
-- Local 인덱스 (파티션별 독립)
CREATE INDEX bix_scores_local
ON big_scores(score_dt, student_id)
LOCAL;

-- Global 인덱스 (단일 구조)
CREATE INDEX bix_scores_global
ON big_scores(student_id)
GLOBAL;

--  한 파티션(P2024_07) TRUNCATE
ALTER TABLE big_scores TRUNCATE PARTITION p2024_07 UPDATED INDEXES;

-- ▶ 인덱스 상태 확인
SELECT index_name, partition_name, status
  FROM user_ind_partitions
 WHERE index_name = 'BIX_SCORES_LOCAL'
ORDER BY partition_name;

SELECT status FROM user_indexes
 WHERE index_name = 'BIX_SCORES_GLOBAL'; -- 'UNUSABLE' → 재빌드 필요
```

## 6.3.3 파티션을 활용한 대량 UPDATE

| 항목 | 설명 |
| --- | --- |
| 핵심 개념 | **교체(Exchange) 기법** : 수정 대상 파티션을 *복사본 테이블* 로 빼내 UPDATE 후 다시 붙이면 잠금이 짧다. |
| 실무 팁 | • *원본과 구조가 100 % 동일* 해야 `ALTER TABLE … EXCHANGE PARTITION` 가능.• 교환 전후 **Global 인덱스** 주의. |

### 실습 ③ Partition Exchange Update

```sql
-- 1) 2024-08 파티션만 복사
CREATE TABLE aug_scores
AS SELECT * FROM big_scores
      PARTITION (p2024_08);

-- 2) 복사본에서 점수 +5
UPDATE aug_scores SET score = score + 5;
COMMIT;

-- 3) 파티션 ↔ 복사본 교환 (1초 이내)
ALTER TABLE big_scores
  EXCHANGE PARTITION p2024_08 WITH TABLE aug_scores
  WITHOUT VALIDATION;  -- 데이터 검증 생략(구조 동일 가정)

-- 4) 복사본 테이블 정리
DROP TABLE aug_scores PURGE;
```

## 6.3.4 파티션을 활용한 대량 DELETE

| 항목 | 설명 |
| --- | --- |
| 핵심 개념 | 파티션을 **TRUNCATE** 하면 Undo·Redo 거의 없이 1초에 수백만 행 삭제 효과. |
| 실무 팁 | • 논리적으로 “2019년 이전 자료 폐기” 같은 조건이 뚜렷할 때 사용.• Global 인덱스는 `UPDATE INDEXES` 옵션 필요. |

### 실습 ④ Partition Truncate

```sql
-- 대상 : p2024_09 파티션 내용 삭제
ALTER TABLE big_scores
  TRUNCATE PARTITION p2024_09
  DROP STORAGE         -- 공간 반환
  UPDATE INDEXES;      -- 글로벌 인덱스 상태 유지
```

## 6.3.5 파티션을 활용한 대량 INSERT

| 항목 | 설명 |
| --- | --- |
| 핵심 개념 | **PARTITION FOR(N)** 와 Direct Path Insert를 병용하면 *해당 파티션* 만 빠르게 채울 수 있다. |
| 실무 팁 | • ETL 배치에서 “당월 파티션만 적재” 패턴에 적합.• 필요 시 `NOLOGGING` + `PARALLEL` 로 극대화. |

### 실습 ⑤ Target Partition Insert

```sql
-- 1) 9월 파티션을 다시 빈 상태로 확보
ALTER TABLE big_scores
  TRUNCATE PARTITION p2024_09 DROP STORAGE;

-- 2) Direct Path Insert (9월 데이터 20만 행)
INSERT /*+ APPEND PARALLEL(b 4) */
INTO big_scores PARTITION (p2024_09) b
SELECT * FROM big_scores_raw
 WHERE score_dt BETWEEN DATE '2024-09-01' AND DATE '2024-09-30';

COMMIT;
```

# 6.4 Lock과 트랜잭션 동시성 제어

## 6.4.1 오라클 Lock

| 소절 | 핵심 개념 (쉽게) | 실무 팁 | 실습 ① : 행 락·테이블 락 |
| --- | --- | --- | --- |
| **Row Lock (TX)** | 한 행(row)만 잠금. 
다른 행은 **동시에 수정 가능** | OLTP의 기본. 
*커밋 전* 까지만 잠금 유지 | **세션 A**`UPDATE students SET grad='Y' WHERE stu_id=1;`(커밋 보류)  ↔  **세션 B**`UPDATE students SET grad='N' WHERE stu_id=1;`→ 대기(락 충돌)`UPDATE students SET grad='N' WHERE stu_id=2;`→ 즉시 실행 |
| **Table Lock (TM)** | **테이블 전체** 잠금. 
DDL·대량 DML에서 주로 발생 | • `ALTER TABLE` 전에 한산 시간대 확보
• 대량 Insert 시 `APPEND` 옵션은 *공유 데이터 락*만 사용 | **세션 A**`ALTER TABLE students ADD (flag CHAR(1));`(DDL 실행 중)  ↔  **세션 B**`INSERT INTO students VALUES (99,'Delay','M',NULL);`→ 대기 |
| **Dictionary Lock (DD)** | 카탈로그 객체에 대한 메타 락 | 여러 DDL이 동시에 실행될 때 충돌 위험 | — |
| **Deadlock** | 서로가 서로의 락을 기다려 **순환 대기** | 설계 상 키 순서, 처리 순서를 고정하여 예방 | 실습 ② 참고 |

### 실습 ② : Deadlock 만들기 → 알아보기

```sql
-- 세션 A
UPDATE students SET sex='M' WHERE stu_id=1;
-- 세션 B
UPDATE students SET sex='F' WHERE stu_id=2;
-- 세션 A
UPDATE students SET sex='M' WHERE stu_id=2;   -- 대기
-- 세션 B
UPDATE students SET sex='F' WHERE stu_id=1;   -- 60초 후 ORA-00060 (deadlock)
```

확인 :

```sql
SELECT * FROM v$locked_object;
ALTER SYSTEM DUMP DATAFILE 1 BLOCK 0;  -- 필요 시 트레이스
```

---

## 6.4.2 트랜잭션 동시성 제어

| 항목 | 설명 |
| --- | --- |
| **MVCC** | 오라클은 다중 버전(Multi-Version Concurrency Control) 구조. → SELECT는 **락을 잡지 않고** Undo 영역을 이용해 “과거 이미지”를 읽는다. |
| **Isolation Level** | 기본값 *Read Committed* (가장 최근 커밋 값). 필요 시 `SET TRANSACTION ISOLATION LEVEL SERIALIZABLE` 사용 가능하지만, 대량 경합 시 **ORA-08177** 가능성. |
| **락 대기 시간** | `ALTER SESSION SET DDL_LOCK_TIMEOUT = 60;SET LOCK_TIMEOUT 30`(12c↑). → 무한 대기 방지. |
| **지연(Wait) 진단** | `v$session`, `v$session_wait`, `dba_blockers`, `dba_waiters` 뷰 활용. |

### 실습 ③ : 락 대기 시간 한도 설정

```sql
-- 세션 B
ALTER SESSION SET LOCK_TIMEOUT = 5;  -- 5초만 기다림
UPDATE students SET grad='Y' WHERE stu_id=1;  -- 세션 A가 락 중
```

5 초 후 ORA-30006 : **resource busy** 오류를 확인할 수 있다.

---

## 6.4.3 채번 방식에 따른 INSERT 성능 비교

| 방식 | 특징 | 장단점 | 실습 ④ 성능 확인 |
| --- | --- | --- | --- |
| **일반 시퀀스 + NOCACHE** | 매 INSERT 때 redo·undo 기록 | 동시성 안전, 그러나 **로그 I/O↑** | ▶ `SEQ_NOCACHE` |
| **시퀀스 + CACHE 100** | 번호 100개를 **메모리 캐시**. 로그·랙 감소 | 인스턴스 다운 시 **번호 건너뜀** 가능 | ▶ `SEQ_CACHE` |
| **시퀀스 + CACHE 1000 NOORDER** | RAC 환경에서 노드별 캐시 | 순서 보장 안 함 → 최고 성능 | ▶ `SEQ_CACHE_NOORD` |
| **IDENTITY 컬럼** | 시퀀스 자동 관리, 내부적으로 CACHE 사용 | 동일 | 동일 |

### 실습 ④ : 1초 동안 INSERT TPS(Time Per Second) 비교

```sql
-- 1) 시퀀스 준비
CREATE SEQUENCE seq_nocache      NOCACHE;
CREATE SEQUENCE seq_cache        CACHE 100;
CREATE SEQUENCE seq_cache_noord  CACHE 1000 NOORDER;

-- 2) 테스트 테이블
DROP TABLE t_insert PURGE;
CREATE TABLE t_insert (id NUMBER PRIMARY KEY, v CHAR(10));

-- 3) PL/SQL 루틴
DECLARE
  t1 NUMBER := DBMS_UTILITY.get_time;
  n  NUMBER := 0;
BEGIN
  WHILE DBMS_UTILITY.get_time - t1 < 100  -- 약 1초(1/100초 단위)
  LOOP
     INSERT /*+ APPEND */ INTO t_insert
       VALUES (seq_cache.NEXTVAL, 'X');   -- 시퀀스 교체하며 3회 반복
     n := n + 1;
  END LOOP;
  COMMIT;
  DBMS_OUTPUT.put_line('Rows='||n);
END;
/

-- 위 블록에서 seq_cache → seq_nocache / seq_cache_noord 교체하며 3번 실행
```

> 일반적으로 CACHE 100이 NOCACHE 대비 3 ~ 5 배 이상 TPS가 높고,
> 
> 
> `CACHE 1000 NOORDER` 가 단일 노드에서도 가장 빠르다.
>

# 🧭 DML 튜닝 종합실습 예제

**“베이스라인 → 튜닝 → 효과 검증”** 흐름이 포함되어 있어, 한 눈에 성능 개선을 확인할 수 있습니다.

> 실행 환경
> 
> - Oracle 19c 이상, `DMLTEST` 스키마
> - 실습 전 `SET TIMING ON` (소요 시간 자동 출력)
> - 병행 DML(PDML)을 위해 `ALTER SESSION ENABLE PARALLEL DML;`
> - 세션 A, 세션 B 두 개를 띄워 Lock 실습까지 확인

---

## 0. 공통 준비

```sql
ALTER SESSION SET CURRENT_SCHEMA = DMLTEST;

-- (0-1) 대용량 원본 테이블 - 1천만 행
DROP TABLE big_sales_raw PURGE;
CREATE TABLE big_sales_raw
AS
SELECT
    ROWNUM                         AS sale_id,
    TRUNC(SYSDATE)-MOD(ROWNUM, 365) AS sale_dt,
    MOD(ROWNUM, 1000)+1            AS cust_id,
    TRUNC(DBMS_RANDOM.VALUE(1, 1e4)) AS amount
FROM dual
CONNECT BY LEVEL <= 10_000_000;

EXEC DBMS_STATS.gather_table_stats(user,'BIG_SALES_RAW');
```

## 1. 배열 처리 + 커밋 간격 (6.1 기본 DML 튜닝)

| 단계 | 내용 | 코드 요약 |
| --- | --- | --- |
| ① 베이스라인 | 일반 `INSERT … VALUES …` + 건당 커밋 | PL/SQL 루프 10만 행 → 시간 측정 |
| ② 튜닝 | **FORALL / BULK COLLECT** 로 Array Insert + **1000건당 COMMIT** | 동일 데이터 삽입, 시간 비교 |

```sql
-- ① 기준 (건당 커밋)
DECLARE
  v_start TIMESTAMP := SYSTIMESTAMP;
BEGIN
  FOR i IN 1..100000 LOOP
    INSERT INTO sales_test VALUES (i, SYSDATE, 1, 100);
    COMMIT;
  END LOOP;
  DBMS_OUTPUT.put_line('Row-by-row = '||
        (SYSTIMESTAMP-v_start));
END;
/

-- ② 튜닝 (배열 + 간격커밋)
DECLARE
  TYPE t_num IS TABLE OF NUMBER INDEX BY PLS_INTEGER;
  v_id   t_num; v_amt t_num;
  v_start TIMESTAMP := SYSTIMESTAMP;
BEGIN
  -- 10만 row bulk load
  FOR i IN 1..100000 LOOP
      v_id(i)  := i;
      v_amt(i) := 100;
  END LOOP;

  FORALL idx IN v_id.FIRST..v_id.LAST SAVE EXCEPTIONS
      INSERT INTO sales_test VALUES
        (v_id(idx), SYSDATE, 1, v_amt(idx));

  COMMIT;  -- 한 번만
  DBMS_OUTPUT.put_line('Bulk = '||(SYSTIMESTAMP-v_start));
END;
/
```

## 2. Direct-Path Insert & 병렬 DML (6.2)

```sql
-- (2-1) Range 파티션 테이블
DROP TABLE big_sales PURGE;
CREATE TABLE big_sales
PARTITION BY RANGE (sale_dt)
( PARTITION p_old  VALUES LESS THAN (DATE '2024-01-01'),
  PARTITION p2024  VALUES LESS THAN (DATE '2025-01-01'),
  PARTITION pmax   VALUES LESS THAN (MAXVALUE) );

-- (2-2) Direct Path Insert + PDML
ALTER SESSION ENABLE PARALLEL DML;
INSERT /*+ APPEND PARALLEL(bs 4) */
INTO big_sales PARTITION (p2024) bs
SELECT * FROM big_sales_raw;
COMMIT;
ALTER SESSION DISABLE PARALLEL DML;
```

- **효과**
    - Redo / Undo 최소화(NOLOGGING 효과)
    - 4개 슬레이브가 파티션(p2024)만 직접 쓰기

## 3. 파티션 교환·잘라내기·붙이기 (6.3)

### 3-1. Exchange Update

```sql
CREATE TABLE tmp_sales AS
SELECT * FROM big_sales PARTITION (p2024);

UPDATE tmp_sales SET amount = amount * 1.1;
COMMIT;

ALTER TABLE big_sales
  EXCHANGE PARTITION p2024 WITH TABLE tmp_sales
  WITHOUT VALIDATION;
DROP TABLE tmp_sales PURGE;
```

### 3-2. 파티션-단위 DELETE(Truncate)

```sql
ALTER TABLE big_sales
  TRUNCATE PARTITION p_old DROP STORAGE UPDATE INDEXES;
```

## 4. Lock & 동시성 (6.4)

### 4-1. 행 락 vs 테이블 락

```sql
-- 세션 A
UPDATE big_sales SET amount=amount+10
 WHERE sale_id BETWEEN 1 AND 100;  -- 커밋 보류

-- 세션 B (곧바로 실행)
UPDATE big_sales SET amount=amount-5
 WHERE sale_id = 200;

-- 세션 B (락 대기 발생)
UPDATE big_sales SET amount=amount-5
 WHERE sale_id = 50;  -- 세션 A가 잡은 행
```

### 4-2. 시퀀스 캐시 성능

```sql
CREATE SEQUENCE seq_sale_id CACHE 1000 NOORDER;
INSERT INTO sales_test
SELECT seq_sale_id.NEXTVAL, SYSDATE, 1, 50
FROM dual CONNECT BY LEVEL <= 100000;
COMMIT;
```

> CACHE 선언으로 Redo·Latch 가 크게 줄어 TPS 향상.

### 요약 정리 – 6 장 핵심 룰

| 영역 | 핵심 문장 |
| --- | --- |
| **기본 DML** | “**배열(Forall) + 주기적 Commit** 로 네트워크·Redo 줄여라.” |
| **Direct Path** | “대량 적재는 **APPEND + PARALLEL** 로, 일반 작업과 I/O 경합 최소.” |
| **Partition** | “대량 UPDATE·DELETE = **교환·잘라내기** 한 방에 끝낸다.” |
| **Lock** | “Row Lock 범위 유지, DDL 전 한산 시간 확보, 캐시 시퀀스로 병목 예방.” |

해당 스크립트를 차례로 실행하며 **타이밍 결과·대기 이벤트** 를 관찰하시면 6 장 ”DML 튜닝”의 전술이 하나의 흐름으로 연결됨을 확인하실 수 있습니다.
