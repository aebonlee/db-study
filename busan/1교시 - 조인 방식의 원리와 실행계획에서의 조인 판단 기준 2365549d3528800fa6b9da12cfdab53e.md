# 1교시 - 조인 방식의 원리와 실행계획에서의 조인 판단 기준

### 🎯 학습 목표

- Oracle에서 사용되는 대표적인 조인 방식(Nested Loop, Hash Join, Merge Join)의 내부 동작 방식을 이해한다.
- 실행계획 내에서 각 조인 방식이 선택되는 조건과 상황을 판단할 수 있다.
- 통계 정보에 따라 옵티마이저가 선택하는 조인 전략의 변화 과정을 설명할 수 있다.
- 힌트를 활용해 조인 방식을 변경하고 성능 차이를 실습을 통해 분석할 수 있다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “SQL에서 가장 많은 비용을 유발하는 연산이 무엇일까요? 바로 ‘조인’입니다.
    
    오늘은 옵티마이저가 어떤 조인 방식을 선택하고, 그 선택이 왜 중요한지를 배웁니다.
    
    SQL 성능의 절반 이상은 조인 전략이 결정합니다. 실행계획에서 조인 연산이 어떻게 표현되고, 어떤 원리로 선택되는지를 이번 시간에 파헤쳐 봅시다.”
    

---

1. **[조인 방식 설명 – 15분]**
    
    🔹 **Nested Loop Join (NLJ)**
    
    - 작은 테이블에서 루프를 돌며 큰 테이블을 탐색
    - 인덱스가 존재하면 효율적
    - 일반적으로 `인덱스 기반 조인`, 소규모 조합에서 유리
    
    🔹 **Hash Join**
    
    - 해시 버킷을 만들어 연결 → 대량 데이터에 효율적
    - 인덱스 불필요, Full Scan 기반
    - `병렬처리 가능`, OLAP/통계계산 등에 유리
    
    🔹 **Sort Merge Join**
    
    - 두 테이블 모두 정렬 후 병합
    - 데이터가 이미 정렬돼 있을 때만 유리
    - 실무에서는 거의 사용되지 않지만, 조인 순서 오류 시 나올 수 있음

---

1. **[실행계획에서의 조인 방식 해석 – 10분]**
    - 실행계획 키워드:
        - `NESTED LOOPS`
        - `HASH JOIN`
        - `MERGE JOIN`
    - 예시:
        
        ```sql
        
        SELECT A.ENAME, B.DNAME
        FROM EMP A, DEPT B
        WHERE A.DEPTNO = B.DEPTNO;
        ```
        
    - 실행계획 확인:
        
        ```sql
        
        EXPLAIN PLAN FOR ...
        SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
        ```
        
    - 조인 방식 선택 기준:
        - 작은 테이블이 선행 테이블일 경우 → Nested Loop
        - Full Scan이 불가피하고 대용량일 경우 → Hash Join

---

1. **[실습 – 조인 방식에 따른 성능 비교 – 15분]**
    
    🔸 실습 1: 기본 SQL
    
    ```sql
    
    SELECT A.ENAME, B.DNAME
    FROM EMP A, DEPT B
    WHERE A.DEPTNO = B.DEPTNO;
    ```
    
    🔸 실습 2: 조인 방식 힌트 적용
    
    ```sql
    
    SELECT /*+ USE_NL(A B) */ A.ENAME, B.DNAME ...
    SELECT /*+ USE_HASH(A B) */ A.ENAME, B.DNAME ...
    ```
    
    🔸 실습 3: 조인 순서 제어
    
    ```sql
    
    SELECT /*+ LEADING(B A) USE_HASH(A B) */ ...
    ```
    
    ➤ 실행계획에서 `Id`, `Operation`, `Cost`, `Rows` 항목 해석 훈련
    
    ➤ 조인 순서 변경 → 성능 차이 및 Row 접근 수 변화 체감
    

---

1. **[조인 전략 판단 기준 정리 – 10분]**
    - 옵티마이저는 조인 순서, 조인 방식 모두 **통계 정보 기반으로 판단**
    - 통계가 부정확할 경우 비효율적인 Hash Join이나 Sort Merge Join 선택 가능
    - 조인 순서 변경 시 Join Tree 구조가 전면 재편 → 실행계획 해석 필수
    - 힌트를 사용할 때는 반드시 통계 정보, 클러스터링 팩터, 인덱스 유무 등 함께 고려해야 함

---

1. **[마무리 정리 – 5분]**
    - 모든 SQL 튜닝의 중심에는 ‘조인’이 있다
    - 옵티마이저의 판단이 틀릴 수도 있고, 튜닝의 핵심은 **그 판단의 의도를 읽고 고쳐주는 것**
    - “조인은 SQL의 무게중심이다.
        
        조인이 잘못되면 아무리 좋은 인덱스도 무용지물이 된다.”
