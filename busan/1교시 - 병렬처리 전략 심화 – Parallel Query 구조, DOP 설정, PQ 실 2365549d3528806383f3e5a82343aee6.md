# 1교시 - 병렬처리 전략 심화 – Parallel Query 구조, DOP 설정, PQ 실행계획 해석 및 튜닝 실습

### 🎯 학습 목표

- Oracle에서 병렬처리가 작동하는 구조(Producer–Consumer 모델, PX 프로세스)를 이해한다.
- SQL 수준에서 병렬 힌트를 사용하고, Degree of Parallelism(DOP)을 조정할 수 있다.
- 병렬 실행계획(PX BLOCK, PX SEND 등)을 해석하고 성능에 미치는 영향을 분석할 수 있다.
- 병렬 수행 시 발생할 수 있는 병목 요소(PX Wait, Resource 소비 등)를 진단할 수 있다.

    다수의 CPU 코어를 동시에 활용하는 것이 성능 향상의 핵심이 됩니다.
    
    오늘은 병렬 쿼리 구조, 실행계획 해석, DOP 설정 전략 등을 실습하며 확인하겠습니다.”
    

---

1. **[Parallel Query 기본 구조 이해 – 10분]**
    
    ▶ Oracle PQ 구조
    
    - **Query Coordinator (QC)**: 병렬 처리 전체 관리
    - **Parallel Execution Server (PX)**: 실제 데이터를 처리하는 프로세스들
    - Producer/Consumer 구조: 데이터를 나누고, 처리하고, 모아서 반환
    
    ▶ PQ의 효과
    
    - 대량 테이블에 대해 Full Scan, Join, Aggregation 작업을 병렬로 분산 처리
    - 주로 배치, DW 환경, 통계 생성, 리포트 쿼리에 적용

---

1. **[DOP 설정 방법과 힌트 사용 – 10분]**
    
    ▶ 힌트 방식
    
    ```sql
    SELECT /*+ PARALLEL(SALES 4) */ * FROM SALES;
    ```
    
    ▶ 테이블 단위 설정
    
    ```sql
    ALTER TABLE SALES PARALLEL 4;
    ```
    
    ▶ 자동 병렬화
    
    ```sql
    ALTER SESSION ENABLE PARALLEL QUERY;
    ALTER SESSION SET parallel_degree_policy = AUTO;
    ```
    
    ▶ 확인:
    
    ```sql
    SELECT DEGREE FROM USER_TABLES WHERE TABLE_NAME = 'SALES';
    ```
    

---

1. **[병렬 실행계획 해석 실습 – 15분]**
    
    ▶ 실행계획에서 확인 가능한 키워드:
    
    | 키워드 | 의미 |
    | --- | --- |
    | PX SEND | 데이터 전송 (QC ↔ PX ↔ PX) |
    | PX BLOCK | PX가 데이터를 블록 단위로 분할 |
    | PX COORDINATOR | 병렬 실행 전체 제어자 |
    | PARTITION RANGE | 파티션과 병렬이 함께 동작함 |
    
    ▶ 예제 SQL:
    
    ```sql
    EXPLAIN PLAN FOR
    SELECT /*+ PARALLEL(SALES 4) */ CUST_ID, SUM(AMOUNT)
    FROM SALES
    GROUP BY CUST_ID;
    SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
    ```
    
    ➤ `PX BLOCK`, `PX SEND`, `TABLE ACCESS FULL` 등이 표시됨
    
    ➤ 실행 시간, COST 감소 여부 확인
    

---

1. **[병렬 처리 병목 요소와 튜닝 전략 – 5분]**
    
    ▶ 병목 증상 및 원인:
    
    | 증상 | 원인 | 대응 전략 |
    | --- | --- | --- |
    | PX Wait 발생 | PX 간 동기화 지연 | 데이터 균등 분할 유도 |
    | CPU 과부하 | 너무 높은 DOP | 서버 자원 고려해 DOP 설정 |
    | TEMP 사용 급증 | GROUP BY 등 정렬 병렬처리 | Sort Merge 줄이기, 메모리 할당 증가 |
    
    ▶ 진단 뷰:
    
    ```sql
    SELECT * FROM V$PX_SESSION;
    SELECT * FROM V$PQ_TQSTAT;
    ```
    

---

1. **[정리 및 마무리 – 5분]**
    - 병렬처리는 튜닝의 ‘마지막 무기’지만, 잘못 쓰면 자원 경합의 원인이 될 수 있음
    - DOP는 성능을 높이는 동시에, 병목도 함께 만들 수 있음
    - “병렬 처리는 **빠르게 나누는 것이 아니라,
    잘게 나누어 잘 모으는 전략**이다.”
