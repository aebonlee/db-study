# Day05 - [친절한 SQL튜닝] 7장. SQL 옵티마이저

# 🔍 SQL 옵티마이저 핵심 요약

### 1. **SQL 옵티마이저란?**

- SQL 실행 시 **최적의 실행 계획(Execution Plan)** 을 선택하는 Oracle의 핵심 엔진.
- SQL의 성능을 좌우하는 **자동 계획 선택 시스템**.

### 2. **옵티마이저의 종류**

| 종류 | 설명 | 상태 |
| --- | --- | --- |
| **RBO (Rule-Based Optimizer)** | 규칙 기반. 우선순위 룰에 따라 실행계획 선택 | Oracle 10g 이후 폐기 |
| **CBO (Cost-Based Optimizer)** | **비용(비용 = I/O + CPU)** 을 계산하여 가장 낮은 실행계획 선택 | 현재 Oracle 표준 |

### 3. **옵티마이저 모드**

| 모드 | 설명 | 용도 |
| --- | --- | --- |
| **ALL_ROWS** | 전체 처리 성능(처리량) 중심 | Batch 작업, DSS |
| **FIRST_ROWS(n)** | 빠른 응답 중심 | OLTP, 웹 화면 조회 등 |
| **CHOOSE** | 통계 있으면 CBO, 없으면 RBO 선택 | 거의 사용 안 함 |

`optimizer_mode` 파라미터 또는 `ALTER SESSION SET optimizer_mode = ...` 구문으로 지정 가능

### 4. **실행 계획 결정 절차**

Oracle 데이터베이스는 SQL 문장을 실행할 때, **"어떤 방법으로 데이터를 가장 빠르게 가져올 수 있을까?"** 를 먼저 고민합니다.

이것이 바로 **실행 계획(Execution Plan)** 을 만드는 과정이며, 이 과정을 담당하는 것이 **옵티마이저(Optimizer)** 입니다.

- 각 단계는 마치 사람이 문제를 푸는 여러 가지 방법을 고민한 뒤, 가장 빠르고 쉬운 방법을 선택하는 과정과 비슷합니다.

**✅ 옵티마이저는 다음과 같은 5단계를 거쳐 실행 계획을 결정합니다.**

| 단계 | 설명 |
| --- | --- |
| 1. SQL 파싱 | 문법 확인 및 구조 검사 |
| 2. 쿼리 변환 | 더 효율적인 형태로 내부 변환 |
| 3. 실행계획 후보 생성 | 다양한 실행 방법 탐색 |
| 4. 비용 계산 | 각 후보 방법의 예상 자원 소모 계산 |
| **5. 최적 계획 선택** | **가장 효율적인 방법 선택하여 실행** |

### **1단계. SQL 파싱 (SQL Parsing)**

SQL 문장이 데이터베이스에 전달되면, 오라클은 먼저 문장이 문법에 맞는지 확인합니다.

이 작업은 사람이 글을 쓸 때 철자나 맞춤법을 검사하는 것과 같습니다.

SQL 문장에 오류가 없다면, 다음 단계로 넘어갑니다.

예시:

```sql
SELECT * FROM students WHERE grade = 'A';
```

👉 위 SQL 문장이 문법적으로 올바른지를 먼저 검사합니다.

### **2단계. 쿼리 변환 (Query Transformation)**

문법 검사가 끝난 SQL은 더 효율적으로 실행될 수 있도록 내부적으로 변형됩니다.

예를 들어, 복잡한 서브쿼리를 단순한 조인으로 바꾸거나, 불필요한 조건을 제거하기도 합니다.

💡 예시:

```sql
SELECT * FROM employees WHERE salary > (SELECT AVG(salary) FROM employees);
```

👉 위와 같은 서브쿼리를 옵티마이저는 내부에서 다른 형태로 바꾸어 더 빠르게 실행되도록 준비합니다.

### **3단계. 실행 계획 후보 생성**

SQL을 실행할 수 있는 다양한 방법(후보)을 옵티마이저가 만들어 냅니다.

예를 들어 테이블을 전체 검색할 수도 있고, 인덱스를 활용할 수도 있습니다.

각 방법마다 장단점이 있으며, 옵티마이저는 이 중 어떤 방법들이 가능한지를 모두 나열해 봅니다.

🎲 마치 여러 경로로 학교에 갈 수 있는 것처럼,

- 지하철 타기
- 버스 타기
- 자전거 타기
    
    이런 여러 가지 경로를 떠올리는 과정입니다.
    

### **4단계. 비용 계산 (Cost Calculation)**

각 실행 방법(후보 경로)에 대해 얼마나 시간이 걸릴지, 얼마나 많은 자원이 필요한지 계산합니다.

이때 계산되는 수치는 '비용(Cost)'이라고 하며, 숫자가 낮을수록 효율적인 실행 방법을 의미합니다.

📊 예:

- 경로 A (Full Scan): 비용 500
- 경로 B (Index Scan): 비용 200
    
    👉 이 경우, B가 더 효율적입니다.
    

옵티마이저는 CPU 사용량, 디스크 읽기 횟수, 메모리 사용량 등을 바탕으로 비용을 추정합니다.

### **5단계. 최저 비용 실행계획 선택**

마지막으로, 옵티마이저는 계산된 여러 경로 중 가장 비용이 낮은 방법을 **최종 실행계획**으로 선택합니다.

이 실행계획이 바로 실제로 데이터베이스가 사용하는 **일 처리 방법(Execution Plan)**이 됩니다.

📌 실행계획은 다음과 같은 형태로 확인할 수 있습니다:

```sql
EXPLAIN PLAN FOR SELECT * FROM employees WHERE department_id = 10;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### ✅ 마무리 요약

옵티마이저는 단순히 SQL을 그대로 실행하지 않고, **최대한 빠르고 적은 자원으로 결과를 얻기 위해 최선을 다하는 'SQL의 길잡이'** 역할을 합니다.

따라서 SQL을 잘 짜는 것도 중요하지만, 옵티마이저가 어떻게 작동하는지 이해하는 것은 **성능 좋은 데이터베이스 프로그램을 만들기 위한 필수 지식**입니다.

### 5. **실행계획에 영향 주는 주요 요소**

| 구분 | 예시 |
| --- | --- |
| **SQL 문장 구조** | `LIKE`, `NVL`, `IS NOT NULL`, `NOT IN` 등 |
| **통계정보** | DBMS_STATS로 수집된 테이블/인덱스/히스토그램 정보 |
| **인덱스 유무 및 구조** | B-Tree, Bitmap, Composite Index 등 |
| **SQL Hint** | `/*+ FULL */`, `/*+ INDEX */`, `/*+ USE_HASH */` 등 |
| **옵티마이저 파라미터** | `optimizer_mode`, `cursor_sharing`, `query_rewrite_enabled` 등 |
| **Oracle 버전** | 11g, 19c, 23ai 등 버전에 따라 계획 생성 방식 달라짐 |

### 6. **통계정보 관련 핵심**

- 옵티마이저는 항상 최신 통계정보를 참고하여 비용을 계산
- 통계정보 없거나 오래된 경우 비효율적 실행계획 선택 가능
- 수집 명령:

```sql
EXEC DBMS_STATS.GATHER_TABLE_STATS(USER, '테이블명');
```

### 7. **선택도(Selectivity)와 카디널리티(Cardinality)**

- **선택도**: 조건 만족 행의 비율 (낮을수록 좋음)
- **카디널리티**: 조건 만족 예상 행 수
- 옵티마이저는 **선택도 × 총 행 수** 로 카디널리티 계산 → 비용 산정

### 8. **바인드 변수와 실행계획 공유**

| 구분 | 설명 |
| --- | --- |
| 리터럴 SQL | 매번 Hard Parsing 발생 |
| 바인드 변수 SQL | 실행계획 재사용 (Soft Parsing 유도) |
| 효과 | SQL 공유 증가, CPU·Latch 부하 감소 |

```sql
SELECT * FROM customers WHERE region = :region;
```

### 9. **옵티마이저 힌트(Hint)**

- 옵티마이저의 실행계획 선택을 개발자가 직접 유도하는 방법

| 힌트 | 설명 |
| --- | --- |
| `/*+ FULL(tab) */` | 테이블 전체 스캔 유도 |
| `/*+ INDEX(tab idx_name) */` | 인덱스 사용 유도 |
| `/*+ USE_HASH(a b) */` | 해시 조인 강제 |
| `/*+ LEADING(tab1 tab2) */` | 조인 순서 지정 |
| `/*+ PARALLEL(tab 4) */` | 병렬처리 유도 |

### 10. **실행계획 확인 방법**

```sql
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### 11. **비용 계산 기준**

- **I/O 비용 + CPU 비용 + 메모리 활용** 등으로 추정
- 비용은 절대 수치가 아닌 **상대적인 비교값**
- 예: Cost = 1,000 → 다른 실행계획이 2,000이면 1,000을 선택

### 12. **SQL 튜닝에서 옵티마이저를 이해해야 하는 이유**

- 동일 SQL도 실행계획이 달라지면 속도가 수십 배 차이 발생
- 잘못된 실행계획 → 인덱스 미사용, 조인 순서 오류, 소트 과다 등
- **SQL의 성능 문제 80% 이상은 옵티마이저 실행계획 오판 때문**

### 🔁 함께 외워야 할 고정 표현들

- **“옵티마이저는 최소 비용의 실행계획을 선택한다.”**
- **“비용은 실행계획 간 상대적 기준으로 계산된다.”**
- **“통계정보가 최신이 아니면 옵티마이저 판단은 실패한다.”**
- **“바인드 변수를 사용해야 SQL이 공유되고 Soft Parsing 된다.”**

# [친절한 SQL튜닝] **7장. SQL 옵티마이저**

## **7.1 통계정보와 비용 계산 원리**

### **7.1.1 선택도와 카디널리티**

- **선택도(Selectivity)**: 특정 조건에 부합하는 데이터의 비율 (값이 낮을수록 좋음)
- **카디널리티(Cardinality)**: 조건에 해당하는 예상 행 수
- 옵티마이저는 통계정보를 통해 선택도 및 카디널리티를 예측하고, 최적 실행계획 선택에 활용

### 📌 7.1.1 선택도와 카디널리티 실습

### 🧪 실습 예제

```sql
-- 사전 테이블 생성 및 데이터 삽입
CREATE TABLE customers (
  id NUMBER PRIMARY KEY,
  gender CHAR(1),
  region VARCHAR2(20)
);

BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO customers VALUES (
      i,
      CASE WHEN MOD(i,2)=0 THEN 'M' ELSE 'F' END,
      CASE
        WHEN MOD(i,10)=0 THEN '서울'
        WHEN MOD(i,10)=1 THEN '부산'
        ELSE '기타'
      END
    );
  END LOOP;
  COMMIT;
END;
/

-- 통계정보 수집
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(USER, 'CUSTOMERS');
END;
/

-- 실행계획 확인
EXPLAIN PLAN FOR
SELECT * FROM customers WHERE gender = 'M';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### 💡 실행 결과 비교

| 항목 | Oracle 11g | Oracle 23ai Free |
| --- | --- | --- |
| `gender='M'` 조건 선택도 계산 | 기본 통계 기반 | 기본 통계 + 예측 기반 |
| 실행계획 | Index Full Scan or Table Access Full | Cost 기준 Table Access 선택 |
| 힌트 적용 가능 여부 | 동일 (`/*+ INDEX(customers pk_customers) */`) | 동일 |

### **7.1.2 통계정보**

- 테이블, 인덱스, 컬럼 등에 대한 **DBMS_STATS** 수집 필요
- 히스토그램을 통한 분포 정보 보완
- 통계정보가 부정확하면 옵티마이저 판단이 잘못될 수 있음

### 📌 7.1.2 비용 계산 실습 예제

```sql
-- SELECT gender='M' 조건으로 Full Scan 유도
EXPLAIN PLAN FOR
SELECT /*+ FULL(customers) */ * FROM customers WHERE gender = 'M';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### **7.1.3 비용 계산 원리**

- 옵티마이저는 다양한 실행계획을 비교하고, 각 경로의 예상 비용(Cost)을 계산
- 비용은 I/O, CPU 자원 소모량에 기반
- 실행 계획의 비용 = 각 연산 단계의 총합

---

### **7.2 옵티마이저에 대한 이해**

### **7.2.1 옵티마이저 종류**

- **Cost-Based Optimizer(CBO)**: 통계정보 기반으로 비용을 계산해 최적 실행계획 선택
    - **Rule-Based Optimizer(RBO)**: 미사용됨 (10g 이후 Deprecated)

### 📌 7.2.1 옵티마이저 종류 + 모드 실습

```sql
-- 옵티마이저 모드 변경
ALTER SESSION SET optimizer_mode = ALL_ROWS;

-- ALL_ROWS (처리량 중심)
SELECT /*+ FULL(customers) */ * FROM customers WHERE region = '서울';

ALTER SESSION SET optimizer_mode = FIRST_ROWS_10;

-- FIRST_ROWS_10 (응답속도 중심)
SELECT /*+ FULL(customers) */ * FROM customers WHERE region = '서울';
```

### ✅ 실습 분석

| 모드 | Oracle 11g | Oracle 23ai Free |
| --- | --- | --- |
| ALL_ROWS | 전체 처리 효율 고려 (Full Scan 경향) | Cost 계산 중심 선택 |
| FIRST_ROWS_10 | 인덱스 우선 고려 경향 | AI Optimizer가 유사 실행계획으로 보정 |

### **7.2.2 옵티마이저 모드**

- `ALL_ROWS`: 처리량(Throughput) 중심 → 대용량 처리 시 유리
- `FIRST_ROWS(n)`: 응답속도 중심 → 빠르게 일부 결과 필요 시 유리
- `CHOOSE`: 통계 정보 존재 여부에 따라 ALL_ROWS 또는 RBO 선택 (이제는 거의 사용 안 함)

### **7.2.3 옵티마이저에 영향을 미치는 요소**

- SQL 문장 구조 (`LIKE`, `NVL`, `IS NOT NULL`, `NOT IN` 등)
- 인덱스 존재 유무 및 구조
- 통계정보의 유무 및 최신성
- 옵티마이저 초기화 파라미터 설정 (`optimizer_mode`, `cursor_sharing`, 등)
- Oracle DB 버전 차이 (e.g., 11g vs 19c)

### 📌 7.2.3 옵티마이저에 영향을 미치는 요소 실습

```sql
-- 인덱스 생성
CREATE INDEX idx_customers_region ON customers(region);

-- NVL 사용: 옵티마이저가 인덱스 사용 못함
EXPLAIN PLAN FOR
SELECT * FROM customers WHERE NVL(region, '서울') = '서울';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### 🔍 결과 차이

- 11g에서는 NVL 사용 시 인덱스 미사용 → Full Scan
- 23ai에서는 AI 기반 예측 보정되어 인덱스 사용 가능성 ↑ (단, 반드시 실행계획 확인)

### **7.2.4 옵티마이저의 한계**

- 통계정보가 없거나 부정확하면 실행계획이 비효율적일 수 있음
- 복잡한 조인, 서브쿼리 등에서 잘못된 계획 선택 가능
- 실행계획 캐시(Shared Pool)로 인해 동일한 SQL이 항상 최적화되지는 않음

### **7.2.5 개발자의 역할**

- **바인드 변수 사용**을 통해 Soft Parsing 유도
- 적절한 인덱스 설계 및 힌트 사용
- 통계정보 수집 및 모니터링
- SQL을 모듈화하고, 가능한 집합(Set) 기반 처리 유도
- 성능 모니터링 도구(AWR, V$SQL 등) 활용

### 📌 7.2.5 개발자의 역할 실습

### 바인드 변수 vs 리터럴 SQL

```sql
-- 리터럴 SQL (Hard Parsing 유도)
SELECT * FROM customers WHERE region = '부산';

-- 바인드 변수 SQL (Soft Parsing 유도)
VAR v_region VARCHAR2(20)
EXEC :v_region := '부산';
SELECT * FROM customers WHERE region = :v_region;
```

### 🔍 확인 포인트

- `V$SQLAREA` 또는 `V$SQL`에서 `EXECUTIONS`, `PARSE_CALLS` 차이 비교
- Oracle 23ai Free에서도 동일 적용됨

### **7.2.6 튜닝 전문가 되는 공부방법**

- **실제 실행계획 분석 능력** 필수
- 다양한 Wait Event 및 V$ 뷰 이해
- SQL Tuning Advisor, AWR 보고서 분석 등 실습 반복
- Oracle 공식 문서 및 실무 사례 기반 경험 축적
- **한 SQL을 다양한 방식으로 튜닝해보는 실습** 필요 (e.g., Hint 사용, 인덱스 추가/제거, 통계정보 변경 등)

### 📌 7.2.6 튜닝 전문가가 되는 방법

- AWR 리포트 분석 연습
- V$SQL, V$SESSION, V$SQL_PLAN 사용 능력
- 다양한 옵티마이저 힌트 테스트 (`FULL`, `INDEX`, `LEADING`, `USE_HASH`, `USE_NL` 등)
- SQL Tuning Advisor 실습 (11g 이후부터 가능, 23ai Free에서는 일부 기능 제외)

## ✅ 옵티마이저 전체 실습 예제

```sql
-- =========================================
-- SQL 옵티마이저 실습 워크북 (Oracle 11g & 23ai Free)
-- 대상: 옵티마이저 동작원리, 통계정보, 실행계획, 힌트 실습
-- =========================================

-- [0] 준비: 기존 테이블 삭제 (있다면)
DROP TABLE customers PURGE;

-- [1] 테이블 생성 및 데이터 삽입
CREATE TABLE customers (
  id NUMBER PRIMARY KEY,
  gender CHAR(1),
  region VARCHAR2(20)
);

BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO customers VALUES (
      i,
      CASE WHEN MOD(i,2)=0 THEN 'M' ELSE 'F' END,
      CASE 
        WHEN MOD(i,10)=0 THEN '서울'
        WHEN MOD(i,10)=1 THEN '부산'
        ELSE '기타'
      END
    );
  END LOOP;
  COMMIT;
END;
/

-- [2] 통계정보 수집
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(USER, 'CUSTOMERS');
END;
/

-- [3] 실행계획 보기: 기본 조건 검색 (Index 미사용 가능성)
EXPLAIN PLAN FOR
SELECT * FROM customers WHERE gender = 'M';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- [4] FULL 스캔 강제 유도
EXPLAIN PLAN FOR
SELECT /*+ FULL(customers) */ * FROM customers WHERE gender = 'M';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- [5] 옵티마이저 모드 실습
ALTER SESSION SET optimizer_mode = ALL_ROWS;
SELECT /*+ FULL(customers) */ * FROM customers WHERE region = '서울';

ALTER SESSION SET optimizer_mode = FIRST_ROWS_10;
SELECT /*+ FULL(customers) */ * FROM customers WHERE region = '서울';

-- [6] 인덱스 생성 및 사용 테스트
CREATE INDEX idx_customers_region ON customers(region);
EXPLAIN PLAN FOR
SELECT * FROM customers WHERE region = '부산';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- [7] 인덱스 비활용 예: NVL 사용
EXPLAIN PLAN FOR
SELECT * FROM customers WHERE NVL(region, '서울') = '서울';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- [8] 바인드 변수 실습
VAR v_region VARCHAR2(20);
EXEC :v_region := '부산';
SELECT * FROM customers WHERE region = :v_region;

-- [9] SQL 힌트 실습
EXPLAIN PLAN FOR
SELECT /*+ INDEX(customers idx_customers_region) */ * FROM customers WHERE region = '부산';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```
