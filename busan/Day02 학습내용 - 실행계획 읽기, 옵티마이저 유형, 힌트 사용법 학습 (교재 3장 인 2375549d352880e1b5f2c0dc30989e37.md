## 🎯 3.1 테이블 액세스 최소화

```sql

-- ▼ 1. 고객 테이블 생성 및 초기화
DROP TABLE customer PURGE;

CREATE TABLE customer (
    id NUMBER PRIMARY KEY,
    name VARCHAR2(100),
    region VARCHAR2(10),
    birth_date DATE,
    phone_no VARCHAR2(20)
);

-- ▼ 2. 고객 데이터 삽입 (총 10,000건)
BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO customer VALUES (
      i,
      '고객_' || MOD(i, 1000),
      CASE MOD(i, 3)
        WHEN 0 THEN '서울'
        WHEN 1 THEN '부산'
        ELSE '대구'
      END,
      TO_DATE('1980-01-01','YYYY-MM-DD') + MOD(i, 3650),
      '010-' || TO_CHAR(MOD(i, 10000), 'FM0000')
    );
  END LOOP;
  COMMIT;
END;
/

-- ▼ 3. 이름 컬럼 인덱스 생성
CREATE INDEX idx_customer_name ON customer(name);

-- ▼ 4. 고객 이름을 기준으로 인덱스를 타는 SELECT 쿼리
EXPLAIN PLAN FOR
SELECT * FROM customer WHERE name = '고객_99';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

```

## ✅ 3.1.1 테이블 랜덤 액세스

### ▣ 설명

- 인덱스를 통해 해당 row의 RowID를 찾아가고, 이 RowID를 이용해 테이블 블록에 접근하는 방식입니다.
- 인덱스의 탐색은 순차적이지만, 테이블 접근은 랜덤하게 발생합니다

### ▣ 실습 예제

```sql
CREATE INDEX idx_customer_name ON customer(name);

SELECT * FROM customer WHERE name = '고객_99';
```

### ▣ 기대 결과

- `EXPLAIN PLAN` 결과에서 `INDEX RANGE SCAN` + `TABLE ACCESS BY ROWID`가 뜨면 성공입니다.
- 실제 결과 예시:

```

| Id | Operation                   | Name               |
|----|-----------------------------|--------------------|
|  0 | SELECT STATEMENT            |                    |
|  1 |  TABLE ACCESS BY INDEX ROWID| CUSTOMER           |
|  2 |   INDEX RANGE SCAN          | IDX_CUSTOMER_NAME  |

```

---

## ✅ 3.1.2 인덱스 클러스터링 팩터

### ▣ 설명

- 인덱스의 leaf 블록 순서와 테이블의 row 블록 순서의 유사도.
- 클러스터링 팩터가 낮을수록 랜덤 I/O가 줄어듦.

### ▣ 실습 예제

```sql

SELECT INDEX_NAME, CLUSTERING_FACTOR FROM USER_INDEXES WHERE TABLE_NAME = 'CUSTOMER';

-- 정렬된 테이블 재생성
CREATE TABLE customer_sorted AS
SELECT * FROM customer ORDER BY name;

CREATE INDEX idx_sorted_name ON customer_sorted(name);
SELECT INDEX_NAME, CLUSTERING_FACTOR FROM USER_INDEXES WHERE TABLE_NAME = 'CUSTOMER_SORTED';

```

### ▣ 기대 결과

- `customer_sorted`의 클러스터링 팩터가 `customer`보다 낮게 나와야 합니다.

---

## ✅ 3.1.3 인덱스 손익분기점

### ▣ 설명

- 소량의 Row를 반환할 때는 인덱스가 유리, 다량의 Row를 반환할 때는 Full Scan이 유리함.

### ▣ 실습 예제

```sql

EXPLAIN PLAN FOR SELECT * FROM customer WHERE name LIKE '고객_%';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### ▣ 기대 결과

- 조건의 Selectivity가 낮으면 `INDEX RANGE SCAN`, 높으면 `TABLE ACCESS FULL`이 나옵니다.

---

## ✅ 3.1.4 인덱스 컬럼 추가 (Covering Index)

### ▣ 설명

- 자주 조회되는 컬럼을 인덱스에 포함시켜 **Table Access 없이** 인덱스만으로 결과를 반환.

### ▣ 실습 예제

```sql

CREATE INDEX idx_customer_cover ON customer(name, phone_no);

EXPLAIN PLAN FOR SELECT name, phone_no FROM customer WHERE name = '고객_10';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

```

### ▣ 기대 결과

- `TABLE ACCESS BY ROWID` 없이 `INDEX RANGE SCAN`만 나와야 함.
- PLAN에서 `TABLE ACCESS`가 보이지 않아야 “인덱스만 읽고 처리”에 해당.

---

## ✅ 3.1.5 인덱스만 읽고 처리

### ▣ 설명

- 위 3.1.4와 동일.
- 인덱스에 필요한 모든 컬럼이 들어 있으면 테이블 블록에 접근하지 않음.

### ▣ 실습 예제

- 위 예제를 그대로 사용 가능.

---

## ✅ 3.1.6 인덱스 구조 테이블 (Index Organized Table)

### ▣ 설명

- 테이블 자체가 인덱스 구조로 저장됨 (행이 leaf block에 존재).
- Random I/O를 줄이고, 공간도 절약 가능.

### ▣ 실습 예제

```sql

CREATE TABLE iot_customer (
    id NUMBER,
    name VARCHAR2(100),
    CONSTRAINT pk_iot_customer PRIMARY KEY(id)
) ORGANIZATION INDEX;

INSERT INTO iot_customer SELECT id, name FROM customer;
COMMIT;

EXPLAIN PLAN FOR SELECT name FROM iot_customer WHERE id = 500;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### ▣ 기대 결과

- `TABLE ACCESS` 없이 `INDEX UNIQUE SCAN`으로만 결과 출력
- 실행계획 예시:

```

| Id | Operation           | Name               |
|----|---------------------|--------------------|
|  0 | SELECT STATEMENT    |                    |
|  1 | INDEX UNIQUE SCAN   | PK_IOT_CUSTOMER    |

```

---

## ✅ 3.1.7 클러스터 테이블

### ▣ 설명

- 여러 테이블을 **같은 클러스터 키** 기준으로 같은 블록에 저장하여 관련 레코드 접근시 성능 향상.

### ▣ 실습 예제

```sql

CREATE CLUSTER cust_cluster (region VARCHAR2(10)) SIZE 512;
CREATE TABLE clustered_customer (
    id NUMBER,
    name VARCHAR2(100),
    region VARCHAR2(10)
) CLUSTER cust_cluster(region);
CREATE INDEX idx_cluster_region ON CLUSTER cust_cluster;

INSERT INTO clustered_customer SELECT id, name, region FROM customer;
COMMIT;

EXPLAIN PLAN FOR SELECT * FROM clustered_customer WHERE region = '서울';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### ▣ 기대 결과

- 클러스터 인덱스를 통해 효율적으로 블록 접근.
- 실행계획에 `TABLE ACCESS CLUSTER`와 `INDEX UNIQUE SCAN`이 함께 나옴.

## 🎯 3.2 부분범위 처리 활용

```sql

-- ▼ 3.2.1 부분범위 처리 예제: Index Range Scan 유도
DROP TABLE orders PURGE;
CREATE TABLE orders (
    order_id NUMBER PRIMARY KEY,
    customer_id NUMBER,
    order_date DATE,
    amount NUMBER
);

-- 예제 데이터 삽입
BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO orders VALUES (
      i,
      MOD(i, 1000),
      TO_DATE('2020-01-01','YYYY-MM-DD') + MOD(i, 365 * 3),
      ROUND(DBMS_RANDOM.VALUE(1000, 100000), -3)
    );
  END LOOP;
  COMMIT;
END;
/

-- 인덱스 생성
CREATE INDEX idx_orders_orderdate ON orders(order_date);

-- ▼ 3.2.2 부분범위 처리 구현: 특정 날짜 범위만 가져오기
-- 전체 테이블 스캔 방지 & Index Range Scan 유도
EXPLAIN PLAN FOR
SELECT * FROM orders
WHERE order_date BETWEEN TO_DATE('2021-01-01','YYYY-MM-DD') AND TO_DATE('2021-01-31','YYYY-MM-DD');
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 날짜 조건 없이 실행했을 경우 (비교)
EXPLAIN PLAN FOR
SELECT * FROM orders;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- ▼ 3.2.3 OLTP 환경에서의 성능개선 원리 시뮬레이션
-- 주문 ID 검색 (PK 인덱스 이용)
EXPLAIN PLAN FOR
SELECT * FROM orders WHERE order_id = 500;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 주문 일자 + 금액 조건 복합 조회 (복합 인덱스로 튜닝 가능 예시)
CREATE INDEX idx_orders_date_amt ON orders(order_date, amount);

EXPLAIN PLAN FOR
SELECT * FROM orders
WHERE order_date BETWEEN TO_DATE('2022-01-01','YYYY-MM-DD') AND TO_DATE('2022-12-31','YYYY-MM-DD')
AND amount >= 50000;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

```

## ✅ 3.2.1 부분범위 처리

### ▣ 개념 설명

- `부분범위 처리(Partial Range Processing)`는 조건에 따라 **전체 테이블 대신 일부 블록만 액세스**하도록 제한함.
- 예: 최근 한 달치만 조회하는 **날짜 조건 필터** 적용

### ▣ 실습 예제

```sql

-- 특정 날짜 범위만 조회
SELECT * FROM orders
WHERE order_date BETWEEN TO_DATE('2021-01-01','YYYY-MM-DD')
                    AND TO_DATE('2021-01-31','YYYY-MM-DD');

```

### ▣ 기대 결과

- `INDEX RANGE SCAN`을 통해 필요한 범위만 읽음
- `TABLE ACCESS BY ROWID` 병행 발생 → Random I/O 적지만 빠름

---

## ✅ 3.2.2 부분범위 처리 구현

### ▣ 개념 설명

- 부분범위 처리를 위해서는 **조건절 WHERE에 인덱스를 타는 컬럼**을 활용해야 함
- 특히 `BETWEEN`, `>=`, `<=`, `IN` 등이 자주 사용됨

### ▣ 실습 예제

```sql

-- 날짜 인덱스 + 금액 조건 복합 필터
CREATE INDEX idx_orders_date_amt ON orders(order_date, amount);

SELECT * FROM orders
WHERE order_date BETWEEN TO_DATE('2022-01-01','YYYY-MM-DD') AND TO_DATE('2022-12-31','YYYY-MM-DD')
  AND amount >= 50000;

```

### ▣ 기대 결과

- 실행계획에서 `INDEX RANGE SCAN`이 발생하며 범위에 해당하는 레코드만 읽음
- Index 활용이 되지 않으면 `TABLE FULL SCAN`이 나타남 (성능 저하)

---

## ✅ 3.2.3 OLTP 환경에서 부분범위 처리에 의한 성능개선 원리

### ▣ 개념 설명

- OLTP는 빠른 응답 속도를 요구하므로, 가능한 **불필요한 Full Scan을 줄이는 게 핵심**
- **부분범위 처리**는 Index와 함께 사용될 때 Random I/O를 최소화하며 빠른 응답을 제공함

### ▣ 예시 실습

```sql

-- OLTP에서 주로 사용하는 단건 조회
SELECT * FROM orders WHERE order_id = 500;

-- 조건 추가한 복합 조회 (날짜 + 금액)
SELECT * FROM orders
WHERE order_date BETWEEN TO_DATE('2022-01-01','YYYY-MM-DD') AND TO_DATE('2022-12-31','YYYY-MM-DD')
AND amount >= 50000;

```

### ▣ 기대 결과

- 단건 조회는 `INDEX UNIQUE SCAN` 발생
- 복합 조회는 `INDEX RANGE SCAN` + 조건 최적화로 매우 빠른 응답
- 실제 OLTP 환경에서 1초 미만의 응답 시간 확보 가능

## 🎯 3.3  인덱스 스캔 효율화

```sql
-- ▼ 실습용 테이블 초기화
DROP TABLE emp PURGE;
CREATE TABLE emp (
    emp_id NUMBER PRIMARY KEY,
    dept_id NUMBER,
    job_id VARCHAR2(10),
    hire_date DATE,
    salary NUMBER
);

-- 데이터 삽입
BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO emp VALUES (
      i,
      MOD(i, 10),  -- dept_id: 0~9
      CASE MOD(i, 5)
        WHEN 0 THEN 'CLERK'
        WHEN 1 THEN 'MANAGER'
        WHEN 2 THEN 'ANALYST'
        WHEN 3 THEN 'SALES'
        ELSE 'IT_PROG'
      END,
      TO_DATE('2015-01-01','YYYY-MM-DD') + MOD(i, 365*5),
      ROUND(DBMS_RANDOM.VALUE(3000, 10000), -2)
    );
  END LOOP;
  COMMIT;
END;
/

-- 3.3.1 인덱스 탐색 (기본 인덱스 조회)
CREATE INDEX idx_emp_job ON emp(job_id);
EXPLAIN PLAN FOR SELECT * FROM emp WHERE job_id = 'CLERK';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.2 인덱스 스캔 효율성 (selectivity 차이)
EXPLAIN PLAN FOR SELECT * FROM emp WHERE job_id IN ('CLERK', 'IT_PROG');
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.3 액세스 조건과 필터 조건 차이
EXPLAIN PLAN FOR SELECT * FROM emp WHERE job_id = 'CLERK' AND salary > 5000;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.4 비교 연산자와 컬럼 순서
CREATE INDEX idx_emp_dept_job ON emp(dept_id, job_id);
EXPLAIN PLAN FOR SELECT * FROM emp WHERE dept_id = 5 AND job_id = 'MANAGER';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.5 선행 컬럼이 = 조건이 아닐 경우
EXPLAIN PLAN FOR SELECT * FROM emp WHERE dept_id > 5 AND job_id = 'CLERK';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.6 BETWEEN vs IN-List
EXPLAIN PLAN FOR SELECT * FROM emp WHERE dept_id BETWEEN 1 AND 3;
EXPLAIN PLAN FOR SELECT * FROM emp WHERE dept_id IN (1, 2, 3);
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.7 Index Skip Scan
CREATE INDEX idx_emp_job_dept ON emp(job_id, dept_id);
EXPLAIN PLAN FOR SELECT * FROM emp WHERE dept_id = 1;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.8 IN 조건은 ‘=’인가?
EXPLAIN PLAN FOR SELECT * FROM emp WHERE job_id IN ('CLERK', 'ANALYST');
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.9 BETWEEN vs LIKE 비교
EXPLAIN PLAN FOR SELECT * FROM emp WHERE hire_date BETWEEN TO_DATE('2017-01-01','YYYY-MM-DD') AND TO_DATE('2017-12-31','YYYY-MM-DD');
EXPLAIN PLAN FOR SELECT * FROM emp WHERE TO_CHAR(hire_date, 'YYYY') = '2017';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.10 범위검색 조건 남용
EXPLAIN PLAN FOR SELECT * FROM emp WHERE salary > 3000;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.11 다양한 조건 처리 방식
EXPLAIN PLAN FOR SELECT * FROM emp WHERE dept_id IN (1, 2, 3) AND job_id LIKE 'C%';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 3.3.12 함수 호출 해소 인덱스 구성
CREATE INDEX idx_emp_func ON emp(TO_CHAR(hire_date, 'YYYYMM'));
-- 실행계획 확인
EXPLAIN PLAN FOR SELECT * FROM emp WHERE TO_CHAR(hire_date, 'YYYYMM') = '201701';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

## ✅ 3.3.1 인덱스 탐색

- **설명:** 인덱스를 통해 조건 검색 수행

```sql
CREATE INDEX idx_emp_job ON emp(job_id);
SELECT * FROM emp WHERE job_id = 'CLERK';
```

- **기대 결과:** `INDEX RANGE SCAN` + `TABLE ACCESS BY ROWID`

---

## ✅ 3.3.2 인덱스 스캔 효율성

- **설명:** 선택도(Selectivity)가 높을수록 효율적인 인덱스 사용 가능

```sql
SELECT * FROM emp WHERE job_id IN ('CLERK', 'IT_PROG');
```

- **기대 결과:** 여러 번의 `INDEX RANGE SCAN` 또는 `BITMAP OR`

---

## ✅ 3.3.3 액세스 조건 vs 필터 조건

- **설명:** `WHERE` 조건 중 일부만 인덱스를 타고, 나머지는 필터링됨

```sql
SELECT * FROM emp WHERE job_id = 'CLERK' AND salary > 5000;
```

- **기대 결과:** `job_id`만 인덱스 탐색, `salary`는 Filter

---

## ✅ 3.3.4 비교 연산자와 컬럼 순서

- **설명:** 결합 인덱스에서 컬럼 순서가 인덱스 사용 효율에 영향

```sql
CREATE INDEX idx_emp_dept_job ON emp(dept_id, job_id);
SELECT * FROM emp WHERE dept_id = 5 AND job_id = 'MANAGER';
```

- **기대 결과:** `INDEX RANGE SCAN`

---

## ✅ 3.3.5 선행 컬럼이 등치가 아닐 경우

```sql
SELECT * FROM emp WHERE dept_id > 5 AND job_id = 'CLERK';
```

- **기대 결과:** `INDEX RANGE SCAN` 대신 `FULL SCAN` 가능성 높음

---

## ✅ 3.3.6 BETWEEN vs IN-List

```sql
SELECT * FROM emp WHERE dept_id BETWEEN 1 AND 3;
SELECT * FROM emp WHERE dept_id IN (1, 2, 3);
```

- **기대 결과:** 둘 다 `INDEX RANGE SCAN`, 단 IN은 OR 처리일 수 있음

## ✅ 3.3.7 Index Skip Scan

- **설명:** 선행 컬럼 없이도 후행 컬럼으로 인덱스 활용 가능

```sql
CREATE INDEX idx_emp_job_dept ON emp(job_id, dept_id);
SELECT * FROM emp WHERE dept_id = 1;
```

- **기대 결과:** `INDEX SKIP SCAN`

---

## ✅ 3.3.8 IN 조건은 '=' 인가?

```sql
SELECT * FROM emp WHERE job_id IN ('CLERK', 'ANALYST');
```

- **기대 결과:** 각각의 조건에 대해 개별 `INDEX RANGE SCAN` 발생

---

## ✅ 3.3.9 BETWEEN vs LIKE 범위 비교

```sql

-- 날짜 범위
SELECT * FROM emp WHERE hire_date BETWEEN TO_DATE('2017-01-01') AND TO_DATE('2017-12-31');

-- TO_CHAR 사용
SELECT * FROM emp WHERE TO_CHAR(hire_date, 'YYYY') = '2017';

```

- **기대 결과:** `TO_CHAR`는 인덱스를 사용하지 못함 → Full Scan

## ✅ 3.3.10 범위검색 조건 남용

```sql
SELECT * FROM emp WHERE salary > 3000;
```

- **기대 결과:** 선택도가 낮을 경우 `FULL SCAN` 발생 가능

## ✅ 3.3.11 다양한 옵션 조건 처리

```sql

SELECT * FROM emp WHERE dept_id IN (1, 2, 3) AND job_id LIKE 'C%';
```

- **기대 결과:** 복합 조건이지만 인덱스 활용 가능 여부는 옵티마이저에 따라 달라짐

## ✅ 3.3.12 함수 호출 해소를 위한 인덱스 구성

- **설명:** `TO_CHAR`, `SUBSTR` 등을 WHERE절에서 사용하면 일반 인덱스는 무효

```sql
CREATE INDEX idx_emp_func ON emp(TO_CHAR(hire_date, 'YYYYMM'));
SELECT * FROM emp WHERE TO_CHAR(hire_date, 'YYYYMM') = '201701';
```

- **기대 결과:** 함수 기반 인덱스로 `INDEX RANGE SCAN` 가능

-- ▼ 3.4 인덱스 설계 실습용 테이블
DROP TABLE orders PURGE;

CREATE TABLE orders (
    order_id NUMBER PRIMARY KEY,
    customer_id NUMBER,
    order_date DATE,
    product_id NUMBER,
    quantity NUMBER,
    amount NUMBER
);

-- 데이터 삽입
BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO orders VALUES (
      i,
      MOD(i, 500),
      TO_DATE('2020-01-01', 'YYYY-MM-DD') + MOD(i, 1000),
      MOD(i, 100),
      MOD(i, 5) + 1,
      ROUND(DBMS_RANDOM.VALUE(10000, 50000), -2)
    );
  END LOOP;
  COMMIT;
END;
