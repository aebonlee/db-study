# 3교시 - 오라클 데이터 저장 구조와 블록 기반 I/O 메커니즘

### 🎯 학습 목표

- 오라클의 데이터 저장 단위인 블록(Block), 익스텐트(Extent), 세그먼트(Segment)의 구조를 이해한다.
- 테이블의 저장 방식이 SQL 수행 성능에 미치는 영향을 설명할 수 있다.
- 클러스터링 팩터(Clustering Factor)의 개념과 SQL 튜닝과의 연관성을 학습한다.
- Random I/O와 Sequential I/O의 성능 차이를 실습으로 비교한다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “앞 시간에 SQL이 어떻게 공유되고 파싱되는지 알아봤습니다.
    
    이제 SQL이 데이터에 실제 접근할 때, Oracle은 물리적으로 어떤 방식으로 데이터를 저장하고 접근하는지 살펴보겠습니다.
    
    이 구조를 이해해야만 인덱스 전략, 파티셔닝, 힌트 등이 왜 필요한지를 명확히 알 수 있습니다.”
    

---

1. **[Oracle 데이터 저장 구조 설명 – 15분]**
    - Oracle은 데이터를 다음과 같은 계층으로 저장:
        
        ```
        Segment → Extent → Bl%oc
        ```
        
        - **Block**: Oracle에서 I/O의 최소 단위 (보통 8KB)
        - **Extent**: 연속된 블록 묶음
        - **Segment**: 테이블 또는 인덱스 전체를 의미
    - 한 개의 행을 검색할 때도 반드시 최소 1블록이 메모리로 로드됨 → 블록 단위 접근 중요
    - 대량 데이터가 분산되어 저장되면 I/O 발생량 증가 → 성능 저하

---

1. **[Random vs Sequential I/O 설명 – 10분]**
    - **Random I/O**: 디스크 헤드가 여러 곳을 움직이며 특정 블록을 찾아야 함 → 시간 소요 큼
        - 주로 인덱스 스캔 + 테이블 RowID 접근 시 발생
    - **Sequential I/O**: 블록이 연속적으로 배치된 상태에서 읽음 → 빠름
        - Full Table Scan, Index Full Scan 등에서 발생
    - **중요:** Oracle은 항상 Block 단위로 데이터를 읽고 쓰기 때문에 “한 행만 읽더라도 최소 1Block I/O가 발생”한다는 개념 필수

---

1. **[클러스터링 팩터(Clustering Factor) 설명 – 10분]**
    - 클러스터링 팩터란?
        
        → 인덱스의 순서와 테이블 내 실제 데이터 저장 순서가 얼마나 유사한지를 나타내는 수치
        
        - 낮을수록 효율적 (인덱스를 따라가면 데이터도 인접하게 존재)
        - 높을수록 Random I/O 많아짐
    - 예시 설명:
        - 고객 ID 기준으로 정렬된 테이블: 인덱스 효율 높음
        - 고객 이름 기준으로 랜덤하게 저장된 테이블: 클러스터링 팩터 높음 → 성능 저하

---

1. **[실습 – I/O 유형 비교 및 클러스터링 팩터 확인 – 15분]**
    1. 🔸 테이블 생성 및 인덱스 생성
        
        ```sql
        CREATE TABLE CUST (
          CUST_ID NUMBER,
          NAME VARCHAR2(100),
          REGION VARCHAR2(10)
        );
        
        BEGIN
          FOR i IN 1..10000 LOOP
            INSERT INTO CUST VALUES (i, '홍길동'||i, CASE WHEN MOD(i, 2)=0 THEN '서울' ELSE '부산' END);
          END LOOP;
        END;
        /
        CREATE INDEX IDX_CUST_REGION ON CUST(REGION);
        ```
        
    2. 🔸 실행계획 비교
        
        ```sql
        SET AUTOTRACE ON
        SELECT * FROM CUST WHERE REGION = '부산';
        ```
        
    3. 🔸 클러스터링 팩터 확인
        
        ```sql
        SELECT INDEX_NAME, TABLE_NAME, CLUSTERING_FACTOR, NUM_ROWS, LEAF_BLOCKS
        FROM USER_INDEXES
        WHERE INDEX_NAME = 'IDX_CUST_REGION';
        ```
        
    
    👉 설명: 테이블이 REGION 값 기준으로 정렬되지 않아 클러스터링 팩터가 높게 나올 것
    

---

1. **[정리 및 마무리 – 5분]**
    - 블록 단위 저장 구조는 SQL 수행 속도에 결정적인 영향을 준다.
    - **Random I/O를 줄이고, Sequential I/O로 바꾸는 전략**이 바로 SQL 튜닝의 핵심이다.
    - 클러스터링 팩터는 인덱스 효율성 판단의 핵심 지표이다.
    - “모든 SQL 튜닝은 결국 I/O 튜닝이다. Oracle이 무엇을, 얼마나, 어떻게 읽느냐를 통제해야 성능이 나온다.”

### 📌 추가 실습 필요

- 실습 SQL 예시
    
    ```sql
    SELECT * FROM customer WHERE color = 'RED';  -- Full Scan 유도
    CREATE INDEX idx_color ON customer(color);
    SELECT /*+ INDEX(customer idx_color) */ * FROM customer WHERE color = 'RED'; -- Index Scan 유도
    ```
    
- 실습 데이터: 클러스터링 팩터 차이를 유도하기 위한 인위적 데이터 정렬 패턴
- 시각화 자료: 클러스터링 팩터 좋은/나쁜 예 비교 이미지

### 📝 과제

1. CUST 테이블을 다른 기준(예: CUST_ID 기준)으로 인덱스를 만들고, 클러스터링 팩터를 비교하라.
2. `SELECT * FROM CUST WHERE REGION = '서울'` 실행계획을 보고 I/O 효율성 판단해보기
3. 클러스터링 팩터가 낮아지도록 테이블 재정렬 후 재측정 시도
