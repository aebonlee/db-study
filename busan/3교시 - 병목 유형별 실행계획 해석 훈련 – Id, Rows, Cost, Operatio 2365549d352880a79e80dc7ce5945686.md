# 3교시 - 병목 유형별 실행계획 해석 훈련 – Id, Rows, Cost, Operation별 정밀 해석법

### 🎯 학습 목표

- SQL 실행계획의 핵심 요소(`Id`, `Rows`, `Cost`, `Operation`)를 정확히 해석할 수 있다.
- 실행계획 계층 구조를 기반으로 병목 구간(비용 집중 구간)을 판별할 수 있다.
- 액세스 방식(Index Scan, Full Scan, Nested Loop 등)과 연결 전략의 병목을 숫자 기반으로 분석할 수 있다.
- 실습을 통해 SQL 실행계획에서 성능 문제를 “문장 없이 숫자로” 설명할 수 있다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “SQL 튜닝에서 실행계획은 항상 등장합니다.
    
    하지만 문제는 **숫자를 읽지 않고 단순 구조만 보는 경우가 많다는 것**입니다.
    
    오늘은 실행계획의 `Id`, `Cost`, `Rows`, `Bytes`, `Operation`의 의미를 하나하나 정확히 해석하고,
    
    **실행계획만 보고 병목을 판단하는 훈련**을 진행합니다.”
    

---

1. **[실행계획 기본 항목 정리 – 10분]**
    
    
    | 항목 | 의미 | 해석 포인트 |
    | --- | --- | --- |
    | `Id` | 수행 순서/트리 구조 | 항상 아래에서 위로 실행 |
    | `Operation` | 수행 연산 종류 | Index Scan, Full Scan 등 |
    | `Object Name` | 액세스 대상 객체 | 테이블, 인덱스 |
    | `Rows` | 예측 결과 행 수 | Cardinality 추정 (통계 기반) |
    | `Bytes` | 반환 데이터 크기 | 네트워크 전송 부하 |
    | `Cost` | 추정 수행 비용 | 옵티마이저가 선택 기준으로 삼음 |
    | `Access Predicates` | 인덱스 조건 | Index 사용 여부 |
    | `Filter Predicates` | 나중 필터 조건 | Full Scan 후 조건 적용 |
    
    ➤ Tip: Rows가 급격히 증가하는 구간 = 병목 가능성
    

---

1. **[계층 구조 해석법 – 10분]**
    - `Id`가 클수록 하위 → 먼저 실행됨
    - 하위 연산자에서 상위로 행(Row)이 전달됨
    - 병목 진단 포인트:
        
        ```
        Id  Operation                      Rows  Cost
        0   SELECT STATEMENT              100K   540
        1    NESTED LOOPS                 100K   540
        2     TABLE ACCESS FULL EMP       100K   120
        3     TABLE ACCESS BY INDEX ROWID DEPT  100K   420
        ```
        
        ➤ Row가 너무 많고 Cost가 특정 Id에 집중되면 해당 연산이 병목
        

---

1. **[실습 – 실행계획 병목 지점 찾기 실습] – 15분**
    
    🔸 SQL 예제:
    
    ```sql
    SELECT E.ENAME, D.DNAME
    FROM EMP E JOIN DEPT D ON E.DEPTNO = D.DEPTNO
    WHERE TO_CHAR(E.HIREDATE, 'YYYY') = '1982';
    ```
    
    🔸 실행계획 도출:
    
    ```sql
    EXPLAIN PLAN FOR ...
    SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
    ```
    
    🔸 분석 훈련:
    
    - TO_CHAR로 인한 Full Scan
    - Rows 수가 지나치게 많음
    - Index Scan 안 되는 이유 확인
    - Filter Predicate vs Access Predicate 구분

---

1. **[실행계획 병목 유형 정리 – 10분]**
    
    
    | 병목 유형 | 대표 징후 | 개선 포인트 |
    | --- | --- | --- |
    | Full Scan 발생 | TABLE ACCESS FULL, Disk Read↑ | 인덱스 생성/적용, 조건절 변경 |
    | Rows 증가 급격 | Id 증가마다 Rows 배↑ | 조인 순서 조정, 필터 조건 재설계 |
    | Filter Predicate 사용 | 인덱스 미적용 | 조건절 구조 변경, 함수 제거 |
    | Index Skip Scan | 후행컬럼만 조건 | 인덱스 재구성 또는 힌트 적용 |
    | Nested Loop 반복 | Rows 많음 + Loop | Hash Join, 병렬처리 유도 |

---

1. **[정리 및 마무리 – 5분]**
    - 실행계획은 단지 구조가 아니라, **행과 비용의 흐름을 숫자로 보여주는 로직 해석서**
    - 병목은 '구조'보다 '숫자'에 있다
    - “SQL 튜닝은 실행계획을 '읽는 것'이 아니라,
        
        그 실행계획이 **왜 그렇게 됐는지를 해석하는 일**이다.”
