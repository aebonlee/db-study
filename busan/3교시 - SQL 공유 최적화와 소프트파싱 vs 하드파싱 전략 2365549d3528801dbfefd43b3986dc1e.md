# 3교시 - SQL 공유 최적화와 소프트파싱 vs 하드파싱 전략

### 🎯 학습 목표

- SQL 실행 시 발생하는 파싱 과정의 종류(하드파싱, 소프트파싱)를 구분할 수 있다.
- Oracle의 SQL 공유 구조(Library Cache)의 동작 원리를 이해하고 성능에 미치는 영향을 분석할 수 있다.
- 바인드 변수 사용이 SQL 재사용과 파싱 비용 절감에 어떤 영향을 주는지 실습을 통해 확인한다.
- `V$SQL`, `V$SQLAREA` 뷰를 활용하여 SQL 공유 상태를 확인하고 진단할 수 있다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “여러분이 작성하는 SQL은 과연 Oracle 내부에서 어떻게 저장되고 얼마나 재활용될까요?
    
    오늘 수업에서는 SQL 공유의 원리를 알아보고, 성능 저하의 핵심 원인 중 하나인 하드파싱의 실체를 확인합니다.
    
    특히 바인드 변수를 어떻게 사용하느냐에 따라 수천, 수만 개의 SQL이 발생할 수도 있고, 단 하나의 SQL로도 끝낼 수도 있습니다.”
    

---

1. **[SQL 파싱의 종류 설명 – 10분]**
    
    🔹 **하드파싱(Hard Parsing)**
    
    - SQL이 처음 등장하거나, 공유 커서가 없을 경우
    - 구문 분석, 권한 체크, 최적화, 실행계획 생성 등 전 과정을 거침
    - CPU 자원 소모 큼, 라이브러리 캐시 락 발생, 대기 시간 증가
    
    🔹 **소프트파싱(Soft Parsing)**
    
    - 동일한 SQL이 Library Cache에 존재할 경우
    - 실행계획을 재사용하고 빠르게 실행됨
    
    ➤ 하드파싱을 줄이고 소프트파싱을 유도하는 것이 고성능 SQL의 핵심
    

---

1. **[Library Cache 및 커서 공유 구조 – 10분]**
    - SQL이 캐시에 저장되는 방식: 해시 버킷 → 커서 → 실행계획
    - 커서 종류:
        - Shared Cursor: 같은 SQL + 같은 실행계획
        - Child Cursor: 같은 SQL이지만 바인드 값, 통계, 파라미터 차이로 실행계획이 다를 때 생성
    - 커서 공유 실패 원인:
        - Literal SQL
        - NLS 파라미터 차이
        - 옵티마이저 모드 차이
        - 바인드 변수 미사용

---

1. **[실습 – 바인드 변수 미사용 vs 사용 비교 – 15분]**
    
    🔸 하드파싱 유발 SQL
    
    ```sql
    ALTER SYSTEM FLUSH SHARED_POOL;
    
    SELECT * FROM EMP WHERE EMPNO = 7369;
    SELECT * FROM EMP WHERE EMPNO = 7499;
    SELECT * FROM EMP WHERE EMPNO = 7521;
    ```
    
    🔸 바인드 변수 사용 SQL
    
    ```sql
    VARIABLE v_empno NUMBER
    EXEC :v_empno := 7369;
    SELECT * FROM EMP WHERE EMPNO = :v_empno;
    ```
    
    🔸 커서 공유 상태 확인
    
    ```sql
    SELECT SQL_TEXT, EXECUTIONS, PARSE_CALLS, CHILD_NUMBER
    FROM V$SQLAREA
    WHERE SQL_TEXT LIKE 'SELECT * FROM EMP WHERE EMPNO%';
    ```
    
    ➤ 실행 횟수 대비 파싱 횟수 비교
    
    ➤ Literal SQL은 각기 다른 커서 생성, 바인드 SQL은 공유됨
    

---

1. **[Cursor Sharing 정책 및 설정 – 5분]**
    - `CURSOR_SHARING = EXACT` (기본값) → 문자까지 동일해야 공유
    - `FORCE`: 옵티마이저가 내부적으로 바인드처럼 처리
    - `SIMILAR`: 11g 이하에서 사용되던 방식, 비추천
    
    ```sql
    ALTER SYSTEM SET CURSOR_SHARING = FORCE;
    ```
    
    ➤ Literal SQL이 많은 시스템에서 FORCE 설정 시 하드파싱 급감
    

---

1. **[마무리 및 정리 – 5분]**
    - SQL은 하나의 문장처럼 보이지만 내부에서는 수천 개의 커서로 존재할 수 있다
    - 바인드 변수는 성능뿐 아니라 Library Cache 관리, CPU 사용량에 직결된다
    - “성능 좋은 SQL은 잘 쓴 SQL이 아니라, 잘 공유되는 SQL이다.”
