# 1교시 - 조인 튜닝 심화 – 다중 조인, Outer Join, 뷰 기반 조인 튜닝 전략

### 🎯 학습 목표

- 3개 이상의 테이블이 조인되는 상황에서 옵티마이저가 선택하는 조인 순서를 해석할 수 있다.
- Outer Join(LEFT, RIGHT OUTER)의 튜닝 특성과 옵티마이저가 이를 처리하는 방식의 차이를 이해한다.
- 인라인 뷰 또는 뷰 기반 조인이 튜닝에 어떤 영향을 미치는지 분석할 수 있다.
- 조인 순서와 조건의 위치에 따라 실행계획과 성능이 어떻게 달라지는지를 실습을 통해 비교한다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “SQL에서 조인은 한 번만 쓰지 않습니다. 대부분의 실무 SQL은 3개 이상의 테이블이 조인되며, 때로는 뷰나 인라인 뷰를 통해 간접적으로 연결되기도 합니다.
    
    오늘은 복합 조인 구조에서 옵티마이저가 조인 순서를 어떻게 선택하고, OUTER JOIN이 성능에 어떤 영향을 미치는지를 확인합니다.
    
    SQL 튜닝의 수준을 한 단계 더 높이는 시간이 될 것입니다.”
    

---

1. **[다중 조인 시 옵티마이저 조인 순서 결정 방식 – 10분]**
    - 옵티마이저는 **조건이 많은 테이블**, **행 수가 적은 테이블**, **선행 필터링 효과가 높은 테이블**을 먼저 수행하려 한다.
    - Nested Loop 조인은 Driving Table 선정이 매우 중요
    - 옵티마이저는 Cost 기반으로 조인 순서를 자동 선택하지만, 비정상적인 실행계획이 나오는 경우도 많음
    
    🔸 실행계획에서 `Id`가 가장 높은 테이블이 **가장 마지막에 수행**됨
    
    🔸 LEADING 힌트로 조인 순서 강제 가능:
    
    ```sql
    SELECT /*+ LEADING(D E S) */ ...
    ```
    

---

1. **[Outer Join의 튜닝 포인트 설명 – 10분]**
    - OUTER JOIN은 옵티마이저가 **조인 순서를 강제로 고정**해야 하므로, 조인 재정렬이 불가능
    - OUTER JOIN 테이블이 필터링 대상이 되면 성능이 나빠질 수 있음
    - **가능하면 OUTER JOIN은 필요한 경우에만 사용**하고, 뷰 등으로 우회 구조 사용 검토
    
    🔸 예제:
    
    ```sql
    SELECT E.ENAME, D.DNAME
    FROM EMP E LEFT OUTER JOIN DEPT D ON E.DEPTNO = D.DEPTNO
    WHERE D.LOC = 'CHICAGO';
    ```
    
    ➤ OUTER JOIN 이후 필터링 조건이 OUTER 대상 테이블에 존재 → OUTER JOIN 의미 무효화 → 튜닝 포인트
    

---

1. **[실습 – 다중 조인 SQL 튜닝 전후 비교] – 15분**
    
    🔸 실습 SQL 1 – 뷰 기반 조인 (비효율 구조)
    
    ```sql
    SELECT E.ENAME, D.DNAME, S.GRADE
    FROM EMP E
      JOIN DEPT D ON E.DEPTNO = D.DEPTNO
      JOIN (SELECT * FROM SALGRADE WHERE LOSAL < 2000) S
        ON E.SAL BETWEEN S.LOSAL AND S.HISAL;
    ```
    
    🔸 실습 SQL 2 – 튜닝된 구조
    
    ```sql
    SELECT /*+ USE_NL(E S) USE_HASH(D) */ E.ENAME, D.DNAME, S.GRADE
    FROM SALGRADE S
      JOIN EMP E ON E.SAL BETWEEN S.LOSAL AND S.HISAL
      JOIN DEPT D ON E.DEPTNO = D.DEPTNO;
    ```
    
    🔸 실행계획 비교
    
    ```sql
    EXPLAIN PLAN FOR ...
    SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
    ```
    
    ➤ 조인 순서, Access Type, COST 수치 비교
    

---

1. **[인라인 뷰와 뷰 Merge/Materialize 전략 – 5분]**
    - 옵티마이저는 인라인 뷰를 **Merge** 또는 **Materialize**로 처리
    - WHERE 절에서 인라인 뷰 조건을 최대한 밖으로 끌어내야 Merge 가능 → 성능 향상
    - 힌트: `MERGE`, `NO_MERGE` 등으로 제어 가능

---

1. **[정리 및 마무리 – 5분]**
    - 다중 조인은 옵티마이저 판단의 자유도가 줄어들 수록 성능 저하 가능성 ↑
    - OUTER JOIN은 반드시 필요한 경우만 사용하고, 조건의 위치를 신중하게 고려해야 함
    - 뷰는 가독성을 높이지만 튜닝을 어렵게 만들 수 있음
    - “복잡한 SQL일수록 튜닝의 중심은 **조인 순서와 필터링 위치**다.”

### 📝 과제

2. OUTER JOIN에서 조인 후 조건절이 OUTER 테이블에 있을 경우와 INNER 테이블에 있을 경우 성능 차이 실험
3. 동일 SQL을 인라인 뷰와 일반 JOIN으로 구성하고 Merge 여부 및 성능 비교
