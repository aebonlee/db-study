# Day05 - 튜닝 포인트 진단 및 최적화 시나리오 수립 훈련

## 1. 튜닝 포인트란 무엇인가?

데이터베이스에서  **“튜닝 포인트”** 란 속도를 느리게 만드는 **병목 지점**을 뜻합니다.

마치 고속도로에서 차가 막혀 있는 구간을 찾아내는 것과 같습니다. 

이 지점을 정확히 찾으면, **도로를 넓히거나 우회도로를 만드는 방식**으로 문제를 해결할 수 있습니다.

---

## 2. 진단(현황 파악) 절차 6단계

| 단계 | 핵심 질문 | 대표 명령·도구 |
| --- | --- | --- |
| ① 목표 설정 | “무엇이 느린가?” (예: 주문 조회 화면이 3 초 이상) | 없음 |
| ② 측정 | “느림의 증거가 있는가?” | `SELECT DB_TIME, CPU_TIME … FROM v$sysmetric;` |
| ③ 상위 대기 이벤트 확인 | “어디서 막히는가?” | `SELECT event, time_waited FROM v$system_event ORDER BY time_waited DESC;` |
| ④ 상위 SQL 추출 | “어떤 SQL이 가장 오래 걸리나?” | AWR 리포트, `v$sql` |
| ⑤ 실행 계획 분석 | “계획(Plan)이 비효율적인가?” | `EXPLAIN PLAN FOR …` + `DBMS_XPLAN.DISPLAY` |
| ⑥ 원인 가설 수립 | “인덱스 부재? 통계 미갱신? 조인 순서 불량?” | 문제별 점검 체크리스트 |

> • **DB_TIME** = 전체 운행 시간
> 
> 
> • **EVENT** = 정체 구간 이름
> 
> • **SQL_ID** = 정체 구간에서 서행 중인 특정 자동차
>

## 3. 최적화(해결) 시나리오 수립 절차

1. **가설 작성**
    
    “`orders` 테이블에 날짜 함수가 있어 인덱스를 못 쓴다”
    
2. **해결 방법 설계**
    - 인덱스 생성 혹은 함수 제거 쿼리 재작성
    - 통계정보(`DBMS_STATS`) 갱신
3. **변경 사항 적용** (테스트 DB → 운영 DB)
4. **재측정 & 검증**
    - AWR 이전·이후 **DB Time** 비교
    - 실행 계획이 **Full Scan → Index Range Scan**으로 변했는지 확인
5. **문서화**
    - 조치 내역, 효과(응답 시간 40 % 감소) 기록
6. **사후 모니터링** (1 주일)
    - 지표가 다시 악화되면 롤백 또는 추가 조치

## 4. 단계별 **실습 예제**

### 4-1. 문제 상황 재현

```sql
-- 1) 주문 테이블 (100만 건) 준비
CREATE TABLE orders (
    order_id   NUMBER PRIMARY KEY,
    order_date DATE,
    amount     NUMBER
);

-- 2) 데이터 채우기 (임의 생성 예시)
DECLARE
    v_date DATE := DATE '2024-01-01';
BEGIN
    FOR i IN 1 .. 1000000 LOOP
        INSERT INTO orders VALUES (i, v_date + MOD(i, 365), DBMS_RANDOM.VALUE(10, 500));
    END LOOP;
    COMMIT;
END;
/

-- 3) 인덱스 없이 실행
EXPLAIN PLAN FOR
SELECT * FROM orders
WHERE TO_CHAR(order_date, 'YYYY-MM') = '2024-05';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**예상 결과**

- **TABLE ACCESS FULL** → 전체를 훑어 시간이 오래 걸림.

### 4-2. 첫 번째 해결책: 함수 제거

```sql
EXPLAIN PLAN FOR
SELECT * FROM orders
WHERE order_date BETWEEN DATE '2024-05-01' AND DATE '2024-05-31';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

- 함수(`TO_CHAR`)를 없애면 **조건 컬럼(order_date)** 가 그대로 사용되어 **범위 스캔** 후보가 생김.

### 4-3. 두 번째 해결책: 인덱스 추가

```sql
CREATE INDEX idx_orders_date ON orders(order_date);

EXPLAIN PLAN FOR
SELECT * FROM orders
WHERE order_date BETWEEN DATE '2024-05-01' AND DATE '2024-05-31';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

- **INDEX RANGE SCAN** 으로 변경되며 I/O 횟수 급감.

### 4-4. 성능 전·후 비교

| 구분 | 블록 읽기 수 | 실행 시간(초) |
| --- | --- | --- |
| Full Scan | 30,000 | 5.2 |
| Function 제거만 | 30,000 | 5.2 |
| + 인덱스 사용 | **1,200** | **0.4** |

> 인덱스는 **책의 목차**와 같아서 원하는 페이지를 바로 찾게 해 줍니다. 
목차가 없으면, 책 전체를 넘겨야 하므로 시간이 오래 걸립니다.

## 5. 추가 연습 과제

1. **조인 순서 튜닝**
    - `employees` vs `departments` 테이블을 다양한 힌트(`LEADING`, `USE_NL`, `USE_HASH`)로 실행해 보고 **실행-계획 비용**을 비교하십시오.
2. **통계정보 최신화 실습**
    
    ```sql
    EXEC DBMS_STATS.GATHER_TABLE_STATS(USER, 'ORDERS', CASCADE => TRUE);
    ```
    
3. **AWR 스냅샷 분석**
    - 튜닝 전 & 후 스냅샷을 생성하여 **Top SQL**과 **Top Wait Event** 변화를 확인하십시오.

## 6. 마무리

- **진단**은 “어디가 느린가?”를 **측정**과 **분석**으로 찾는 단계입니다.
- **최적화 시나리오**는 “어떻게 고칠까?”를 **가설-검증** 방식으로 실행하는 단계입니다.
- 늘 **변경 전후** 수치를 기록하여 효과를 증명하시기 바랍니다.

이 과정을 반복적으로 연습하면, 데이터베이스 속도를 **직접 체감**하며 튜닝 감각을 키울 수 있습니다.
