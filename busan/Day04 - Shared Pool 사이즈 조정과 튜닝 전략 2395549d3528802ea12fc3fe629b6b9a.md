# Day04 - Shared Pool 사이즈 조정과 튜닝 전략

### "Shared Pool 사이즈 조정과 튜닝 전략(V$SHARED_POOL_ADVICE 기반)"을 주제로 실제 튜닝 관점에서 **사이즈 분석 → 성능 영향 진단 → 조정 전략**을 실습합니다.

### 🎯 학습 목표

- Shared Pool의 역할과 성능 영향 요소를 이해한다.
- `V$SHARED_POOL_ADVICE` 뷰를 통해 적절한 크기를 예측한다.
- SQL, PL/SQL, 메타데이터 캐시, Result Cache 등 구성요소의 활용도를 분석한다.
- Shared Pool과 Library Cache, Dictionary Cache의 Hit Ratio를 확인하고 개선 전략을 수립한다.

---

## 📘 이론 요약

| 구성 요소 | 설명 |
| --- | --- |
| **Shared Pool** | SQL, PL/SQL, Data Dictionary 등 캐싱하는 메모리 영역 |
| **Library Cache** | SQL, 실행계획, PL/SQL 코드가 저장됨 (Hard/Soft Parsing에 영향) |
| **Data Dictionary Cache** | 오브젝트, 권한, 사용자 정보 등 메타데이터 저장 |
| **V$SHARED_POOL_ADVICE** | 사이즈 변경 시 예상 성능 효과 예측 |

---

## 🧪 실습 1: Shared Pool 사용 현황 확인

```sql
-- 전체 Shared Pool 크기 및 구성 요소 확인
SELECT * FROM v$sgastat WHERE pool = 'shared pool';

-- Library Cache Hit Ratio 확인
SELECT namespace, gethitratio
FROM v$librarycache;

-- Data Dictionary Cache Hit Ratio
SELECT parameter, gets, gethits,
ROUND(gethits/gets*100,2) AS hit_ratio
FROM v$rowcache;
```

👉 **Hint**: Hit Ratio가 90% 이하이면 사이즈 부족 가능성 있음

---

## 🧪 실습 2: Shared Pool Advice 분석

```sql
-- 공유풀 크기 조정에 따른 예상 성능 분석
SELECT
  shared_pool_size_for_estimate "SIZE_MB",
  shared_pool_size_factor "FACTOR",
  estd_lc_size "LIB_CACHE_MB",
  estd_lc_memory_objects "OBJ_CNT",
  estd_lc_time_saved "TIME_SAVED",
  estd_lc_time_saved_factor "TIME_FACTOR"
FROM v$shared_pool_advice
ORDER BY shared_pool_size_for_estimate;
```

👉 분석 기준:

- TIME_SAVED가 크면 메모리 증설로 효과 기대
- FACTOR 1.2 ~ 1.5 구간에서 가장 효율적일 수 있음

---

## 🧪 실습 3: 실시간 SQL Aging 확인

```sql
-- SQL Aging 상태 확인 (자주 Age Out 되는지 여부 확인)
SELECT sql_id, executions, loads, invalidations, sharable_mem
FROM v$sql
WHERE loads > 1
ORDER BY loads DESC;
```

👉 `loads > 1`은 재사용되지 못하고 메모리에서 퇴출되었다는 의미

---

## 🧪 실습 4: Shared Pool 크기 조정

```sql
-- 현재 크기 확인
SHOW PARAMETER shared_pool_size;

-- 예: 512MB → 768MB로 조정
ALTER SYSTEM SET shared_pool_size = 768M SCOPE = BOTH;

-- ASMM을 사용하는 경우에는 shared_pool_size가 최소값 역할
```

👉 `SGA_TARGET` 또는 `MEMORY_TARGET` 사용 시 조정 방식은 다름
