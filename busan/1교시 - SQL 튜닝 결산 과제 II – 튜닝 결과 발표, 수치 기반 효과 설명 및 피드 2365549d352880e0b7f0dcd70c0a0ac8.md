# 1교시 -  SQL 튜닝 결산 과제 II – 튜닝 결과 발표, 수치 기반 효과 설명 및 피드백

### 🎯 학습 목표

- 수강생이 직접 수행한 SQL 튜닝 결과를 발표하고 전후 실행계획과 지표의 차이를 설명할 수 있다.
- 튜닝의 논리적 근거와 수치 기반 개선 결과를 커뮤니케이션할 수 있다.
- 다양한 튜닝 접근 방식과 시각을 서로 공유하고, 피드백을 통해 전략을 보완할 수 있다.
- SQL 튜닝 보고서 작성 역량과 실무 적용 가능성을 점검받는다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[오프닝 – 5분]**
    
    “이제 40시간 SQL 튜닝 과정의 마지막을 향해 가고 있습니다.
    
    여러분이 직접 분석하고 리팩토링한 SQL을 오늘 발표하고,
    
    **그 효과를 수치로 얼마나 정확히 설명할 수 있는지** 점검해보겠습니다.
    
    지금부터는 단순한 발표가 아니라, 실무 회의에서 **내가 튜닝한 SQL의 효과를 설명하는 것**이라고 생각하고 참여해 주세요.”
    

---

1. **[발표 순서 안내 및 양식 공유 – 5분]**
    
    ▶ 발표 순서: 조별 또는 개인별 발표 (3~5분 이내)
    
    ▶ 발표 내용:
    
    - ✅ 튜닝 전 SQL 구조 설명
    - ✅ 병목 원인 분석 (실행계획, COST, Buffer Gets 등)
    - ✅ 개선 SQL 설명 및 변경 포인트
    - ✅ 전후 성능 비교표 (COST, ELAPSED TIME, DISK READS 등)
    - ✅ 종합 소감 또는 적용 가능성 평가
    
    📄 추천 양식:
    
    ```
    - 대상 SQL 설명:
    - 병목 원인:
    - 개선 포인트:
    - 튜닝 전/후 비교:
      · COST: 220 → 40
      · BUFFER GETS: 12,000 → 900
      · EXECUTIONS: 동일 (10회)
    - 결론 및 실무 적용성:
    ```
    

---

1. **[튜닝 결과 발표 및 질의응답 – 30분]**
    
    ▶ 각 조(또는 개인) 발표 – 최대 5분 × 5~6명
    
    ▶ 튜터/강사 피드백 (예시):
    
    - “이 SQL은 Join 순서만 바꿔도 Cost가 크게 줄 수 있었던 좋은 예시입니다.”
    - “조건절 안에서 LIKE '%값'을 개선한 점이 특히 중요했습니다.”
    - “단지 실행시간만 줄인 것이 아니라, 공유 가능한 구조로 바꿨다는 점이 인상적입니다.”
    
    ➤ 발표 중 중요 내용은 화이트보드 또는 노션 공유 게시판에 기록
    

---

1. **[우수 사례 소개 및 추가 설명 – 5분]**
    - Buffer Gets 95% 이상 절감
    - Index 전략 개선으로 Full Scan 제거
    - Cursor Sharing 실패 → FORCE 설정으로 전환하여 Soft Parsing 유도
    
    🏆 우수 발표자 선정 시:
    
    - 실무 도입 가능성 높은 개선안
    - 성능 개선 수치가 명확하고 구조 변경 이유 설명이 설득력 있음

---

1. **[정리 및 마무리 – 5분]**
    - 수치로 설명할 수 없는 튜닝은 신뢰받기 어렵다
    - 오늘 발표를 통해 SQL 튜닝 결과를 **보고서화 + 발표 + 설득**하는 실무 역량까지 학습한 것
    - “SQL 튜닝의 핵심은 수치로 증명하고, 구조로 설명하는 능력이다.
        
        튜닝은 기술이 아니라 커뮤니케이션이다.”

### 📝 과제 (강의 후 선택형)

1. 발표한 SQL 튜닝 결과를 PPT 또는 A4 문서 형태로 정리
2. 개선 SQL을 공유 게시판에 업로드하고, 타인 SQL에 피드백 1회 작성
3. 실제 업무에서 유사한 SQL이 있을 경우 적용 계획 메모 제출
