# 5교시 - 실무 SQL 튜닝 사례분석 I – 비즈니스 로직 기반 복합 SQL 구조 해석

- 튜닝 대상 SQL을 기능적으로 해석한 후, 성능 관점에서 리팩토링 전략을 수립할 수 있다.
- SQL 구조와 도메인 지식이 결합된 상황에서의 튜닝 기법을 익힌다.

1. **[사례 제공 – 복합 SQL 구조 설명] – 5분**
    
    🔸 예시 SQL: 고객별 거래 통계 + 등급 산정 + 이상 거래 패턴 필터
    
    ```sql
    SELECT C.CUST_ID, C.CUST_NAME,
           (SELECT COUNT(*) FROM TRANS T WHERE T.CUST_ID = C.CUST_ID AND T.TRANS_DT BETWEEN SYSDATE - 90 AND SYSDATE) AS RECENT_CNT,
           (SELECT MAX(AMOUNT) FROM TRANS WHERE CUST_ID = C.CUST_ID) AS MAX_AMOUNT,
           CASE WHEN EXISTS (
               SELECT 1 FROM BLOCK_LIST B WHERE B.CUST_ID = C.CUST_ID
           ) THEN 'Y' ELSE 'N' END AS IS_BLOCKED
    FROM CUSTOMER C
    WHERE C.STATUS = 'ACTIVE'
      AND EXISTS (
          SELECT 1 FROM TRANS T WHERE T.CUST_ID = C.CUST_ID AND T.AMOUNT > 500000
      )
      AND C.CUST_TYPE = 'VIP';
    ```
    

---

1. **[구조 해석 – 기능별 블록 나누기] – 10분**
    - 기능 단위로 구분:
        - **SELECT 절**: 스칼라 서브쿼리 (집계, Max, Count)
        - **CASE WHEN EXISTS**: 제어 구조
        - **WHERE절**: EXISTS + 조건
    - 성능 저하 원인 가능성:
        - 스칼라 서브쿼리 반복 실행 (고객 수만큼 서브쿼리 N회)
        - `EXISTS` 안에 추가 SELECT
        - 고객 기준으로 Join 안 쓰고 Subquery만 사용함

---

1. **[실행계획 분석 – 병목 확인] – 10분**
    - 실행계획 도출
        
        ```sql
        EXPLAIN PLAN FOR ...
        SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
        ```
        
    - 주요 병목 확인:
        - `TABLE ACCESS FULL TRANS` 반복
        - `FILTER` 연산 반복
        - Buffer Gets 급증 확인
        - Index 미사용 (조건절 내 연산 구조)

---

1. **[튜닝 전략 수립 – 구조 리팩토링 및 조인 방식 변경] – 15분**
    
    🔧 리팩토링 SQL (JOIN 방식으로 통합)
    
    ```sql
    SELECT C.CUST_ID, C.CUST_NAME,
           COUNT(T1.TRANS_ID) AS RECENT_CNT,
           MAX(T2.AMOUNT) AS MAX_AMOUNT,
           CASE WHEN B.CUST_ID IS NOT NULL THEN 'Y' ELSE 'N' END AS IS_BLOCKED
    FROM CUSTOMER C
    LEFT JOIN TRANS T1 ON C.CUST_ID = T1.CUST_ID AND T1.TRANS_DT BETWEEN SYSDATE - 90 AND SYSDATE
    LEFT JOIN TRANS T2 ON C.CUST_ID = T2.CUST_ID
    LEFT JOIN BLOCK_LIST B ON C.CUST_ID = B.CUST_ID
    WHERE C.STATUS = 'ACTIVE'
      AND C.CUST_TYPE = 'VIP'
      AND EXISTS (
          SELECT 1 FROM TRANS TX WHERE TX.CUST_ID = C.CUST_ID AND TX.AMOUNT > 500000
      )
    GROUP BY C.CUST_ID, C.CUST_NAME, B.CUST_ID;
    ```
    
    - 스칼라 서브쿼리 제거 → JOIN + GROUP BY 전환
    - INDEX 적용 가능 조건절로 재설계
    - 블록 여부 확인도 OUTER JOIN으로 처리

1. **[정리 및 마무리 – 5분]**
    - 실무 SQL은 복잡하고, 성능 이슈는 구조 안에 숨겨져 있다
    - Subquery는 깔끔해 보이지만 반복 구조는 병목
    - “튜닝은 문장 개선이 아니라,
        
        **구조와 의미를 해석한 뒤, 재설계하는 행위**다.”
