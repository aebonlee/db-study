# 2교시 - 튜닝 패턴 회고 – SQL 튜닝 전략 유형별 정리 및 조건절/힌트/인덱스 최종 정리

### 🎯 학습 목표

- SQL 튜닝에서 자주 발생하는 문제 유형을 정리하고, 각 유형별 핵심 전략을 체계적으로 정리한다.
- 조건절, 인덱스, 힌트, 조인, 서브쿼리 등 핵심 영역별로 SQL 튜닝의 핵심 기법을 복습한다.
- 수강생이 자신의 SQL 스타일에서 발생하기 쉬운 비효율 패턴을 파악하고 개선 전략을 제시할 수 있다.
- 실무 튜닝에서 바로 적용 가능한 ‘SQL 튜닝 체크리스트’를 정리해 둔다.

1. **[유형별 튜닝 전략 표 정리 – 15분]**
    
    
    | 문제 유형 | 원인 | 핵심 전략 |
    | --- | --- | --- |
    | 조건절에 함수 사용 | 인덱스 무력화 | 조건절 재작성, 함수 기반 인덱스 |
    | LIKE '%aaa' | Prefix Index 미사용 | Full Text Index, 구조 변경 |
    | OR 조건 | Index Merge 또는 Full Scan 유도 | `UNION ALL`, `IN`으로 전환 |
    | NOT IN | NULL 이슈, 인덱스 미사용 | `NOT EXISTS` 또는 `ANTI JOIN` |
    | 다중 조인 | Join 순서 문제 | 옵티마이저 힌트, 통계 기반 Join 순서 유도 |
    | 서브쿼리 병목 | 반복 수행 구조 | `EXISTS`/`JOIN` 전환, 인라인 뷰 병합 |
    | 바인드 변수 없음 | 공유 실패 | 바인드 변수 사용, `CURSOR_SHARING = FORCE` |
    | 통계정보 부정확 | 실행계획 왜곡 | `DBMS_STATS` 수집, 히스토그램 활용 |
    | INDEX SKIP SCAN 발생 | 선행컬럼 조건 없음 | 인덱스 컬럼 순서 조정 또는 힌트 |
    
    ➤ 이 표를 실무에서 **초기 진단 기준표**로 활용
    

---

1. **[조건절 최적화 전략 정리 – 10분]**
    - WHERE 절 튜닝 핵심:
        - 컬럼 = 상수 구조 유지
        - 함수 제거 (`TO_CHAR`, `UPPER` 등)
        - `BETWEEN`, `IN`, `EXISTS` 활용
        - `NOT IN`, `!=` 회피
    - 복합 조건: **가장 필터링이 강한 조건을 앞에 위치**
    - 날짜 조건은 항상 `TO_DATE`로 범위 명시
    
    🔍 실무 팁:
    
    ```sql
    -- 비효율
    WHERE TO_CHAR(HIREDATE, 'YYYY') = '2022'
    
    -- 효율
    WHERE HIREDATE BETWEEN TO_DATE('2022-01-01','YYYY-MM-DD')
                       AND TO_DATE('2022-12-31','YYYY-MM-DD')
    ```
    

---

1. **[힌트 사용 전략 총정리 – 10분]**
    
    ▶ 자주 사용하는 힌트 정리:
    
    | 힌트 | 설명 |
    | --- | --- |
    | `FULL(TABLE)` | Full Scan 유도 |
    | `INDEX(TABLE INDEX_NAME)` | 특정 인덱스 강제 사용 |
    | `USE_NL(TABLE)` | Nested Loop Join 유도 |
    | `USE_HASH(TABLE)` | Hash Join 유도 |
    | `LEADING(TABLE1 TABLE2)` | 조인 순서 고정 |
    | `MERGE(VIEW_ALIAS)` | 인라인 뷰 병합 허용 |
    | `NO_MERGE(VIEW_ALIAS)` | 인라인 뷰 머티리얼라이즈 유지 |
    | `PARALLEL(TABLE n)` | 병렬 실행 유도 |
    
    🧠 힌트는 항상 “옵티마이저가 잘못 판단했을 때 예외적으로” 사용 → 남용 금지
    

---

1. **[인덱스 설계 전략 회고 – 5분]**
    - 단일 컬럼보다는 **다중 컬럼 인덱스**
    - 선행 컬럼이 항상 조건절에 포함되도록 설계
    - 자주 쓰는 함수는 **함수 기반 인덱스** 고려
    - 너무 많은 인덱스는 DML 성능 저하 유발 → **업데이트 빈도 고려**

---

1. **[정리 및 마무리 – 5분]**
    - 지금까지 배운 전략은 **하나하나 따로 쓰는 기술이 아니라, 문제 유형에 맞게 조합하는 전략 집합**
    - 튜닝은 시행착오를 줄이는 대신, **패턴 인식으로 빨리 정답을 가는 기술**
    - “SQL 튜닝은 케이스 바이 케이스지만,
        
        해결은 결국 **패턴 + 전략 매핑**에서 시작된다.”
