# 4교시 -  I/O 최적화를 위한 인덱스 전략과 Table Access 방식 분석

### 🎯 학습 목표

- Table Access 방식의 차이(Index Scan vs Full Scan)를 이해한다.
- 인덱스 사용 여부에 따른 실행계획 차이를 분석하고 성능을 비교할 수 있다.
- 인덱스 범위 조건, 컬럼 분포도, 힌트 등을 이용한 SQL 튜닝 전략을 학습한다.
- I/O 비용과 실행 계획 상의 Cost가 어떻게 연계되는지 확인한다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “앞 시간에는 Oracle이 데이터를 어떻게 저장하고 접근하는지, 그리고 블록 기반 I/O의 개념을 배웠습니다.
    
    이제는 SQL 실행 시 Oracle이 테이블에 접근하는 방식—즉, 인덱스를 사용할 것인지, 아니면 Full Scan을 수행할 것인지—그 판단 기준과 성능 차이를 본격적으로 분석해 보겠습니다.”
    

---

1. **[Table Access 유형 설명 – 10분]**
    - **Table Full Scan**
        - 테이블의 모든 블록을 순차적으로 읽음 (Sequential I/O)
        - 조건절이 부정확하거나, 반환 행 수가 많을 때 선택
        - 옵티마이저는 ‘통계 정보’와 ‘비용(Cost)’을 기반으로 판단
    - **Index Access (Index Scan)**
        - 인덱스를 통해 RowID를 찾고, 해당 RowID를 사용해 테이블 블록을 Random I/O로 접근
        - Index Unique Scan / Range Scan / Full Scan 등 다양한 방식 존재
        - 소량의 결과가 필요할 경우 효율적
    
    🔎 옵티마이저는 항상 “가장 적은 COST의 접근 방식”을 선택
    
    → 하지만 통계 정보가 부정확하면 비효율적인 실행계획이 나올 수 있음
    

---

1. **[I/O 유형별 Wait Event 비교 – 5분]**
    
    
    | 유형 | Wait Event | 발생 상황 |
    | --- | --- | --- |
    | Random I/O | `db file sequential read` | 인덱스 경유한 RowID 방식 |
    | Sequential I/O | `db file scattered read` | Full Table Scan 발생 시 |
    
    ➤ AWR에서 이 Wait Event가 많이 보일수록 해당 I/O 방식이 SQL 수행 시간의 병목이 될 수 있음
    

---

1. **[실습 – 실행계획 비교 및 효율 분석 – 20분]**
    
    🔸 실습 테이블 및 인덱스 생성
    
    ```sql
    
    CREATE TABLE SALES (
      SALE_ID NUMBER,
      CUST_ID NUMBER,
      AMOUNT NUMBER,
      SALE_DATE DATE
    );
    
    BEGIN
      FOR i IN 1..100000 LOOP
        INSERT INTO SALES VALUES (
          i, MOD(i, 500), ROUND(DBMS_RANDOM.VALUE(1000, 5000)),
          TRUNC(SYSDATE) - MOD(i, 365)
        );
      END LOOP;
    END;
    /
    
    CREATE INDEX IDX_SALES_CUST ON SALES(CUST_ID);
    ANALYZE TABLE SALES COMPUTE STATISTICS;
    
    ```
    
    🔸 실행계획 비교 실습
    
    ```sql
    
    EXPLAIN PLAN FOR SELECT * FROM SALES WHERE CUST_ID = 123;
    SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
    
    EXPLAIN PLAN FOR SELECT * FROM SALES WHERE AMOUNT > 2000;
    SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
    
    ```
    
    🔸 힌트 적용 실습
    
    ```sql
    
    SELECT /*+ FULL(SALES) */ * FROM SALES WHERE CUST_ID = 123;
    SELECT /*+ INDEX(SALES IDX_SALES_CUST) */ * FROM SALES WHERE CUST_ID = 123;
    ```
    
    ➤ **학습 포인트:**
    
    - 옵티마이저가 자동 선택한 방식 vs 힌트로 유도한 방식의 COST 비교
    - Index Scan → Random I/O 발생량 확인
    - Full Scan → 대량 검색 시 오히려 효율적일 수 있음

---

1. **[이론적 정리 – 5분]**
    - 인덱스는 무조건 성능을 향상시키지 않음
    - 인덱스 사용 조건이 안 맞으면 오히려 Random I/O만 증가
    - 옵티마이저는 통계 정보를 기반으로 COST를 산정하므로, **정확한 통계 갱신과 인덱스 구조 설계**가 중요
    - 힌트는 강제 유도 도구이며 남용 금지 (실제 환경에서는 통계 기반 튜닝이 우선)

---

1. **[마무리 및 개념 요약 – 5분]**
    - “Table에 어떻게 접근하는지가 튜닝의 핵심이다.”
    - 실행계획을 보면 Oracle이 무엇을 어떻게 Access하는지 보인다.
    - **실행계획은 SQL의 행동지침서이며, COST는 그 행동의 비용을 수치로 보여주는 설계도다.**

2. 힌트를 제거하고 옵티마이저가 어떤 방식을 선택하는지 확인
