# Day05 - 부록 총정리

### 1. 실행계획 확인(EXPLAIN PLAN) ― 왜 필요한가?

- **비용(Cost)** 값이 가장 낮은 경로를 선택하도록 옵티마이저가 작동합니다.
- 우리는 실행계획을 통해 “인덱스를 사용했는가?”, “풀스캔인가?”를 미리 확인하여 병목을 예방합니다.
- *SQL이 테이블을 어떤 순서·방법으로 읽을지* 미리 보여 주는 지도입니다. 실행 전에 병목을 예측할 수 있습니다.

> 친구 집 가는 길을 네이버지도에서 미리 보고 가장 빠른 버스를 고르는 것과 같습니다.
> 

**핵심 명령**

- `EXPLAIN PLAN FOR …`
- `SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);`

**예제 스크립트**

```sql
-- (1) 실습용 테이블·인덱스 생성
CREATE TABLE emp_temp AS SELECT * FROM scott.emp;
CREATE INDEX idx_emp_dept ON emp_temp(deptno);

-- (2) 계획 생성 & 확인
EXPLAIN PLAN FOR
SELECT * FROM emp_temp WHERE deptno = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### 2. AUTOTRACE ― 30 초 만에 건강검진

1. `SET AUTOTRACE ON STATISTICS` : 실행 후 **I/O·CPU 사용량** 표시
2. `SET AUTOTRACE ON EXPLAIN` : **실행계획**만 표시
3. 끝나면 `SET AUTOTRACE OFF` 로 원상복구
4. 실행계획과 **실제 통계치(논리·물리 I/O, CPU 시간)** 를 한 번에 보여 주는 빠른 점검 도구입니다.

> 팁 : 23c Free에서는 SQL Developer → F10 (Explain) 버튼을 눌러도 동일한 실행계획을 볼 수 있습니다.
> 

**핵심 명령**

- `SET AUTOTRACE ON EXPLAIN` (계획만)
- `SET AUTOTRACE ON STATISTICS` (계획 + 통계)
- `SET AUTOTRACE OFF`

**예제 스크립트**

```sql
-- SQL*Plus 또는 SQLcl 환경
SET AUTOTRACE ON STATISTICS
SELECT * FROM emp_temp WHERE deptno = 20;
SET AUTOTRACE OFF
```

### 3. SQL Trace + TKPROF ― 현미경 분석

1. 세션 단계에서 Trace를 켭니다.
2. 쿼리를 실행하면 `.trc` 파일이 `$ORACLE_BASE/diag/.../trace` 폴더에 생성됩니다.
3. `TKPROF` 툴로 사람이 읽기 쉬운 보고서로 변환합니다.
    - 주요 열 : `count`, `elapsed`, `disk`, `query`, `current`, `rows`
4. 보고서에서 **대기 이벤트(Events)** 와 **실제 I/O 블록 수**를 확인합니다.
5. 세션의 모든 호출을 **.trc 파일**로 남긴 뒤 TKPROF로 사람이 읽기 쉬운 보고서로 바꿉니다.
대기 이벤트·호출 횟수 등을 현미경처럼 확인합니다.

**핵심 명령**

- `ALTER SESSION SET SQL_TRACE = TRUE;`
- `ALTER SESSION SET SQL_TRACE = FALSE;`
- OS 프롬프트: `tkprof <trace_file>.trc out.txt EXPLAIN=scott/tiger`

**예제 스크립트**

```sql
-- (1) 추적 시작
ALTER SESSION SET SQL_TRACE = TRUE;

-- (2) 대상 SQL
SELECT /*+ FULL(emp_temp) */ * FROM emp_temp WHERE sal > 2000;

-- (3) 추적 종료
ALTER SESSION SET SQL_TRACE = FALSE;

-- (4) OS에서 실행
--    tkprof ORCL_12345.trc report_emp.txt EXPLAIN=scott/tiger
```

### 4. DBMS_XPLAN 패키지 ― 실행 후 실제 통계

- `ALLSTATS LAST` 옵션을 주면 **실제 실행 시** 읽은 블록 수, 걸린 시간, CPU 사용량이 행 단위로 나타납니다.
- 커서가 사라지기 전이라면 **실행 직후** 곧바로 분석 가능.
- 실행 후 실제 통계를 포함해 **다양한 형식**으로 계획을 출력합니다. 예상과 실제 차이를 바로 확인할 수 있습니다.

**핵심 명령**

- `DBMS_XPLAN.DISPLAY_CURSOR`
- `DBMS_XPLAN.DISPLAY_AWR` 등

**예제 스크립트**

```sql
-- (1) 쿼리 실행
SELECT * FROM emp_temp WHERE job = 'CLERK';

-- (2) 실행 직후 실제 통계 확인
SELECT *
FROM   TABLE(
         DBMS_XPLAN.DISPLAY_CURSOR(NULL, NULL, 'ALLSTATS LAST')
       );
```

### 5. 실시간 SQL 모니터링 ― 지금 어디까지 왔지?

- 11gR2 Enterprise부터 제공되는 기능(23c Free도 지원).
- `MONITOR` 힌트를 주거나 `elapsed_time`이 일정 시간(5초) 이상이면 자동으로 모니터링 대상이 됩니다.
- HTML 보고서를 열어 보면 **단계별 진행률(%)**, **CPU vs I/O 막대그래프**, **실제 Row 수 vs 예측 Row 수**가 시각화됩니다.
- 오래 실행되는 쿼리의 **진행률·CPU·I/O 소비**를 실시간으로 HTML 보고서로 시각화합니다. (11g Enterprise 이상 / 23c Free 지원)

**핵심 명령**

- `SELECT DBMS_SQLTUNE.REPORT_SQL_MONITOR(...) FROM DUAL;`

**예제 스크립트**

```sql
-- (1) 무거운 쿼리 실행
SELECT /*+ MONITOR */ SUM(sal)
FROM   emp_temp
CONNECT BY LEVEL <= 10000;

-- (2) 최근 SQL_ID 확보
COLUMN sql_id NEW_VALUE v_sql
SELECT sql_id
FROM   v$sql
ORDER  BY last_active_time DESC
FETCH  FIRST 1 ROWS ONLY;

-- (3) HTML 보고서 생성
SET LONG 1000000
SELECT DBMS_SQLTUNE.REPORT_SQL_MONITOR(
         sql_id => '&v_sql',
         type   => 'ACTIVE') AS report
FROM   dual;
```

### 6. V$SQL ― 인기 검색어 순위 보듯이

- **가장 많이 호출된 SQL**, **가장 오래 걸린 SQL** 등을 실시간으로 파악.
- 공유 풀에 남아 있는 **모든 SQL의 누적 통계**(호출 수, 총 소요 시간, 버퍼 읽기 수 등)를 조회해 문제 SQL을 빠르게 찾습니다.
- 열 설명
    - `EXECUTIONS` : 호출 횟수
    - `ELAPSED_TIME` : 누적 걸린 시간(마이크로초)
    - `PARSE_CALLS` : 파싱 횟수(높으면 공유 SQL 재사용 실패)
    - `BUFFER_GETS` : 읽은 버퍼 블록 수

**핵심 명령**

- `SELECT … FROM V$SQL …`

**예제 스크립트**

```sql
-- CPU·Elapsed Time 기준 상위 5건 조회
SELECT sql_id,
       executions,
       elapsed_time / 1e6 AS elapsed_sec,
       cpu_time     / 1e6 AS cpu_sec
FROM   v$sql
WHERE  executions > 0
ORDER  BY elapsed_time DESC
FETCH  FIRST 5 ROWS ONLY;
```
