# **AWR Report - 한 방에 끝내는 실습 가이드 :** 테이블 ORDER_ITEMS의 DDL 관련 답변

🖍 작성자 : 이애본 

### ✅ 예상 ORDER_ITEMS 테이블 DDL

```sql
CREATE TABLE ORDER_ITEMS (
    ORDER_ID     NUMBER NOT NULL,
    LINE_ITEM_ID NUMBER NOT NULL,
    PRODUCT_ID   NUMBER NOT NULL,
    UNIT_PRICE   NUMBER(8,2),
    QUANTITY     NUMBER,
    CONSTRAINT PK_ORDER_ITEMS PRIMARY KEY (ORDER_ID, LINE_ITEM_ID),
    CONSTRAINT FK_ORDER_ITEMS_ORDER FOREIGN KEY (ORDER_ID)
        REFERENCES ORDERS (ORDER_ID)
);
```

---

### ✅ 쿼리 해설

```sql
SELECT /*+ USE_NL(A B) */
       A.SALES_REP_ID,
       COUNT(A.ORDER_TOTAL),
       SUM(B.QUANTITY)
FROM   ORDERS A, ORDER_ITEMS B
WHERE  A.ORDER_ID = B.ORDER_ID
AND    A.SALES_REP_ID = 400
GROUP BY A.SALES_REP_ID;
```

- `USE_NL(A B)` 힌트: **Nested Loop 조인** 유도
- `A.ORDER_ID = B.ORDER_ID`: 조인 조건 (ORDERS와 ORDER_ITEMS의 기본 키-외래 키 관계)
- `A.SALES_REP_ID = 400`: 특정 영업사원 조건
- `GROUP BY A.SALES_REP_ID`: 조건 대상자 한 명이므로 사실상 그룹핑 효과 없음
- `COUNT(A.ORDER_TOTAL)`: 주문 건수
- `SUM(B.QUANTITY)`: 총 수량

---

### 🛠️ 성능 분석 및 해결방안

| 항목 | 설명 |
| --- | --- |
| 🔎 문제점 | **ORDER_TOTAL의 COUNT가 의미 없음** → NULL 포함 여부 고려 필요 |
| 📊 성능 고려 | `Nested Loop` 사용 시 **ORDER_ITEMS에 인덱스**가 없으면 성능 저하 가능 |
| ✅ 해결 방안 1 | `ORDER_ITEMS.ORDER_ID`에 인덱스 생성 → `INDEX RANGE SCAN` 유도 |
| ✅ 해결 방안 2 | 필요시 `ORDER_TOTAL` 대신 `COUNT(*)` 또는 `COUNT(DISTINCT A.ORDER_ID)`로 변경 고려 |
| ✅ 해결 방안 3 | 옵티마이저가 인식할 수 있도록 통계정보 수집 `EXEC DBMS_STATS.GATHER_TABLE_STATS` |
| ✅ 해결 방안 4 | 11g 이하에서는 `USE_NL`이 실제 실행계획에서 무시될 수 있으므로 `LEADING`, `INDEX` 힌트 병행 |

---

### ✨ 최적화된 대체 쿼리 예시

```sql
SELECT /*+ LEADING(A B) USE_NL(B) INDEX(B ORDER_ITEMS_PK) */
       A.SALES_REP_ID,
       COUNT(*) AS ORDER_CNT,
       SUM(B.QUANTITY) AS TOTAL_QTY
FROM   ORDERS A
JOIN   ORDER_ITEMS B
  ON   A.ORDER_ID = B.ORDER_ID
WHERE  A.SALES_REP_ID = 400
GROUP BY A.SALES_REP_ID;
```

이처럼 튜닝 시 **인덱스 힌트 + 드라이빙 테이블 명시 + 불필요한 COUNT 대상 제거**가 병행되면 옵티마이저가 효율적인 계획을 수립하게 됩니다. 

필요 시 `DBMS_XPLAN.DISPLAY_CURSOR`로 실행계획을 확인하세요.
