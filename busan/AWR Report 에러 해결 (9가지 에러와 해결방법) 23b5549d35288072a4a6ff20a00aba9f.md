### ORA-01031(권한이 불충분합니다) 해결 절차

*(Oracle 11g · 23c Free 공통 - 격식체)*

| 단계 | 명령 예시 | 설명 |
| --- | --- | --- |
| 1. **SYSDBA로 접속** | `sqlplus / as sysdba` | `ALTER DATABASE`-급 명령은 **데이터베이스 전체 권한**이 필요합니다. 일반 사용자·SYSTEM 계정으로는 실행할 수 없습니다. |
| 2. **컨테이너 확인(CDB 환경만)** | `SHOW CON_NAME;  
-- 결과가 PDB라면

ALTER SESSION SET CONTAINER = CDB$ROOT;` | 23c Free · 19c 이상 CDB 구조에서는 **루트 컨테이너(CDB$ROOT)** 에서만 데이터파일 변경이 가능합니다. |
| 3-A. **기존 데이터파일 크기 조정** | `ALTER DATABASE DATAFILE '/u01/app/oradata/ORCL/sysaux01.dbf' RESIZE 1G;` | 경로는 `V$DATAFILE`에서 확인한 |

## **✅ 두**번째 에러 - ORA-01516: 현재 컨테이너에 존재하지 않는 로그 파일, 데이터 파일 또는 임시 파일

SQL> ALTER DATABASE DATAFILE '/u01/app/oradata/ORCL/sysaux01.dbf' RESIZE 1G;
ALTER DATABASE DATAFILE '/u01/app/oradata/ORCL/sysaux01.dbf' RESIZE 1G
*
1행에 오류:
ORA-01516: 현재 컨테이너에 존재하지 않는 로그 파일, 데이터 파일 또는 임시 파일
"/u01/app/oradata/ORCL/sysaux01.dbf"이(가) 있습니다.
도움말: [https://docs.oracle.com/error-help/db/ora-01516/](https://docs.oracle.com/error-help/db/ora-01516/)

### 1. 오류의 의미

- **ORA-01516**
    
    > “현재 컨테이너(CDB$ROOT 또는 PDB)에 등록되어 있지 않은 데이터파일을 수정하려 했다.”
    > 
- 즉, **(1) 잘못된 컨테이너에서 명령을 실행했거나 (2) 파일 경로가 실제와 다르다**는 뜻입니다.

---

### 2. 올바른 컨테이너에서 SYSAUX 파일 경로 찾기

```sql
-- ① SYSDBA 권한으로 접속
SQL> CONNECT / AS SYSDBA

-- ② 현재 컨테이너 확인
SQL> SHOW CON_NAME;
-- 결과가 PDB$SEED·FREEPDB1 등 PDB이면 → ROOT 로 전환
SQL> ALTER SESSION SET CONTAINER = CDB$ROOT;

-- ③ SYSAUX 데이터파일 실제 경로 조회
SQL> SELECT file#, name
     FROM   v$datafile
     WHERE  tablespace_name = 'SYSAUX';
```

*TIP* : 23c Free 기본 설치의 경우 경로가

`/opt/oracle/oradata/FREE/SYSAUX01.dbf` 처럼 **FREE** 폴더에 있습니다.

---

### 3. 파일 크기 조정 또는 새 파일 추가

| 상황 | 명령 예시 | 설명 |
| --- | --- | --- |
| 공간 **확장**(기존 파일 늘리기) | `sql<br>ALTER DATABASE DATAFILE '/opt/oracle/oradata/FREE/SYSAUX01.dbf' RESIZE 2G;` | “더 크게 만든다” → 실패 시 Extent 가 뒤쪽에 있어서 줄여지지 않을 수도 있음 |
| 공간 **자동 확장** 설정 | `sql<br>ALTER DATABASE DATAFILE '/opt/oracle/oradata/FREE/SYSAUX01.dbf'<br>  AUTOEXTEND ON NEXT 100M MAXSIZE 4G;` | 앞으로 찔끔찔끔 늘어나도록 설정 |
| 새 데이터파일 **추가**(가장 안전) | `sql<br>ALTER TABLESPACE SYSAUX<br>  ADD DATAFILE '/opt/oracle/oradata/FREE/SYSAUX02.dbf' SIZE 1G<br>  AUTOEXTEND ON NEXT 100M;` | 기존 파일이 줄어들 수 없을 때는 “증설” 방식이 빠르고 위험이 적음 |

---

### 4. PDB(Pluggable Database) 에서 작업해야 하는 경우

1. **PDB 전환**
    
    ```sql
    ALTER SESSION SET CONTAINER = FREEPDB1;   -- 예시
    ```
    
2. 위 ②~③ 단계를 PDB 안에서 다시 수행 →
    
    PDB 전용 SYSAUX 파일 경로를 찾은 뒤 **ALTER DATABASE DATAFILE…** 또는 **ADD DATAFILE** 실행.
    

> 주의: PDB 안에서는 자신의 SYSAUX 만 수정할 수 있으며,
> 
> 
> ROOT 의 SYSAUX 는 CDB$ROOT 세션에서만 조정 가능합니다.
>

### 5. 자주 만나는 추가 오류 대처

| 오류 | 원인 | 해결 |
| --- | --- | --- |
| ORA-03297 “file contains used data beyond requested RESIZE value” | 파일 뒤쪽에 사용 중 Extent 존재 → 크기 축소 불가 | *RESIZE* 대신 **ADD DATAFILE** 로 공간 확장 |
| ORA-01950 “no privileges on tablespace” | 일반 계정이 SYSTEM·SYSAUX 등에 접근 시도 | SYS 또는 **GRANT UNLIMITED TABLESPACE** 후 재시도 |

2. **SHOW CON_NAME** 로 **CDB$ROOT/PDB** 위치를 확인했는가?

## **✅ 세**번째 에러 - ORA-00904: "TABLESPACE_NAME": 부적합한 식별자

SQL> SELECT file#, name
2       FROM   v$datafile
3       WHERE  tablespace_name = 'SYSAUX';
WHERE  tablespace_name = 'SYSAUX'
*
3행에 오류:
ORA-00904: "TABLESPACE_NAME": 부적합한 식별자
도움말: [https://docs.oracle.com/error-help/db/ora-00904/](https://docs.oracle.com/error-help/db/ora-00904/)

### ORA-00904 오류 원인

`V$DATAFILE` 뷰에는 **TABLESPACE_NAME** 열이 없습니다. 따라서 존재하지 않는 열을 조건절에 사용하면 ORA-00904(“부적합한 식별자”)가 발생합니다.

---

## 1. SYSAUX 데이터파일 경로 확인 – 올바른 방법 두 가지

| 구분 | 쿼리 예시 | 설명 |
| --- | --- | --- |
| **가장 간단** | `SELECT file_name
     FROM dba_data_files
    WHERE tablespace_name = 'SYSAUX';` | `DBA_DATA_FILES` 뷰에는 **TABLESPACE_NAME** 열이 있어서 조건절 사용이 가능함. |
| **V$DATAFILE 이용(조인)** | `SELECT df.file#, df.name
  FROM   v$datafile   df
  JOIN   v$tablespace ts ON df.ts# = ts.ts#
 WHERE  ts.name = 'SYSAUX';` | `V$DATAFILE`(ts#) ↔ `V$TABLESPACE`(ts#) 조인 후 테이블스페이스명으로 필터링. |

> SYSDBA 권한으로 접속한 상태에서 실행하십시오.
> 

---

## 2. 이후 작업 예시

```sql
-- ① 파일 경로 확인 후 크기 2 GB로 확장
ALTER DATABASE DATAFILE
  '/opt/oracle/oradata/FREE/SYSAUX01.dbf' RESIZE 2G;

-- ② 또는 새 파일 추가(권장)
ALTER TABLESPACE SYSAUX
  ADD DATAFILE '/opt/oracle/oradata/FREE/SYSAUX02.dbf'
  SIZE 1G AUTOEXTEND ON NEXT 100M;
```

---

## 3. 요약 체크리스트

1. `V$DATAFILE`에는 **TABLESPACE_NAME 열이 없다** → `DBA_DATA_FILES` 사용이 가장 쉽다.
2. 23c Free / CDB 환경이라면 **CDB$ROOT** 세션에서 조회·작업.
3. 공간 축소(RESIZE) 가 실패하면 **ADD DATAFILE** 방식으로 확장하는 편이 안전하다.

위 쿼리를 사용해 SYSAUX 데이터파일의 실제 경로를 확인한 뒤 다시 공간 조정 명령을 실행하시면 더 이상 ORA-00904 오류가 발생하지 않습니다.

### ORA-32771: “bigfile 테이블스페이스에 파일을 추가할 수 없음”

SYSAUX 는 **Bigfile Tablespace(BFTS)** 로 생성되어 있습니다.

- **Bigfile**: 데이터파일을 **단 하나만** 가질 수 있는 구조이며, 대신 최대 128 TB(블록 32 KB 기준)까지 **크기를 무한히 확장**하도록 설계되었습니다.
- 따라서 **ADD DATAFILE** 명령은 허용되지 않고, **기존 파일을 확장**(RESIZE 또는 AUTOEXTEND)해야 합니다.

---

## 1. 현재 테이블스페이스 유형 확인

```sql
SELECT tablespace_name,
       bigfile,            -- YES = Bigfile, NO = Smallfile
       allocation_type     -- SYSTEM, UNIFORM, etc.
  FROM dba_tablespaces
 WHERE tablespace_name = 'SYSAUX';
```

`BIGFILE = YES` 로 표시되면 Bigfile TS 가 확실합니다.

---

## 2. 공간 확장 방법 (Bigfile 전용)

| 방법 | 명령 예시 | 특징 |
| --- | --- | --- |
| **AUTOEXTEND** (권장) | `ALTER DATABASE DATAFILE
'C:\APP\ASUS\PRODUCT\23AI\ORADATA\FREE\SYSAUX01.DBF'
  AUTOEXTEND ON NEXT 100M MAXSIZE 4G;` | 앞으로 공간이 부족할 때마다 100 MB씩 자동 증가 (최대 4 GB). |
| **즉시 RESIZE** | `ALTER DATABASE DATAFILE
  'C:\APP\ASUS\PRODUCT\23AI\ORADATA\FREE\SYSAUX01.DBF'
  RESIZE 2G;` | 파일을 한 번에 크게 늘림. 이미 AUTOEXTEND 가 켜져 있으면 생략 가능. |

> 축소(RESIZE 작게) 는 사용 중 블록이 뒤쪽에 있으면 ORA-03297 로 실패할 수 있으므로, 늘리기 또는 AUTOEXTEND 만 사용하는 편이 안전합니다.
> 

---

## 3. Bigfile → Smallfile 로 바꾸고 싶은 경우?

- **테이블스페이스 자체를 다시 만드는 방법**밖에 없습니다.
    1. 새 Smallfile 테이블스페이스 생성
    2. SYSAUX 구성요소(DBA_SYSAUX_OCCUPANTS에 나열)를 `ALTER … MOVE` 로 이동
    3. 데이터베이스 재시작 후 이전 SYSAUX 드롭
- 관리가 복잡하고 리스크가 크므로, **권장되지 않습니다.** Bigfile 방식 그대로 두고 파일을 확장·자동확장 설정하는 것이 가장 간단합니다.

### 핵심 체크리스트

1. **Bigfile TS** 는 **데이터파일이 하나**만 존재한다.
2. 공간 부족 시 **AUTOEXTEND ON** 또는 **RESIZE** 로 확장한다.

4. 구조 변경( bigfile → smallfile )은 재구성 작업이 필요하므로 가급적 피한다.

위 절차대로 **AUTOEXTEND** 를 설정하거나 **RESIZE** 로 파일을 키우시면 추가 오류 없이 SYSAUX 공간을 확보하실 수 있습니다. 추가 궁금증이 있으면 언제든 문의해 주십시오.

1. `C:\awr_demo\scripts\awr_compare.py` 파일을 만듭니다.
    
    ```python
    import sys, pandas as pd, re, os, zipfile
    from bs4 import BeautifulSoup
    
    def extract(path):
        with open(path, "r", encoding="utf-8", errors="ignore") as f:
            soup = BeautifulSoup(f, "html.parser")
        t = soup.find("table", summary=re.compile("load profile", re.I))
        if t is None:
            return {}
        col = [c.get_text(strip=True) for c in t.find_all("th")][1:]
        data = {}
        for r in t.find_all("tr"):
            cells = [c.get_text(strip=True) for c in r.find_all("td")]
            if cells and cells[0].startswith(("DB Time", "DB CPU", "Logical read", "Physical read")):
                for i, v in enumerate(cells[1:]):
                    key = f"{cells[0]} {col[i]}"
                    data[key] = float(v.replace(",", "")) if v else None
        return data
    
    before, after = extract(sys.argv[1]), extract(sys.argv[2])
    keys = [k for k in before if 'Per Second' in k]
    rows = []
    for k in keys:
        b, a = before[k], after.get(k)
        rows.append([k.replace(" (blocks)", ""), b, a, a-b, (a-b)/b*100 if b else None])
    df = pd.DataFrame(rows, columns=["지표", "튜닝 전", "튜닝 후", "변화량", "변화율(%)"])
    out_csv = os.path.join("..", "output", "awr_comparison_summary.csv")
    df.to_csv(out_csv, index=False)
    
    # 패키징
    out_zip = os.path.join("..", "output", "awr_demo_package.zip")
    with zipfile.ZipFile(out_zip, "w") as z:
        z.write(sys.argv[1], arcname="awr_before.html")
        z.write(sys.argv[2], arcname="awr_after.html")
        z.write(out_csv, arcname="awr_comparison_summary.csv")
        z.writestr("README.txt", "AWR demo package\n\n- awr_before.html\n- awr_after.html\n- awr_comparison_summary.csv")
    print("완료 →", out_zip)
    ```

### ORA-20200: “Begin Snapshot Id … does not exist” 오류 원인

- **DBMS_WORKLOAD_REPOSITORY**가 기록한 스냅 번호(**SNAP_ID**)가 403, 404가 아닙니다.
- 즉 **스냅을 아직 찍지 않았거나**, 잘못된 인스턴스(DBID·INSTANCE_NUMBER)가 선택되었습니다.

## 1. 현재 데이터베이스에 존재하는 스냅 번호 확인

-- SYSDBA 권한으로 실행

COL begin_time         FORMAT a19
COL end_time           FORMAT a19
COL inst                FORMAT 999

SELECT snap_id,
       TO_CHAR(begin_interval_time,'YYYY-MM-DD HH24:MI') AS begin_time,

### 왜 ORA-06550가 나타났습니까?

`BEGIN … END;` 형식의 PL/SQL 블록을 작성했으면 **맨 마지막 줄에 슬래시(/)** 를 넣어야 SQL*Plus가 “블록 끝”으로 인식하여 실행합니다.

슬래시가 없으면 **소스가 완결되지 않은 채 EOF(end-of-file)에 도달**했다고 판단하여 PLS-00103 오류가 발생합니다.

## 1. 스냅샷 만드는 올바른 두 가지 방법

| 가장 간단 | 완전한 PL/SQL 블록 |
| --- | --- |
| `sql<br>EXEC DBMS_WORKLOAD_REPOSITORY.create_snapshot;<br>` | `sql<br>BEGIN<br>  DBMS_WORKLOAD_REPOSITORY.create_snapshot;<br>END;<br>/  -- ← 마지막 줄의 슬래시 필수<br>` |

※ `EXEC`(또는 `EXECUTE`)는 SQL*Plus 전용 단축키라 블록 종료 기호가 필요 없습니다.

※ `BEGIN … END;` 로 직접 쓰려면 **슬래시**(`/`)가 꼭 필요합니다.

### 핵심 체크리스트

1. **EXEC** 한 줄로 실행하거나, `BEGIN … END;` 뒤에 **슬래시(/)** 를 붙인다.
2. 스냅이 정상 찍히면 `dba_hist_snapshot` 테이블에 **새 SNAP_ID**가 추가된다.
3. `awrrpt.sql` 실행 시 **DBID·Instance Number·Begin/End Snap Id**를 정확히 입력한다.
4. CMD 창에서는 HTML 경로 앞에 `start ""` 를 붙이거나 탐색기에서 열어야 한다.

# 🧭 AWR 리포트 도출을 위한 **전체 체크리스트**

> 아래 항목만 순서대로 입력·설정하시면 어느 환경에서도 동일한 방식으로 AWR 리포트를 얻을 수 있습니다.
> 

---

### 1. 기본 환경·권한

| 구분 | 필수 조건 | 확인 방법 |
| --- | --- | --- |
| 접속 권한 | **SYSDBA** | `sqlplus / as sysdba` 또는 `SELECT username FROM v$session WHERE audsid = USERENV('SESSIONID');` → ‘SYS’ 확인 |
| 컨테이너 위치 | **CDB$ROOT** (또는 리포트를 만들 PDB) | `SHOW CON_NAME;` → 필요 시 `ALTER SESSION SET CONTAINER = CDB$ROOT;` |
| AWR 설정 | 스냅 주기 60 분·보존 8 일이 기본(변경 필요 시) | `EXEC DBMS_WORKLOAD_REPOSITORY.modify_snapshot_settings(interval => 60, retention => 11520);  -- 분 단위` |
| 출력 폴더 | OS 상에 **쓰기 가능**한 경로 | 예) `C:\awr_demo\output\` 폴더 미리 생성 |

---

### 2. 스냅샷 생성 명령

```sql
-- 첫 스냅 (튜닝 전)
EXEC DBMS_WORKLOAD_REPOSITORY.create_snapshot;

-- 5~10 분(또는 원하는 부하 구간) 경과

-- 둘째 스냅 (튜닝 후)
EXEC DBMS_WORKLOAD_REPOSITORY.create_snapshot;
```

- PL/SQL 블록으로 작성할 때는 **슬래시(`/`)** 로 마무리
    
    ```sql
    BEGIN
      DBMS_WORKLOAD_REPOSITORY.create_snapshot;
    END;
    /
    ```
    

---

### 3. 스냅 번호 확인

```sql
SELECT snap_id,
       TO_CHAR(begin_interval_time,'YYYY-MM-DD HH24:MI') AS begin_time,
       TO_CHAR(end_interval_time  ,'YYYY-MM-DD HH24:MI') AS end_time
  FROM dba_hist_snapshot
 ORDER BY snap_id DESC;
```

*가장 최근 두 개*가 바로 방금 찍은 스냅 ID(예: **75, 76**).

### 4. awrrpt.sql 실행 시 입력값

| 프롬프트 질문 | 반드시 입력할 값 | 비고 |
| --- | --- | --- |
| **report_type** | 엔터(=html) 또는 `text` | 보통 `html` 권장 |
| **DB Id / Instance Number** | 기본값 엔터 | RAC 환경이 아니면 대부분 1 |
| **Begin Snap Id** | **첫 스냅 번호** (예 75) | 위 조회 결과 |
| **End Snap Id** | **둘째 스냅 번호** (예 76) | Begin 보다 커야 함 |
| **Report Name** | `C:\awr_demo\output\awr_test.html` | 전체 경로·파일명 지정 |

*완료 메시지* `Report written to …` 출력 확인.
