### 2. [문제 풀이 및 해설 – 40분]

### ✅ 문제 1. 실행계획 해석 (객관식)

**문제:** 다음 SQL의 실행계획에서 Filter Predicate가 발생하는 이유는?

```sql
SELECT * FROM EMP
WHERE TO_CHAR(HIREDATE, 'YYYY') = '2022';
```

**보기:**

A. 인덱스가 존재하지 않기 때문

B. 조건절에 함수가 사용되어 인덱스가 무력화되었기 때문

C. Nested Loop Join을 사용했기 때문

D. 히스토그램이 없기 때문

**정답:** B

**해설:** TO_CHAR 함수가 HIREDATE 컬럼에 적용되어, 인덱스가 있어도 사용되지 않음 → Filter Predicate 발생

---

### ✅ 문제 2. 조건절 튜닝 판단 (서술형)

**문제:** 다음 SQL이 느린 이유는?

```sql
SELECT * FROM ORDERS WHERE ORDER_DATE + 1 = SYSDATE;
```

**설명:**

- 인덱스 무효: ORDER_DATE에 산술 연산이 적용됨 → `ORDER_DATE BETWEEN SYSDATE - 1 AND SYSDATE - 1` 등으로 구조 변경 필요

---

### ✅ 문제 3. 힌트 판단 문제

**문제:** 다음 SQL에서 `/*+ USE_HASH(D) */` 힌트를 사용할 경우 어떤 결과가 예상되는가?

```sql
SELECT E.ENAME, D.DNAME
FROM EMP E JOIN DEPT D ON E.DEPTNO = D.DEPTNO;
```

**정답 설명:**

- EMP가 큰 테이블일 경우, DEPT는 드라이빙 테이블이 아닌 **빌드 테이블**이 되며
    
    → Hash Join이 적용되고 Full Scan 가능성 있음
    
- 옵티마이저는 기본적으로 NLJ를 선호하나, 힌트로 강제 변경됨

---

### ✅ 문제 4. 인덱스 유지 전략

**문제:** 다음 중 인덱스 Rebuild가 필요한 상황은?

A. BLEVEL = 1

B. CLUSTERING FACTOR가 매우 낮음

C. LEAF_BLOCKS 수가 급격히 증가함

D. DISTINCT_KEYS가 매우 작음

**정답:** C

**해설:** LEAF_BLOCKS는 인덱스의 조각화를 반영하며, Rebuild의 가장 일반적인 사유

---

### ✅ 문제 5. 통계 및 히스토그램 판단

**문제:** 다음 중 옵티마이저가 히스토그램 없이 ‘균등 분포’로 잘못 판단할 가능성이 있는 상황은?

A. 컬럼 값이 3개뿐이고, 분포는 균등함

B. NDV가 매우 높음

C. 특정 값이 전체의 90%를 차지함

D. 히스토그램이 수집되어 있음

**정답:** C

**해설:** 옵티마이저는 히스토그램이 없을 경우 항상 균등 분포로 가정함. 실제로 편중이 있는 경우 실행계획 오류 발생 가능

---

### 3. [요약 정리 및 복습 퀴즈 – 5분]

**암기 Tip:**

- 조건절 함수 → 인덱스 무력화
- `FILTER`는 Index 미사용 증거
- BLEVEL ↑, LEAF_BLOCKS ↑ → Rebuild 고려
- `USE_HASH`, `LEADING` 힌트 → 실행계획 강제 변경
- 히스토그램 필요 = 값의 분포 불균형
