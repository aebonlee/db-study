# 8교시 - 옵티마이저 힌트(Hint) 활용 전략과 SQL 튜닝 실습

### 🎯 학습 목표

- 옵티마이저 힌트(Hint)의 개념과 사용 목적을 이해한다.
- SQL 실행계획에 영향을 미치는 주요 힌트 종류를 구분하고 사용할 수 있다.
- 힌트를 통해 옵티마이저의 조인 방식, 인덱스 사용 여부, 병렬처리 등을 제어할 수 있다.
- 힌트가 미치는 성능 영향과 위험성에 대해 분석할 수 있다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “앞 시간까지는 옵티마이저가 SQL을 스스로 판단하여 실행계획을 결정하는 과정을 살펴봤습니다.
    
    그러나, 어떤 경우에는 옵티마이저가 부정확한 판단을 할 수 있고, 이럴 때 우리가 실행계획을 강제해야 할 필요가 있습니다.
    
    이번 시간에는 그런 상황에서 사용하는 ‘힌트(Hint)’에 대해 배우겠습니다. 힌트는 튜닝을 위한 강력한 도구이지만, 남용하면 해가 될 수 있는 양날의 검입니다.”
    

---

1. **[옵티마이저 힌트 개요 및 원리 – 10분]**
    - 힌트란? → 옵티마이저에게 특정 실행계획을 유도하기 위해 SQL 문장 내에 명시하는 지시어
    - 형식:
        
        ```sql
        
        SELECT /*+ 힌트명 */ 컬럼 FROM 테이블 WHERE 조건;
        ```
        
    - 옵티마이저는 힌트를 가능한 한 따르지만, 불합리한 경우 무시할 수도 있음
    - 힌트는 SQL 재작성 없이 실행계획을 바꾸는 **비침습적 튜닝 기법**

---

1. **[주요 힌트 종류 설명 – 10분]**
    
    ### 🔹 접근 방식 제어 (Access Path Hints)
    
    - `FULL(TABLE_ALIAS)`: Full Table Scan 유도
    - `INDEX(TABLE_ALIAS INDEX_NAME)`: 특정 인덱스 강제 사용
    - `NO_INDEX(TABLE_ALIAS)`: 인덱스 사용하지 않도록 유도
    
    | 힌트 종류 | 설명 | 예시 |
    | --- | --- | --- |
    | `FULL(TABLE_ALIAS)` | 테이블에 대해 Full Table Scan을 강제로 수행 | `SELECT /*+ FULL(e) */ * FROM emp e WHERE deptno = 10;` |
    | `INDEX(TABLE_ALIAS INDEX_NAME)` | 특정 인덱스를 강제로 사용 | `SELECT /*+ INDEX(e emp_idx1) */ * FROM emp e WHERE empno = 7788;` |
    | `NO_INDEX(TABLE_ALIAS INDEX_NAME)` | 특정 인덱스를 사용하지 않도록 유도 | `SELECT /*+ NO_INDEX(e emp_idx1) */ * FROM emp e WHERE empno = 7788;` |
    
    **활용 팁**:
    
    - `FULL` 힌트는 소량의 데이터가 전체 테이블에 분산된 경우 사용.
    - `INDEX`는 분포도가 낮고 조건절이 인덱스를 잘 타는 경우 성능 향상.
    - `NO_INDEX`는 인덱스 선택이 잘못된 경우, 옵티마이저 판단을 배제하고 Full Scan 유도.
    
    ### 🔹 조인 방식 제어 (Join Method Hints)
    
    - `USE_NL(TABLE_ALIAS)`: Nested Loop Join 사용
    - `USE_HASH(TABLE_ALIAS)`: Hash Join 사용
    - `USE_MERGE(TABLE_ALIAS)`: Sort Merge Join 사용
    
    | 힌트 종류 | 설명 | 예시 |
    | --- | --- | --- |
    | `USE_NL(TABLE_ALIAS)` | Nested Loop Join 강제 | `SELECT /*+ USE_NL(e d) */ e.ename, d.dname FROM emp e, dept d WHERE e.deptno = d.deptno;` |
    | `USE_HASH(TABLE_ALIAS)` | Hash Join 강제 | `SELECT /*+ USE_HASH(e d) */ e.ename, d.dname FROM emp e, dept d WHERE e.deptno = d.deptno;` |
    | `USE_MERGE(TABLE_ALIAS)` | Sort-Merge Join 강제 | `SELECT /*+ USE_MERGE(e d) */ e.ename, d.dname FROM emp e, dept d WHERE e.deptno = d.deptno;` |
    
    **활용 팁**:
    
    - `USE_NL`: 소규모 반복 조인에 유리 (인덱스가 드라이빙 테이블에 있을 때)
    - `USE_HASH`: 대용량 테이블 간 조인에서 효율적
    - `USE_MERGE`: 정렬된 데이터 혹은 인덱스 기반 테이블 조인 시 효율적
    
    ### 🔹 조인 순서 제어 (Join Order Hints) 조인 순서 제어
    
    - `LEADING(TABLE1 TABLE2 ...)`: 조인 순서 고정
    
    | 힌트 종류 | 설명 | 예시 |
    | --- | --- | --- |
    | `LEADING(TABLE1 TABLE2 ...)` | 조인 순서를 고정하여 옵티마이저의 조인 순서 결정 방지 | `SELECT /*+ LEADING(e d) */ e.ename, d.dname FROM emp e, dept d WHERE e.deptno = d.deptno;` |
    
    **활용 팁**:
    
    - 조인 순서를 튜닝할 때 필수 사용.
    - 드라이빙 테이블의 선택은 결과적으로 전체 실행 계획을 좌우.
    - 옵티마이저가 비효율적인 순서를 선택할 때 강력한 조정 수단.
    
    ### 🔹 병렬 처리 제어 (Parallel Execution Hints)
    
    - `PARALLEL(TABLE_ALIAS, DEGREE)`: 병렬 실행 유도
    - `NOPARALLEL(TABLE_ALIAS)`: 병렬 실행 방지
    
    | 힌트 종류 | 설명 | 예시 |
    | --- | --- | --- |
    | `PARALLEL(TABLE_ALIAS, DEGREE)` | 병렬 쿼리 실행 유도 | `SELECT /*+ PARALLEL(e, 4) */ * FROM emp e;` |
    | `NOPARALLEL(TABLE_ALIAS)` | 병렬 실행 방지 | `SELECT /*+ NOPARALLEL(e) */ * FROM emp e;` |
    
    **활용 팁**:
    
    - 병렬 처리 유도는 대용량 데이터를 대상으로 한 집계/정렬/조인 시 매우 유용.
    - `PARALLEL` 사용 시, 해당 테이블에 병렬도가 설정되어 있어야 하며 서버 환경에서 병렬 처리 지원이 활성화되어야 함.
    - `NOPARALLEL`은 OLTP 환경처럼 병렬 실행으로 인한 오버헤드 방지가 필요할 때 사용.

## ✅ 추가 실무 팁

- **힌트는 힌트일 뿐, 옵티마이저가 완전히 무조건 따르는 것은 아닙니다**.

       예: 힌트와 통계정보가 너무 맞지 않으면 무시될 수 있습니다.

- 힌트를 쓸 때는 항상 **실행계획을 분석(Explain Plan)**해서 성능 향상을 확인해야 합니다.
- 복수의 힌트를 조합하여 사용하는 것이 일반적입니다:
    
    ```sql
    SELECT /*+ LEADING(e d) USE_NL(d) INDEX(e emp_idx1) */ ...
    ```

1. **[실습 – 힌트 적용에 따른 실행계획 변화 분석 – 20분]**
    
    🔸 실습 SQL 1 – Full Scan 유도
    
    ```sql
    SELECT /*+ FULL(SALES) */ * FROM SALES WHERE CUST_ID = 100;
    SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY)
    ```
    
    🔸 실습 SQL 2 – Index 사용 강제
    
    ```sql
    SELECT /*+ INDEX(SALES IDX_SALES_CUST) */ * FROM SALES WHERE CUST_ID = 100;
    ```
    
    🔸 실습 SQL 3 – 조인 방식 변경
    
    ```sql
    SELECT /*+ USE_NL(A B) */ A.ENAME, B.DNAME
    FROM EMP A, DEPT B
    WHERE A.DEPTNO = B.DEPTNO;
    
    SELECT /*+ USE_HASH(A B) */ A.ENAME, B.DNAME
    FROM EMP A, DEPT B
    WHERE A.DEPTNO = B.DEPTNO;
    ```
    
    ➤ 실행계획 해석:
    
    - Access Path 및 Join Method가 어떻게 바뀌는지 확인
    - COST 값 변화 및 예상 Row 수 변화 체크

1. **[주의사항 및 실무 적용 전략 – 5분]**
    - 힌트는 옵티마이저의 판단이 잘못됐을 때 예외적으로 사용해야 한다.
    - 통계 정보가 정확하지 않거나, 일시적인 특수 상황에 맞춰 성능을 유도할 때만 사용
    - 코드에 고정된 힌트는 향후 데이터 분포가 달라질 경우 오히려 성능 저하 유발 가능
    - ➤ 실무에서는 힌트를 사용한 SQL은 반드시 테스트 및 모니터링을 병행해야 함

---

1. **[정리 및 마무리 – 5분]**
    - 힌트는 Oracle의 실행계획을 우리가 직접 설계할 수 있게 해주는 유일한 수단
    - 그러나 힌트는 ‘설계자’가 아닌 ‘임시 구조물’로 사용해야 한다
    - “힌트는 실행계획을 바꾸는 명령이 아니라, 옵티마이저와 협상하는 제안이다”
