# 7교시 - 인덱스 설계 전략과 클러스터링 팩터 기반 SQL 튜닝 실습

### 🎯 학습 목표

- B-Tree 인덱스의 내부 구조를 이해하고 생성 방식에 따라 접근 방식이 어떻게 달라지는지 설명할 수 있다.
- 클러스터링 팩터(Clustering Factor)의 개념과 SQL 실행 시 발생하는 I/O와의 연관성을 분석할 수 있다.
- 인덱스가 존재하더라도 효율적이지 않을 수 있는 사례를 이해하고 실행계획으로 판단할 수 있다.
- 실제 인덱스를 활용한 튜닝 실습을 통해 Random I/O 감소 전략을 적용할 수 있다.

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “지난 시간에 실행계획을 통해 SQL이 테이블과 인덱스에 어떻게 접근하는지를 배웠습니다.
    
    이번에는 ‘인덱스가 있어도 성능이 떨어질 수 있는 이유’를 구체적으로 실습과 함께 확인해 보겠습니다.
    
    오늘 수업의 핵심은 바로 클러스터링 팩터입니다. 단지 인덱스를 생성했다고 해서 좋은 성능이 보장되는 것이 아니라는 점을 체감하실 겁니다.”
    

---

1. **[B-Tree 인덱스 구조 설명 – 10분]**
    - 인덱스는 Root Block → Branch Block → Leaf Block 구조의 트리 형태로 구성
    - Leaf Block에는 Key 값과 해당 Key에 대한 RowID가 저장됨
    - 인덱스를 통해 테이블 RowID 접근 시 Random I/O가 발생
        - 즉, 인덱스 효율은 **테이블 내 데이터 배치 순서와 밀접한 관계**
    - 동일한 SELECT 문을 실행하더라도, 테이블이 어떤 순서로 정렬돼 있느냐에 따라 성능 차이 발생

---

1. **[클러스터링 팩터 개념 설명 – 10분]**
    - 클러스터링 팩터란?
        
        → 인덱스에 정렬된 순서와 테이블의 물리적 행 저장 순서가 얼마나 유사한지를 나타내는 값
        
        - 수치가 테이블 블록 수에 가까울수록 우수 (정렬 유사),
            
            행 수에 가까울수록 나쁨 (랜덤 배치)
            
    - 사용 예시:
        
        ```sql
        SELECT INDEX_NAME, CLUSTERING_FACTOR, NUM_ROWS, LEAF_BLOCKS
        FROM USER_INDEXES
        WHERE TABLE_NAME = 'SALES';
        ```
        
    - ➤ 인덱스 효율을 단순히 "있는가/없는가"가 아니라 **어떻게 정렬되어 있는가**로 판단해야 함

---

1. **[실습 – 인덱스 생성, 실행계획 분석, 클러스터링 팩터 확인 – 20분]**
    
    🔸 인덱스 생성
    
    ```sql
    CREATE INDEX IDX_SALES_DATE ON SALES(SALE_DATE);
    ```
    
    🔸 통계 갱신
    
    ```sql
    BEGIN
      DBMS_STATS.GATHER_TABLE_STATS(OWNNAME => USER, TABNAME => 'SALES');
    END;
    /
    ```
    
    🔸 실행계획 비교
    
    ```sql
    EXPLAIN PLAN FOR SELECT * FROM SALES WHERE SALE_DATE = TO_DATE('2023-01-01','YYYY-MM-DD');
    SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
    ```
    
    🔸 클러스터링 팩터 확인
    
    ```sql
    SELECT INDEX_NAME, CLUSTERING_FACTOR, NUM_ROWS
    FROM USER_INDEXES
    WHERE TABLE_NAME = 'SALES';
    ```
    
    ➤ 날짜 기준으로 입력된 데이터라면 SALE_DATE 인덱스는 클러스터링 팩터가 낮아 효율적
    
    ➤ 무작위 데이터였다면 Random I/O 증가 → 실행계획에서 Full Scan 유도될 수 있음
    

---

1. **[인덱스 재정렬 전략 및 고려사항 – 5분]**
    - **CTAS(Create Table As Select + 정렬)** 방식으로 재정렬 가능:
        
        ```sql
        CREATE TABLE SALES_SORTED AS
        SELECT * FROM SALES ORDER BY SALE_DATE;
        ```
        
    - 재정렬 후 인덱스 재생성하면 클러스터링 팩터 대폭 개선 가능
    - 클러스터링 팩터는 **조인 전략, 인덱스 사용 여부 결정의 핵심 변수**로 활용됨

---

1. **[마무리 정리 – 5분]**
    - 인덱스는 무조건 성능을 높이는 도구가 아니다
    - 잘못된 정렬 구조에서는 오히려 인덱스가 Random I/O를 유발
    - 클러스터링 팩터는 **인덱스 효율을 수치로 측정할 수 있는 핵심 지표**
    - “인덱스는 눈에 보이지 않는 조력자이자, 성능을 갉아먹는 배신자일 수도 있다.”

### 📝 과제

1. SALES 테이블을 CTAS 방식으로 정렬 재구성한 후, 클러스터링 팩터 차이를 비교해보기
2. 클러스터링 팩터가 높은 인덱스를 찾고, 해당 인덱스를 활용한 SQL 실행계획 확인
3. 클러스터링 팩터와 실행계획 COST 값 간의 연관성을 정리한 리포트 제출
