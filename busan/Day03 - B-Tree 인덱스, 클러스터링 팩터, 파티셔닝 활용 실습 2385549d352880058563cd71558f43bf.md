### 1. 테스트 테이블 생성

```sql
CREATE TABLE customer (
    cust_id   NUMBER,
    name      VARCHAR2(100),
    gender    CHAR(1),
    city      VARCHAR2(50),
    created_at DATE
);

BEGIN
  FOR i IN 1..100000 LOOP
    INSERT INTO customer
    VALUES (
      i,
      '고객_' || i,
      CASE MOD(i, 2) WHEN 0 THEN 'M' ELSE 'F' END,
      CASE MOD(i, 5)
          WHEN 0 THEN '부산'
          WHEN 1 THEN '서울'
          WHEN 2 THEN '대구'
          WHEN 3 THEN '인천'
          ELSE '광주'
      END,
      SYSDATE - MOD(i, 365)
    );
  END LOOP;
  COMMIT;
END;
```

### 2. 인덱스 실습: 단일 인덱스 vs Full Scan

```sql
-- 인덱스 없는 상태에서

-- 인덱스 생성 후
CREATE INDEX idx_customer_name ON customer(name);

-- 다시 동일 쿼리 실행 후 실행계획 비교

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### 5. 클러스터링 팩터 실습

클러스터링 팩터 차이를 보기 위해 두 테이블을 만들고,

- 하나는 `ORDER BY cust_id`로 데이터를 정렬 후 삽입
- 다른 하나는 무작위 삽입

```sql
-- 순차 입력
CREATE TABLE customer_seq AS SELECT * FROM customer ORDER BY cust_id;

-- 무작위 입력
CREATE TABLE customer_rand AS SELECT * FROM customer ORDER BY DBMS_RANDOM.VALUE;

-- 인덱스 생성
CREATE INDEX idx_seq ON customer_seq(cust_id);
CREATE INDEX idx_rand ON customer_rand(cust_id);

-- 클러스터링 팩터 비교
SELECT TABLE_NAME, INDEX_NAME, CLUSTERING_FACTOR
FROM USER_INDEXES
WHERE TABLE_NAME IN ('CUSTOMER_SEQ', 'CUSTOMER_RAND');
```

### 💡 실습 목표

- 클러스터링 팩터가 좋을 때와 나쁠 때 성능 차이 체험
- 테이블 정렬 삽입 전략의 효과 확인

```sql
-- 실행계획 확인
EXPLAIN PLAN FOR SELECT * FROM customer_seq WHERE cust_id BETWEEN 1000 AND 2000;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

EXPLAIN PLAN FOR SELECT * FROM customer_rand WHERE cust_id BETWEEN 1000 AND 2000;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 인덱스 통계 확인
SELECT TABLE_NAME, INDEX_NAME, CLUSTERING_FACTOR, NUM_ROWS
FROM USER_INDEXES 
WHERE TABLE_NAME IN ('CUSTOMER_SEQ', 'CUSTOMER_RAND');
```

📌 **설명 포인트**: 같은 인덱스를 써도 `customer_seq`는 Random I/O가 적고 `customer_rand`는 많다 → 성능 차이 비교

### 6. 실행계획 해석 및 튜닝

### 💡 주요 개념

- TABLE ACCESS FULL vs TABLE ACCESS BY INDEX ROWID
- COST, CARDINALITY, BYTES 등의 의미
- `FILTER`, `ACCESS`, `INDEX RANGE SCAN` 등 PLAN STEP 이해

```sql
sql
복사편집
EXPLAIN PLAN FOR SELECT * FROM customer WHERE city = '부산';

-- 통계 정보 수집
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(USER, 'CUSTOMER');
END;

```

### 실습 포인트

- `SELECT *`, 범위 조건, LIKE, IN, IS NULL, IS NOT NULL 조건의 실행계획 비교
- 힌트 없이 실행 → 힌트 적용 → 실행계획 변화 분석

## ✅ 실습 주제: B-Tree 인덱스, 클러스터링 팩터, 파티셔닝을 이용한 쿼리 성능 실험

### 📁 1. 실습용 테이블 생성

```sql
-- 실습 전 정리
DROP TABLE customer PURGE;
DROP TABLE customer_hash PURGE;
DROP TABLE customer_part PURGE;

-- 기본 테이블 생성
CREATE TABLE customer (
  cust_id     NUMBER,
  region_cd   VARCHAR2(10),
  birth_year  NUMBER,
  name        VARCHAR2(100),
  gender      CHAR(1)
);
```

### 🔍 해설:

- 고객 정보를 저장하는 테이블로, 지역(region_cd), 출생년도(birth_year), 이름, 성별을 포함합니다.
- 실습 목적: 지역 코드(region_cd)로 인덱스를 만들고, 이후 다양한 성능 테스트에 활용합니다.

```sql
-- 대용량 데이터 삽입 (예: 100만 건)
BEGIN
  FOR i IN 1..1000000 LOOP
    INSERT INTO customer
    VALUES (
      i,
      CASE MOD(i, 5)
        WHEN 0 THEN '서울' WHEN 1 THEN '부산'
        WHEN 2 THEN '대구' WHEN 3 THEN '광주'
        ELSE '제주' END,
      1980 + MOD(i, 40),
      '고객_' || i,
      CASE MOD(i, 2) WHEN 0 THEN 'M' ELSE 'F' END
    );
  END LOOP;
  COMMIT;
END;
```

### 🔍 해설:

- 100만 건의 데이터를 생성함으로써 Full Table Scan과 Index Access 성능 차이를 명확히 보기 위해 설계됨.
- 지역코드, 출생년도, 성별 등 다양한 조회 조건을 실험할 수 있도록 데이터가 균등하게 분포되어 있음.

### 📌 2. B-Tree 인덱스 실습

```sql
-- 인덱스 생성
CREATE INDEX idx_customer_region ON customer(region_cd);
```

### 🔍 해설:

- `region_cd`에 대해 B-Tree 인덱스를 생성합니다.
- B-Tree는 범위 조건, 등호(=), LIKE 등의 질의에 효율적이며 가장 일반적인 인덱스 구조입니다.

```sql
-- 실행계획 비교
EXPLAIN PLAN FOR
SELECT * FROM customer WHERE region_cd = '서울';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### 🔍 해설:

- 위 쿼리는 **'서울' 지역에 속한 고객을 조회**합니다.
- `EXPLAIN PLAN`을 통해 옵티마이저가 인덱스를 사용하는지, Full Scan하는지를 분석합니다.

### 📌 3. 클러스터링 팩터 실습

```sql
-- 테이블 복사 및 정렬된 상태로 저장
CREATE TABLE customer_sorted
AS SELECT * FROM customer ORDER BY region_cd;

-- 인덱스 생성
CREATE INDEX idx_customer_region_sorted ON customer_sorted(region_cd);

-- 인덱스 통계 갱신
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(USER, 'CUSTOMER_SORTED', cascade => TRUE);
END;
```

### 🔍 해설:

- 같은 데이터라도 테이블의 **물리적 정렬 순서(region_cd 기준)**에 따라 인덱스 효율이 달라집니다.
- 이 정렬이 인덱스 key 순서와 유사할수록 **클러스터링 팩터(Clustering Factor)**가 낮아지고, Random I/O가 줄어 성능이 향상됩니다.

```sql
-- 클러스터링 팩터 확인
SELECT
  i.index_name,
  i.clustering_factor,
  t.num_rows,
  t.blocks
FROM user_indexes i
JOIN user_tables t ON i.table_name = t.table_name
WHERE i.table_name IN ('CUSTOMER', 'CUSTOMER_SORTED');
```

### 🔍 해설:

- `CUSTOMER_SORTED` 테이블이 정렬된 상태라면 `clustering_factor` 값이 작게 나옵니다.
- 클러스터링 팩터가 테이블의 블록 수에 가까울수록 인덱스 성능이 좋고, 행 수에 가까우면 비효율적입니다.

### 📌 4. 파티셔닝 실습 (Range + Hash 예시)

```sql
-- Range 파티셔닝
CREATE TABLE customer_part (
  cust_id     NUMBER,
  region_cd   VARCHAR2(10),
  birth_year  NUMBER,
  name        VARCHAR2(100),
  gender      CHAR(1)
)
PARTITION BY RANGE (birth_year) (
  PARTITION p1980s VALUES LESS THAN (1990),
  PARTITION p1990s VALUES LESS THAN (2000),
  PARTITION p2000s VALUES LESS THAN (2010),
  PARTITION pmax   VALUES LESS THAN (MAXVALUE)
);
```

### 🔍 해설:

- `birth_year` 기준으로 연도대별로 테이블을 나눔.
- **Range Partitioning**은 시간대별, 금액별 등 **선형적 증가**되는 데이터를 처리할 때 효율적입니다.

```sql
-- 데이터 삽입
INSERT INTO customer_part SELECT * FROM customer;
COMMIT;

-- 특정 연도 범위 조회 성능 확인
EXPLAIN PLAN FOR
SELECT * FROM customer_part WHERE birth_year BETWEEN 1985 AND 1995;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### 🔍 해설:

- 옵티마이저가 특정 파티션(p1980s, p1990s)만 읽도록 최적화(PARTITION PRUNING)됨.
- 대용량 테이블에서 필요하지 않은 파티션을 읽지 않아 성능이 향상됩니다.

### 📌 5. 해시 파티셔닝 (Hot Block 분산 목적)

```sql
CREATE TABLE customer_hash (
  cust_id     NUMBER,
  region_cd   VARCHAR2(10),
  birth_year  NUMBER,
  name        VARCHAR2(100),
  gender      CHAR(1)
)
PARTITION BY HASH (region_cd)
PARTITIONS 4;

INSERT INTO customer_hash SELECT * FROM customer;
COMMIT;
```

### 🔍 해설:

- 해시 파티셔닝은 자주 조회되는 값이 특정 블록에 집중되어 생기는 **Hot Block 현상**을 완화합니다.
- `region_cd`가 특정 값에만 집중되어 있는 상황에서 유리합니다.
