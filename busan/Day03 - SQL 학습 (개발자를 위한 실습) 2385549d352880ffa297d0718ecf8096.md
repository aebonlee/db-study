### ✅ Q18. **동일한 가입일에 가입한 고객쌍 출력 (SELF JOIN 활용)**

```sql
SELECT a.name 고객A, b.name 고객B, a.created_at
FROM customer a
JOIN customer b ON a.created_at = b.created_at AND a.cust_id < b.cust_id
```

✅ **오류 없음**

🔍 **설명:**

- `SELF JOIN`을 통해 같은 가입일(created_at)을 가진 고객쌍을 출력
- 중복 방지를 위해 `a.cust_id < b.cust_id` 조건이 적절히 사용됨
- 단, `cust_id`는 반드시 숫자형(PK)이고 고유해야 함 → 그렇다면 이 쿼리는 정확함

`a.cust_id < b.cust_id` 대신 `a.cust_id <> b.cust_id`를 쓰지 않은 이유는 **중복 쌍 방지** 때문입니다. 아래에 비교를 정리해드릴게요.

### 🔍 목적: 동일한 가입일에 가입한 "고객 쌍"을 한 번씩만 출력하기

### ① `a.cust_id <> b.cust_id` 사용 시:

**결과에 중복쌍이 포함됨**

```sql
-- 쿼리
SELECT a.name 고객A, b.name 고객B, a.created_at
FROM customer a
JOIN customer b ON a.created_at = b.created_at AND a.cust_id <> b.cust_id;
```

**예시 데이터:**

| cust_id | name | created_at |
| --- | --- | --- |
| 1 | 철수 | 2024-01-01 |
| 2 | 영희 | 2024-01-01 |

**결과:**

| 고객A | 고객B | created_at |
| --- | --- | --- |
| 철수 | 영희 | 2024-01-01 |
| 영희 | 철수 | 2024-01-01 |

### ② `a.cust_id < b.cust_id` 사용 시:

**중복 제거 + 순서 고정된 한 쌍만 출력됨**

```sql
SELECT a.name 고객A, b.name 고객B, a.created_at
FROM customer a
JOIN customer b ON a.created_at = b.created_at AND a.cust_id < b.cust_id;
```

**결과:**

| 고객A | 고객B | created_at |
| --- | --- | --- |
| 철수 | 영희 | 2024-01-01 |

> cust_id가 작은 사람이 A, 큰 사람이 B로 고정되며 한 쌍만 출력됨
> 

### ✅ 결론:

- `a.cust_id < b.cust_id`는 **고객쌍을 중복 없이 1번만 출력**하려는 목적
- `a.cust_id <> b.cust_id`는 **같은 쌍이 양방향으로 2번 출력**됨 → **불필요한 중복**

필요 시 이 쿼리를 활용하여 **중복 제거 없이 전체 조합 출력**, **자기 자신 포함 여부 선택**, 또는 **n명 그룹 매핑**까지 확장도 가능합니다.

## 💡그럼 데이터가 유니크하지 않고 더 많은 양이라면 그래도 동일 할가?

### ChatGPT의 말:

아주 중요한 관점입니다. 질문을 요약하자면:

> "데이터가 유니크하지 않고 동일한 가입일에 여러 명이 있을 경우에도 a.cust_id < b.cust_id 조건이 여전히 쌍을 중복 없이 잘 출력할 수 있는가?"
> 

👉 **정답은: 그렇다. `a.cust_id < b.cust_id`는 중복 없이 쌍을 한 번씩만 출력합니다.**

단, **중복이란 '동일한 두 고객 조합'을 의미하며, 이름이 같아도 cust_id가 다르면 별개 고객으로 간주합니다.**

---

### 📌 예시로 확인해 보겠습니다

### **예시 데이터 (중복된 이름 포함):**

| cust_id | name | created_at |
| --- | --- | --- |
| 101 | 철수 | 2024-01-01 |
| 102 | 영희 | 2024-01-01 |
| 103 | 철수 | 2024-01-01 |
| 104 | 민수 | 2024-01-01 |

---

### 🔎 `a.cust_id < b.cust_id` 조건 결과:

```sql
SELECT a.cust_id, b.cust_id
FROM customer a
JOIN customer b
ON a.created_at = b.created_at AND a.cust_id < b.cust_id;
```

**결과 (총 조합 수 = 6쌍):**

| a.cust_id | b.cust_id |
| --- | --- |
| 101 | 102 |
| 101 | 103 |
| 101 | 104 |
| 102 | 103 |
| 102 | 104 |
| 103 | 104 |

→ 총 4명 중 2명씩 조합하는 **nC2 = 6쌍** 정확히 출력됨

→ `a.cust_id < b.cust_id`는 **순서가 바뀐 중복쌍을 제거**함

---

### 🚫 반면 `a.cust_id <> b.cust_id`를 쓸 경우:

| a.cust_id | b.cust_id |
| --- | --- |
| 101 | 102 |
| 102 | 101 |
| 101 | 103 |
| 103 | 101 |
| … | … |

→ **같은 조합이 두 번 출력됨** (총 12쌍)

---

### ✅ 결론 요약

| 조건 | 중복 여부 | 설명 |
| --- | --- | --- |
| `a.cust_id < b.cust_id` | ❌ 없음 | 조합 1회만 출력 (nC2) |
| `a.cust_id <> b.cust_id` | ✅ 있음 | 같은 쌍이 2번 나옴 (n × (n - 1)) |

💡 **참고:** `cust_id`가 유니크하지 않거나 NULL이 있다면 예상치 못한 결과가 나올 수 있으므로 `PK` 또는 `NOT NULL + UNIQUE` 조건이 있는 컬럼을 비교 기준으로 사용하는 것이 좋습니다.

### ✅ Q19. **제품 가격이 평균보다 높은 제품을 출력**

```sql
SELECT * FROM product
WHERE price > (SELECT AVG(price) FROM product)
ORDER BY price DESC;
```

✅ **오류 없음**

🔍 **설명:**

- `AVG(price)`는 서브쿼리에서 스칼라값(1개 행)을 반환
- 이를 기준으로 `price > 평균` 조건을 만족하는 제품을 `DESC` 정렬
- 성능 향상 위해선 `SELECT *` 대신 필요한 컬럼만 명시하는 것이 좋음 (e.g., `product_name, price`)

---

### ✅ Q20. **고객이 존재하지 않는 부서명 출력 (NOT EXISTS)**

```sql
SELECT dept_name FROM department d
WHERE NOT EXISTS (
  SELECT 1 FROM customer c WHERE c.dept_id = d.dept_id
);
```

✅ **오류 없음**

🔍 **설명:**

- 고객 테이블(`customer`)에 해당 부서(`dept_id`)가 **없는 경우**만 출력
- `NOT EXISTS`는 NULL-safe하므로 안전하게 작동
- `SELECT 1`은 성능상 문제 없음 (관례적으로 사용하는 서브쿼리 최적 구문)

---

### 🔎 결론:

세 개의 쿼리 모두 **SQL 문법 및 논리적 오류 없이 정상적**입니다.

다만 성능 또는 명시적 가독성을 위해 다음 사항을 고려할 수 있습니다:

| 쿼리 | 개선 제안 (선택사항) |
| --- | --- |
| Q18 | `a.created_at` 외에 `a.cust_id`, `b.cust_id`를 함께 출력하면 디버깅에 유용 |
| Q19 | `SELECT product_name, price`처럼 출력 컬럼을 명시해주는 것이 좋음 |
| Q20 | `SELECT dept_id, dept_name` 같이 ID도 함께 출력하면 유용 |

# 📘 Oracle SQL 튜닝 이론 실습

내용은 다음 순서로 구성되어 있습니다:

1. 인덱스 구조 이해 및 생성
2. 인덱스 유무 성능 비교
3. 범위 조건(LIKE, BETWEEN 등)과 인덱스 활용
4. 복합 인덱스 및 컬럼 순서 중요성
5. 힌트로 인덱스 유도 (`INDEX`, `FULL`)
6. 클러스터링 팩터 실습 (정렬 vs 무작위 삽입)

🧾 이 스크립트는 일반 IT 개발자가 Oracle 튜닝을 실습할 수 있도록 설계되었습니다.

- `JOIN`, `USE_NL`, `USE_HASH` 등 조인 방식 실습
- 통계정보 수집(DBMS_STATS)과 실행계획 변화
- 바인드 변수와 SQL 재사용 실습

```sql
-- ============================================
-- [1] 인덱스 기본 구조 이해 - B-Tree 구조 체험
-- ============================================
-- 실습 목표: B-Tree 인덱스 구조와 기본 동작 원리 이해
-- 주 테이블 생성
CREATE TABLE customer (
    cust_id     NUMBER PRIMARY KEY,
    name        VARCHAR2(100),
    gender      CHAR(1),
    city        VARCHAR2(50),
    created_at  DATE
);

-- 샘플 데이터 입력
BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO customer VALUES (
      i,
      '고객_' || i,
      CASE MOD(i, 2) WHEN 0 THEN 'M' ELSE 'F' END,
      CASE MOD(i, 5) WHEN 0 THEN '서울'
                    WHEN 1 THEN '부산'
                    WHEN 2 THEN '대전'
                    WHEN 3 THEN '광주'
                    ELSE '대구' END,
      SYSDATE - MOD(i, 365)
    );
  END LOOP;
  COMMIT;
END;
/

-- 인덱스 없이 검색
EXPLAIN PLAN FOR SELECT * FROM customer WHERE name = '고객_5000';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 인덱스 생성 후 검색
CREATE INDEX idx_customer_name ON customer(name);

EXPLAIN PLAN FOR SELECT * FROM customer WHERE name = '고객_5000';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- ============================================
-- [2] 범위 조건에서의 인덱스 사용 여부 확인
-- ============================================
-- BETWEEN은 인덱스를 사용 가능
EXPLAIN PLAN FOR SELECT * FROM customer WHERE cust_id BETWEEN 100 AND 200;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- LIKE '%단어%'는 인덱스를 사용하지 않음 (앞자리가 와일드카드일 경우)
EXPLAIN PLAN FOR SELECT * FROM customer WHERE name LIKE '%고객_100';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- LIKE '단어%'는 인덱스 사용 가능
EXPLAIN PLAN FOR SELECT * FROM customer WHERE name LIKE '고객_100%';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- ============================================
-- [3] 복합 인덱스 실습 및 컬럼 순서 중요성
-- ============================================
CREATE INDEX idx_city_gender ON customer(city, gender);

-- city만 조건: 인덱스 사용
EXPLAIN PLAN FOR SELECT * FROM customer WHERE city = '부산';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- city + gender: 완전조건 - 인덱스 사용
EXPLAIN PLAN FOR SELECT * FROM customer WHERE city = '부산' AND gender = 'M';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- gender만 조건: 인덱스 사용 불가
EXPLAIN PLAN FOR SELECT * FROM customer WHERE gender = 'M';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- ============================================
-- [4] 힌트로 인덱스 사용 유도 및 차이 확인
-- ============================================
-- FULL 힌트 사용 (풀스캔 유도)
SELECT /*+ FULL(customer) */ * FROM customer WHERE name = '고객_5000';

-- INDEX 힌트 사용 (인덱스 사용 유도)
SELECT /*+ INDEX(customer idx_customer_name) */ * FROM customer WHERE name = '고객_5000';

-- 실행계획 확인용
EXPLAIN PLAN FOR SELECT /*+ INDEX(customer idx_customer_name) */ * FROM customer WHERE name = '고객_5000';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- ============================================
-- [5] 클러스터링 팩터 실습 (데이터 정렬 vs 무작위)
-- ============================================
CREATE TABLE customer_seq AS SELECT * FROM customer ORDER BY cust_id;
CREATE TABLE customer_rand AS SELECT * FROM customer ORDER BY DBMS_RANDOM.VALUE;

-- 인덱스 생성
CREATE INDEX idx_seq ON customer_seq(cust_id);
CREATE INDEX idx_rand ON customer_rand(cust_id);

-- 클러스터링 팩터 비교
SELECT INDEX_NAME, TABLE_NAME, CLUSTERING_FACTOR, NUM_ROWS FROM USER_INDEXES 
WHERE TABLE_NAME IN ('CUSTOMER_SEQ', 'CUSTOMER_RAND');

-- 실행계획 비교
EXPLAIN PLAN FOR SELECT * FROM customer_seq WHERE cust_id BETWEEN 1000 AND 2000;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

EXPLAIN PLAN FOR SELECT * FROM customer_rand WHERE cust_id BETWEEN 1000 AND 2000;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```
