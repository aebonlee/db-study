# Oracle SQL 튜닝 실무 가이드북

## 🏁 1장. SQL 튜닝의 기본 철학

> SQL 튜닝은 빠른 SQL을 만드는 기술이 아니다.
> 
> 
> **병목을 식별하고, 자원을 절약하며, 실행계획을 해석하는 구조적 사고의 과정**이다.
> 

### ✅ 핵심 키워드

- 비용 절감(COST ↓), 반복 제거(EXECUTIONS ↓), 불필요 I/O 제거(BUFFER_GETS ↓)
- “빠른 SQL”보다 “덜 낭비되는 SQL”이 좋은 SQL

---

## 📌 2장. 실행계획 해석의 기술

### 🔹 1. 실행계획 기본 구조

| 항목 | 설명 |
| --- | --- |
| Id | 실행 단계의 계층적 번호 (하위부터 실행됨) |
| Operation | 실행 연산 (TABLE ACCESS, JOIN, FILTER 등) |
| Rows | 예상 리턴 행 수 (Cardinality) |
| Cost | 옵티마이저가 판단한 상대적 수행비용 |
| Access Predicate | 인덱스에 적용된 조건절 |
| Filter Predicate | 인덱스 외 조건절 (추가 필터링) |

### 🧠 해석 팁

- Cost는 상대적 수치이므로 SQL 간 비교용
- Filter만 있고 Access가 없다면 → 인덱스 미사용 가능성 ↑
- Rows가 급격히 증가하는 지점 = 병목 가능성

## 🔍 3장. 조건절 튜닝 전략

### 🔹 1. 함수 제거

| 문제 SQL |

```sql
SELECT * FROM EMP WHERE TO_CHAR(HIREDATE, 'YYYY') = '2022';
```

| 개선 SQL |

```sql
WHERE HIREDATE BETWEEN TO_DATE('2022-01-01','YYYY-MM-DD') AND TO_DATE('2022-12-31','YYYY-MM-DD')
```

→ 함수 제거로 인덱스 활용 가능, 실행계획에서 INDEX RANGE SCAN 유도

### 🔹 2. 산술 연산 제거

```sql
-- 비효율
WHERE SAL * 1.1 > 3000

-- 효율
WHERE SAL > 3000 / 1.1
```

### 🔹 3. OR 조건 분해

```sql
-- OR 사용 시 Index Merge 또는 Full Scan
WHERE JOB = 'CLERK' OR DEPTNO = 10

-- 개선
SELECT * FROM EMP WHERE JOB = 'CLERK'
UNION ALL
SELECT * FROM EMP WHERE DEPTNO = 10
```

## 📂 4장. 인덱스 설계 및 활용 전략

### 🔹 1. 인덱스 활용 조건

- 조건절은 반드시 `컬럼 = 상수` 구조가 바람직
- 후행 컬럼만 조건에 있을 경우 → Index Skip Scan 발생 (비효율)

### 🔹 2. 함수 기반 인덱스

```sql
CREATE INDEX IDX_EMP_HIRE_YEAR ON EMP(TO_CHAR(HIREDATE, 'YYYY'));
```

- 함수 제거가 어렵다면 함수 기반 인덱스로 보완

### 🔹 3. 인덱스 유지보수 기준

| 항목 | 진단 기준 |
| --- | --- |
| BLEVEL | 3 이상이면 REBUILD 고려 |
| LEAF_BLOCKS | 급격히 증가 → 조각화 |
| CLUSTERING_FACTOR | ROWS에 가까우면 인덱스 정렬도 낮음 (효율 ↓) |
| 사용 여부 | V$OBJECT_USAGE → 사용 이력 없으면 DROP 검토 |

## 🔄 5장. 조인 튜닝 전략

### 🔹 1. 조인 방식 비교

| 방식 | 특징 | 추천 상황 |
| --- | --- | --- |
| Nested Loop | 반복 접근 | 소량 조인, 인덱스 있을 때 |
| Hash Join | 해시 테이블 생성 | 대량 집합, 인덱스 없을 때 |
| Merge Join | 정렬 기반 병합 | 정렬된 집합, 인덱스 정렬 활용 |

### 🔹 2. 조인 순서 힌트

```sql
SELECT /*+ LEADING(A B) USE_HASH(B) */ …
```

- LEADING: 조인 순서 지정
- USE_NL / USE_HASH: 조인 방식 강제

## 🔍 6장. 실행계획 수치 진단 전략

### 📈 병목 판별 기준

| 지표 | 해석 |
| --- | --- |
| COST ↑↑ | 복잡한 Join, Full Scan 가능성 |
| BUFFER GETS ↑ | 반복 접근, 캐시 낭비 |
| DISK READS ↑ | Full Scan, 인덱스 미적용 |
| EXECUTIONS ↑ | 공유 실패, 바인드 미사용 |
| ROWS ↓ | 지나친 필터링 또는 Selectivity 오류 가능 |

## 📊 7장. 옵티마이저 통계 & 히스토그램 전략

### 🔹 1. 히스토그램 필요 시점

- NDV(고유값 수)가 작고, 특정 값에 편중된 컬럼
- 예: `STATUS` 컬럼에 'Y' 90%, 'N' 10%

### 🔹 2. 수집 예시

```sql
EXEC DBMS_STATS.GATHER_TABLE_STATS(
  OWNNAME => 'SCOTT',
  TABNAME => 'EMP',
  METHOD_OPT => 'FOR COLUMNS SIZE 10 STATUS'
);
```

## ⚙ 8장. 병렬 처리 전략

### 📌 적용 방식

```sql
SELECT /*+ PARALLEL(SALES 4) */ * FROM SALES;
ALTER TABLE SALES PARALLEL 4;
```

### 📈 진단 뷰

- `V$PX_SESSION`, `V$PQ_TQSTAT`
- 실행계획에서 `PX BLOCK`, `PX SEND`, `PX COORDINATOR` 확인

## 🧠 11장. 암기용 핵심 요약

- TO_CHAR, OR, NOT IN, 함수 → 튜닝 대상
- 힌트: INDEX, FULL, LEADING, USE_NL, USE_HASH
- 인덱스 유지 기준: BLEVEL ↑, LEAF_BLOCK ↑, CLUSTERING_FACTOR ≒ ROWS
- 히스토그램: 값 분포 편중 시 적용
- 병렬 처리: 대량 집합에만, 과도 사용은 Wait 증가
