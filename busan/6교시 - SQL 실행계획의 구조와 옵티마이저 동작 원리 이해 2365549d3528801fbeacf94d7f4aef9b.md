# 6교시 - SQL 실행계획의 구조와 옵티마이저 동작 원리 이해

### 🎯 학습 목표

- 실행계획(Execution Plan)의 계층 구조와 각 연산자(Operation)의 의미를 이해한다.
- 옵티마이저(Optimizer)의 역할과 유형을 구분할 수 있다.
- 실행계획 생성에 영향을 주는 요소들(통계 정보, 힌트, SQL 구조 등)을 설명할 수 있다.
- 실제 SQL 실행계획을 읽고 판단할 수 있는 기본 역량을 갖춘다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “지금부터는 SQL 성능을 눈으로 확인하고 진단하는 가장 강력한 도구, 바로 ‘실행계획(Execution Plan)’을 배웁니다.
    
    이 실행계획은 Oracle이 내부적으로 SQL을 어떻게 처리할지를 나타내는 설계도이자 로직의 지도입니다.
    
    우리가 SQL을 튜닝할 때 단순히 빠른 SQL을 쓰는 게 아니라, Oracle이 ‘어떤 방법으로 처리하려는지’를 보고 판단해야 합니다.”
    

---

1. **[실행계획의 계층 구조 이해 – 10분]**
    - 실행계획은 **Tree 구조**로 구성되어 있음. 아래에서 위로 계산하며 실행
    - 각 행(Row Source)에는 다음 정보가 포함됨:
        - `Operation`: 어떤 방식의 접근 또는 조인인지
        - `Object`: 사용된 테이블 또는 인덱스
        - `Rows`: 예측 결과 건수
        - `Cost`: Oracle이 계산한 상대적 수행 비용
    - 예시:
        
        ```sql
        SELECT * FROM EMP WHERE DEPTNO = 10;
        ```
        
        실행계획 예:
        
        ```sql
        ----------------------------------------
        | Id | Operation                   | Cost |
        ----------------------------------------
        |  0 | SELECT STATEMENT            |   10 |
        |  1 |  TABLE ACCESS BY INDEX ROWID|   10 |
        |  2 |   INDEX RANGE SCAN EMP_IDX  |    5 |
        ----------------------------------------
        ```
        
    - 위에서부터 보면 이해 어려움 → **항상 아래에서 위로 읽어야 함**

---

1. **[옵티마이저의 종류와 동작 방식 – 10분]**
    - 옵티마이저(Optimizer)는 SQL 실행 전에 ‘가장 효율적인 경로’를 판단하는 Oracle의 지능형 판단자
    - **옵티마이저 유형:**
        - `Rule-Based Optimizer(RBO)` – 사용 안함 (11g부터 폐기)
        - `Cost-Based Optimizer(CBO)` – 현재 모든 버전 기본값
            - 통계 정보를 바탕으로 실행계획 비용(Cost)을 계산하여 최적 경로 결정
    - CBO가 판단하는 요소:
        - 테이블의 행 수, 블록 수
        - 인덱스 유무
        - 컬럼 값의 분포도
        - 조인 순서 및 방법
    
    ➤ 결론: 옵티마이저는 **수학적으로 가장 효율적인 경로를 비용 기반으로 판단**함
    

---

1. **[실습 – 실행계획 확인 및 해석 훈련 – 20분]**
    
    🔸 실습 SQL
    
    ```sql
    EXPLAIN PLAN FOR
    SELECT ENAME, SAL
    FROM EMP
    WHERE DEPTNO = 10;
    
    SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
    ```
    
    🔸 설명 포인트
    
    - `TABLE ACCESS BY INDEX ROWID`: 인덱스를 이용해 RowID로 테이블 접근
    - `INDEX RANGE SCAN`: 조건에 맞는 인덱스 범위 탐색
    
    🔸 다양한 SQL 실행계획 비교
    
    ```sql
    SELECT /*+ FULL(EMP) */ * FROM EMP WHERE DEPTNO = 10;
    SELECT /*+ INDEX(EMP EMP_DEPT_IDX) */ * FROM EMP WHERE DEPTNO = 10;
    ```
    
    - 힌트에 따라 실행계획이 어떻게 달라지는지 확인
    - `COST` 수치와 `ROWS` 수치 변화에 주목

---

1. **[옵티마이저 비용 계산 요소 설명 – 5분]**
    - COST = 디스크 I/O 수 + CPU 사용량 + 통신 비용 등을 기반으로 산출
    - 그러나 절대적 수치는 아님 → 상대적 비교용
    - 옵티마이저는 항상 가장 비용이 낮은 실행계획을 선택함
    
    ➤ `Statistics`가 정확하지 않으면 잘못된 실행계획이 선택될 수 있음
    
    ➤ 그래서 `ANALYZE TABLE`, `DBMS_STATS`로 최신 통계 정보 유지가 중요
    

---

1. **[정리 및 마무리 – 5분]**
    - 실행계획은 단순한 SQL 분석 도구가 아니라, Oracle 내부의 판단 결과를 보여주는 유일한 방법
    - 튜닝 시에는 항상 실행계획부터 보고 판단해야 함
    - “실행계획은 SQL의 의도가 아니라, Oracle의 해석이다.
        
        우리는 SQL을 쓰지만, 결국 일을 처리하는 것은 실행계획이다.”
