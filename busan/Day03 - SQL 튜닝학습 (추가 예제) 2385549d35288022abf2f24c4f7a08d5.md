## ✅ 인덱스 튜닝 실습 세트 (친절한 SQL 튜닝 3장 연계)

### 🔸 [예제 1] 조건절 컬럼 순서에 따른 인덱스 효율 실습

```sql
-- 복합 인덱스
CREATE INDEX idx_city_gender_created ON customer(city, gender, created_at);

-- (1) city + gender 조건만 사용
SELECT * FROM customer WHERE city = '서울' AND gender = 'M';

-- (2) gender만 조건 사용 → 인덱스 미사용 확인
SELECT * FROM customer WHERE gender = 'M';

-- (3) city + gender + created_at: 완전조건 → 효율적
SELECT * FROM customer WHERE city = '서울' AND gender = 'M' AND created_at > SYSDATE - 90;

-- 실행계획 비교 필수
EXPLAIN PLAN FOR ...
```

### 🔸 [예제 2] 선택도가 낮은 컬럼 인덱스 사용 실효성 비교

```sql
-- gender: 선택도 낮음 ('M'/'F') → 인덱스 효과 거의 없음
CREATE INDEX idx_gender ON customer(gender);

-- 실행계획 비교
SELECT * FROM customer WHERE gender = 'M';
SELECT * FROM customer WHERE city = '대전'; -- 선택도 높음 컬럼으로 비교
```

---

### 🔸 [예제 3] LIKE 연산자에서 인덱스 사용 가능/불가능 구분

```sql
-- 인덱스 생성
CREATE INDEX idx_name ON customer(name);

-- (1) name LIKE '홍%' → 인덱스 사용
SELECT * FROM customer WHERE name LIKE '홍%';

-- (2) name LIKE '%길동' → 인덱스 사용 안됨 (풀스캔)
SELECT * FROM customer WHERE name LIKE '%길동';

-- (3) name = '홍길동' → 정확히 인덱스 사용
EXPLAIN PLAN FOR SELECT * FROM customer WHERE name = '홍길동';
```

### 🔸 [예제 4] 인덱스 스킵 스캔(Oracle 특징) 유도 실습

```sql
-- 선행 컬럼을 조건절에 사용하지 않더라도 인덱스 스킵 스캔 발생 가능
-- 단, 옵티마이저에 따라 성능 차이 큼

CREATE INDEX idx_gender_city_created ON customer(gender, city, created_at);

-- 선행 컬럼(gender) 조건 없음 → 스킵 스캔 유도 가능
SELECT * FROM customer WHERE city = '서울' AND created_at > SYSDATE - 100;

-- 실행계획에서 INDEX SKIP SCAN 확인
```

### 🔸 [예제 5] 힌트를 통한 인덱스 강제/무시

```sql
-- (1) INDEX 힌트 사용
SELECT /*+ INDEX(customer idx_name) */ * FROM customer WHERE name = '홍길동';

-- (2) FULL 힌트로 인덱스 사용 방지
SELECT /*+ FULL(customer) */ * FROM customer WHERE name = '홍길동';
```

### 🔸 [예제 6] 클러스터링 팩터 차이로 인한 성능 차이 실험

```sql
-- customer_seq: 정렬 삽입
CREATE TABLE customer_seq AS SELECT * FROM customer ORDER BY cust_id;

-- customer_rand: 무작위 삽입
CREATE TABLE customer_rand AS SELECT * FROM customer ORDER BY DBMS_RANDOM.VALUE;

-- 인덱스 생성
CREATE INDEX idx_seq ON customer_seq(cust_id);
CREATE INDEX idx_rand ON customer_rand(cust_id);

-- 실행계획과 클러스터링 팩터 비교
SELECT INDEX_NAME, TABLE_NAME, CLUSTERING_FACTOR FROM USER_INDEXES
WHERE TABLE_NAME IN ('CUSTOMER_SEQ', 'CUSTOMER_RAND');
```
