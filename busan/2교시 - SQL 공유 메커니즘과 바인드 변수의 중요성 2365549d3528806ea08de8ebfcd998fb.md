# 2교시 - SQL 공유 메커니즘과 바인드 변수의 중요성

### 🎯 학습 목표

- SQL이 Oracle 내부에서 어떻게 공유되는지 이해한다.
- 라이브러리 캐시(Library Cache)의 구조와 역할을 설명할 수 있다.
- 바인드 변수 사용의 장점과 소프트파싱(Soft Parsing)을 유도하는 전략을 이해한다.
- SQL 공유 실패 사례와 튜닝 방향을 학습한다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “지난 시간에 SQL이 어떻게 처리되는지 전반적인 흐름을 살펴보았습니다.
    
    이번 시간에는 SQL의 재사용에 초점을 맞추어, 하드파싱을 피하고 성능을 유지하는 핵심 전략인 ‘바인드 변수’와 ‘SQL 공유’에 대해 알아보겠습니다.”
    

---

1. **[핵심 개념 설명 – 15분]**
    
    🔹 **SQL 공유(Shared SQL Area)**
    
    - Oracle은 동일한 SQL 문장을 캐시에 보관하여 여러 세션이 함께 사용할 수 있도록 함 → **Library Cache**
    - 동일한 SQL이 들어오면 새로 파싱하지 않고 기존 실행계획을 재사용 → Soft Parsing
    - SQL이 조금이라도 다르면 공유되지 않음 → Hard Parsing 발생
    
    🔹 **문자열이 다르면 공유 불가 예시**
    
    ```sql
    SELECT * FROM CUSTOMER WHERE CUST_ID = '1001';
    SELECT * FROM CUSTOMER WHERE CUST_ID = '1002';
    ```
    
    → 서로 다른 SQL로 인식됨 → 각각 Hard Parsing 발생
    
    🔹 **바인드 변수 사용**
    
    ```sql
    SELECT * FROM CUSTOMER WHERE CUST_ID = :1;
    ```
    
    → 동일한 SQL로 인식됨 → 첫 실행만 Hard Parsing, 이후 Soft Parsing
    

---

1. **[Library Cache 구조 설명 – 10분]**
    - Library Cache는 Shared Pool의 일부로서 SQL 문장, 실행계획 등을 메모리에 저장
    - SQL 해시값을 기준으로 해시 버킷(Hash Bucket)에 배치되고, 버킷 내에 LCO(Library Cache Object)로 관리됨
    - LCO는 SQL을 수행하기 위한 실행 계획, 권한 정보, 테이블/인덱스 참조 정보 등을 포함
    
    🔸 구조 흐름도 설명:
    
    `해시 값 → Hash Bucket → LCO Handle → SQL 정보 및 실행 계획`
    

---

1. **[바인드 변수 효과 실습 – 10분]**
    
    🔸 실습 목표: Hard Parsing과 Soft Parsing 비교
    
    🔸 SQL Developer 또는 SQL*Plus에서 다음 SQL 실행:
    
    ```sql
    ALTER SYSTEM FLUSH SHARED_POOL;
    SELECT * FROM EMP WHERE EMPNO = 7369;
    SELECT * FROM EMP WHERE EMPNO = 7499;
    ```
    
    → 실행 후 `V$SQL` 또는 `V$SQLAREA`에서 공유 여부 확인:
    
    ```sql
    SELECT SQL_TEXT, PARSING_SCHEMA_NAME, EXECUTIONS, CHILD_NUMBER
    FROM V$SQLAREA
    WHERE SQL_TEXT LIKE 'SELECT * FROM EMP WHERE EMPNO%';
    ```
    
    🔸 바인드 변수 적용 SQL:
    
    ```sql
    VARIABLE empno NUMBER
    EXEC :empno := 7369
    SELECT * FROM EMP WHERE EMPNO = :empno;
    ```
    
    → 이 SQL을 다시 실행하고 `V$SQLAREA`에서 동일 SQL 공유 여부 확인
    

---

1. **[바인드 변수 강제 설정: CURSOR_SHARING – 5분]**
    - 어플리케이션에서 바인드 변수 도입이 어렵다면 DB 차원에서 유도 가능
    - `CURSOR_SHARING = FORCE` 설정 시, SQL 문장이 유사하면 내부적으로 바인드 변수처럼 처리
        
        ```sql
        ALTER SYSTEM SET CURSOR_SHARING = FORCE;
        ```
        
    - 단점: 일부 실행계획 예외 상황 발생 가능 (9i 이상부터 안정성 개선)

---

1. **[마무리 및 정리 – 5분]**
    - SQL 공유를 높이면 CPU 사용량과 Parsing 부하가 급격히 줄어듭니다.
    - 성능 병목의 주요 원인이 되는 Hard Parsing은 반드시 Soft Parsing으로 유도해야 합니다.
    - **“좋은 SQL이란 빠른 SQL이 아니라, 반복되더라도 효율적으로 실행되는 SQL이다.”**

### 📝 과제

1. 아래 두 SQL을 각각 5회 반복 실행 후, `V$SQLAREA`에서 실행횟수와 SQL 공유 여부 확인
    
    ```sql
    SELECT * FROM EMP WHERE EMPNO = 7900;
    SELECT * FROM EMP WHERE EMPNO = 7521;
    ```
    
2. 바인드 변수 방식으로 작성된 SQL을 한 번 실행하고, 다른 값으로도 반복하여 실행한 후 공유 여부를 다시 확인
