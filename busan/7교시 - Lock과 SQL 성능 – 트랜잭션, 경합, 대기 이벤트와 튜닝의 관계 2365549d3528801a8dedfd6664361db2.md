# 7교시 -  Lock과 SQL 성능 – 트랜잭션, 경합, 대기 이벤트와 튜닝의 관계

### 🎯 학습 목표

- 트랜잭션과 Lock 메커니즘이 SQL 성능에 미치는 영향을 이해한다.
- DML(SQL)이 빠름에도 시스템이 느려지는 **경합(WAIT EVENT, LATCH, TX WAIT)** 상황을 진단할 수 있다.
- Lock, Deadlock, Blocking Session 등의 원인을 `V$SESSION`, `V$LOCK` 뷰를 통해 분석하고 해결 방안을 모색할 수 있다.
- SQL 튜닝과 시스템 대기 사이의 연결을 인식하고, 실무에서 발생 가능한 대기 현상에 대응할 수 있다.

1. **[도입 – 5분]**
    
    “지금까지 SQL이 느린 이유를 실행계획과 인덱스, 통계에서만 찾았습니다.
    
    그런데 실제 현장에서는 **SQL은 빠른데 응답이 안 오는** 상황이 자주 발생합니다.
    
    이는 Lock 또는 경합(Wait)이 발생한 것입니다.
    
    이번 시간에는 SQL 성능의 숨겨진 병목 요소인 **락과 대기 이벤트**를 집중 분석하겠습니다.”
    

---

1. **[트랜잭션과 락의 구조 이해 – 10분]**
    
    ▶ Oracle의 기본 락 구조:
    
    | 구분 | 의미 | 설명 |
    | --- | --- | --- |
    | TX | Transaction Lock | DML에 의해 생성됨 |
    | TM | Table Lock | 테이블 전체 구조 잠금 |
    | UL | User Lock | 응용에서 생성한 사용자 락 |
    - 기본적으로 **Oracle은 Row-Level Lock 기반**
    - 하지만 트랜잭션이 커질수록 Lock 유지 시간 증가 → Blocking 발생
    - `COMMIT` 또는 `ROLLBACK`이 발생해야 Lock 해제됨

---

1. **[WAIT EVENT의 종류 및 진단 쿼리 – 10분]**
    
    ▶ 주요 WAIT EVENT:
    
    | 대기 이벤트 | 원인 | 조치 |
    | --- | --- | --- |
    | enq: TX – row lock contention | 트랜잭션 간 Lock 충돌 | 트랜잭션 크기 최소화 |
    | library cache latch | 공유 커서 경합 | 바인드 변수 사용 |
    | log file sync | 커밋 성능 저하 | 커밋 빈도 조정 |
    | buffer busy waits | 핫 블록 경합 | SQL 튜닝 + Block 분산 |
    
    ▶ 진단 SQL:
    
    ```sql
    SELECT SID, EVENT, BLOCKING_SESSION, STATUS
    FROM V$SESSION
    WHERE WAIT_CLASS != 'Idle';
    ```
    
    ```sql
    SELECT * FROM V$LOCK WHERE BLOCK = 1;
    ```
    

---

1. **[실습 – 락 경합 유도 및 대기 진단] – 15분**
    
    ▶ 세션 A:
    
    ```sql
    UPDATE EMP SET SAL = SAL + 100 WHERE EMPNO = 7369;
    -- COMMIT 하지 않고 HOLD 상태
    ```
    
    ▶ 세션 B:
    
    ```sql
    UPDATE EMP SET SAL = SAL + 200 WHERE EMPNO = 7369;
    -- 대기 상태 진입
    ```
    
    ▶ 세션 C (관리자):
    
    ```sql
    SELECT SID, BLOCKING_SESSION, EVENT
    FROM V$SESSION
    WHERE WAIT_CLASS != 'Idle';
    ```
    
    ▶ 확인:
    
    - 세션 B → enq: TX – row lock contention 발생
    - BLOCKING_SESSION 값으로 대기 원인 추적 가능
    - `V$LOCK`, `V$SESSION_WAIT`를 통해 Wait 상세 확인

---

1. **[락과 튜닝의 관계 설명 – 10분]**
    - SQL 튜닝으로도 해결되지 않는 문제 = **Lock 대기**
    - 트랜잭션 튜닝도 함께 고려해야 함:
        - COMMIT 위치 조정
        - DML 분할 처리
        - `NOWAIT`, `SKIP LOCKED` 옵션 활용
    - 비즈니스 단에서 락 해소 구조 설계 병행 필요

---

1. **[정리 및 마무리 – 5분]**
    - SQL 성능 문제는 단순한 실행 속도 문제가 아니다
    - 시스템 전체에서 '대기'가 발생하면 **어떤 SQL도 튜닝 효과 없음**
    - “튜닝은 옵티마이저를 설득하는 일인 동시에,
        
        시스템의 병목을 제거하는 ‘합리적 대화’다.”
