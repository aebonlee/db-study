# 5교시 - 고난도 SQL 튜닝 종합 실습 II – 병목 추적, 실행계획 재작성 및 리팩토링

### 🎯 학습 목표

- 복잡한 SQL에서 발생하는 병목 지점을 실행계획과 통계정보를 통해 식별할 수 있다.
- SQL을 리팩토링하여 가독성과 성능을 동시에 향상시키는 전략을 설계할 수 있다.
- 서브쿼리, 복잡한 조인, 불필요한 조건절 등의 리팩토링 대상 패턴을 인식하고 개선할 수 있다.
- 수치 기반으로 튜닝 전후 성능 차이를 평가하고 문서화할 수 있다.

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “실무에서는 단순한 SELECT 문보다 **5줄 이상, 조인 3개 이상, 조건절 5개 이상**인 SQL이 더 흔합니다.
    
    이럴 경우 실행계획 해석만으로는 부족하고, **병목을 중심으로 구조 전체를 리팩토링**해야 합니다.
    
    오늘은 복잡 SQL의 실행 흐름을 쪼개고, 병목을 추적하고, 최종적으로 SQL 자체를 구조적으로 재작성하는 전략을 실습합니다.”
    

---

1. **[병목 진단 대상 SQL 제시 – 5분]**
    
    🔸 예제 SQL (중첩 조인 + 서브쿼리 + 함수 + 불필요 조건)
    
    ```sql
    sql
    복사편집
    SELECT E.ENAME, D.DNAME, S.GRADE
    FROM EMP E, DEPT D, SALGRADE S
    WHERE E.DEPTNO = D.DEPTNO
      AND E.SAL BETWEEN S.LOSAL AND S.HISAL
      AND TO_CHAR(E.HIREDATE, 'YYYY') = '1982'
      AND E.DEPTNO IN (
        SELECT DEPTNO FROM DEPT WHERE LOC LIKE '%CHICAGO%'
      );
    
    ```
    
    ➤ 문제점 요약:
    
    - TO_CHAR → 인덱스 무효화
    - IN 서브쿼리 → Nested Loop 처리 위험
    - DEPT 테이블 중복 접근
    - 조건절 중복 (DEPTNO Join + IN 서브쿼리)

---

1. **[실행계획 분석 및 병목 추적 – 10분]**
    
    🔸 `EXPLAIN PLAN` 수행
    
    ```sql
    EXPLAIN PLAN FOR ...
    SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
    ```
    
    - `TABLE ACCESS FULL`, `FILTER`, `INDEX SKIP SCAN`, `NESTED LOOP` 확인
    - COST 및 예상 ROWS, BUFFER GETS 체크
    
    ➤ 병목 포인트 식별:
    
    - `TO_CHAR` 인덱스 미사용
    - SALGRADE BETWEEN → RANGE 조인 병목
    - DEPT 중복 접근

---

1. **[SQL 리팩토링 전략 수립 및 구조 재작성 – 15분]**
    
    🔧 개선 SQL 구조 (1차 리팩토링)
    
    ```sql
    SELECT E.ENAME, D.DNAME, S.GRADE
    FROM EMP E
      JOIN DEPT D ON E.DEPTNO = D.DEPTNO
      JOIN SALGRADE S ON E.SAL BETWEEN S.LOSAL AND S.HISAL
    WHERE E.HIREDATE BETWEEN TO_DATE('1982-01-01','YYYY-MM-DD')
                          AND TO_DATE('1982-12-31','YYYY-MM-DD')
      AND D.LOC LIKE '%CHICAGO%';
    ```
    
    - 서브쿼리 → JOIN 전환
    - 함수 제거
    - DEPT 중복 접근 제거
    
    ➤ 실행계획 상 변경 확인: `INDEX RANGE SCAN`, `HASH JOIN`, `COST` 감소
    

---

1. **[성능 비교 및 결과 정리 – 10분]**
    
    
    | 지표 | 기존 SQL | 리팩토링 후 |
    | --- | --- | --- |
    | COST | 250 | 35 |
    | ELAPSED_TIME | 1.8s | 0.2s |
    | BUFFER_GETS | 9,500 | 720 |
    | DISK_READS | 1,200 | 85 |
    
    ➤ 실행계획 분석 결과 보고서 작성
    
    ➤ 튜닝 대상 SQL → 리팩토링 → 검증 → 수치로 설명하는 과정 훈련
    

---

1. **[정리 및 마무리 – 5분]**
    - 실무 SQL은 복잡하고, 가독성은 낮고, 병목은 숨겨져 있다
    - SQL 리팩토링은 튜닝보다 더 고급 기술이며, **해석력과 설계력이 동시에 요구됨**
    - “SQL은 단순한 명령어가 아니라 프로그램이다.
        
        리팩토링이란, 그 프로그램을 다시 설계해 성능과 가독성을 동시에 높이는 작업이다.”
