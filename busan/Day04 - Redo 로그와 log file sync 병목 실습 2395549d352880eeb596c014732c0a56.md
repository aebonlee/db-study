# Day04 - Redo 로그와 log file sync 병목 실습

### 🎯 학습 목표

- Redo 로그 구조와 역할, LGWR(Log Writer)의 작동 방식을 이해한다.
- `log file sync`, `log file parallel write`, `log buffer space` 등 주요 관련 Wait Event를 분석한다.
- 잦은 COMMIT으로 인한 병목을 실습하고, 효과적인 개선 전략을 적용한다.

---

## 📘  Redo 로그와 log file sync 병목 실습

| 항목 | 설명 |
| --- | --- |
| **Redo Log Buffer** | 변경 내용을 메모리에 일시 저장 후 Log Writer가 디스크에 기록 |
| **log file sync** | 사용자가 COMMIT → LGWR이 Redo 로그에 기록 완료될 때까지 대기 |
| **log file parallel write** | LGWR가 실제 Redo 로그 파일에 쓰는 작업 |
| **log buffer space** | Redo Buffer가 가득 찼을 때 발생 (LGWR가 느리면 동반됨) |

## 🧪 실습 2: log file sync 병목 유발

### 1. 테스트 테이블 생성

```sql
CREATE TABLE redo_test (
  id NUMBER,
  msg VARCHAR2(100)
);
```

### 2. COMMIT 빈도 높은 반복 작업 수행

```sql
-- Session A
BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO redo_test VALUES (i, '데이터_' || i);
    COMMIT; -- COMMIT을 매번 실행 → 병목 원인
  END LOOP;
END;
```

👉 다른 세션에서 `log file sync`, `log file parallel write` Wait Event 발생 확인

## 🧪 실습 3: Wait Event 실시간 확인

```sql
-- 현재 세션들의 Wait Event 모니터링
SELECT sid, event, wait_class, state, seconds_in_wait
FROM v$session
WHERE event LIKE 'log file%' AND status = 'ACTIVE';

-- 시스템 전체 log 관련 대기 시간 확인
SELECT event, total_waits, time_waited, average_wait
FROM v$system_event
WHERE event IN ('log file sync', 'log file parallel write', 'log buffer space');
```

## 🧪 실습 4: 개선 전략 – COMMIT 지연 및 BATCH 처리

### 개선 전: 매 INSERT 후 COMMIT

```sql
-- Bad Practice
FOR i IN 1..1000 LOOP
  INSERT INTO redo_test VALUES (i, 'X');
  COMMIT;
END LOOP;
```

### 개선 후: N건 단위로 COMMIT

```sql
-- Good Practice: 100건 단위로 Commit
DECLARE
  commit_interval NUMBER := 0;
BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO redo_test VALUES (i, 'X');
    commit_interval := commit_interval + 1;
    IF MOD(commit_interval, 100) = 0 THEN
      COMMIT;
    END IF;
  END LOOP;
  COMMIT;
END;
```

👉 `log file sync` 대기 시간이 개선되는지 비교 실습
