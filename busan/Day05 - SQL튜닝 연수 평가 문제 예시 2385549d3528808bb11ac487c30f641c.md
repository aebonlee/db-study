## ✅ 단답형 및 서술형 문제 (5문항)

**1. [단답형]**

`db file sequential read`와 `db file scattered read`의 차이를 간단히 서술하시오.

> 예시 답안:
> 
> 
> db file sequential read는 주로 인덱스를 통한 단일 블록 접근 시 발생하며 Random I/O이다.
> 
> db file scattered read는 Full Table Scan 등 다중 블록을 동시에 읽을 때 발생하며 Sequential I/O이다.
> 

> 풀이:
> 
- `db file sequential read`: 주로 **Index Scan** 또는 Rowid 기반의 단일 블록 액세스에서 발생하며, **Random I/O** 유형이다.
- `db file scattered read`: 주로 **Full Table Scan** 시 발생하며 여러 블록을 한 번에 읽는 **Sequential I/O**이다.
    
    ⇒ 즉, sequential은 1블록씩 랜덤하게, scattered는 여러 블록을 한꺼번에 읽는 방식이다.
    

---

**2. [단답형]**

SQL 튜닝 시 ‘바인드 변수’를 사용하는 이유는 무엇인가?

> 풀이:
> 
- 바인드 변수는 **SQL의 실행 계획을 공유**하여 **Hard Parsing을 줄이고 Soft Parsing을 유도**한다.
- 그 결과 **CPU 사용량 감소**, **Library Cache 경합 완화**, **성능 향상** 효과가 있다.

---

**3. [단답형]**

Oracle의 Buffer Cache에서 블록을 관리하기 위한 대표적인 두 가지 메커니즘(LRU와 …)은 무엇인가?

> 풀이:
> 
- **LRU 리스트 (Least Recently Used)**
- **Hash Bucket**

이 두 구조는 Buffer Cache 내 블록의 위치를 빠르게 찾기 위해 사용되며, LRU는 사용 빈도 기반 블록 교체, Hash는 DBA 주소 기반 검색을 의미한다.

---

**4. [서술형]**

Shared Pool에서 Literal SQL이 성능에 미치는 영향과 개선 방안을 설명하시오.

> 풀이:
> 
- Literal SQL은 **바인드 변수를 사용하지 않고 값이 고정된 SQL**을 의미한다.
- 이 경우 SQL 문장이 조금이라도 다르면 **매번 Hard Parsing이 발생**하여 Library Cache 사용량과 CPU 부하가 증가하고, Latch 경합이 심화되어 전체 시스템 성능이 저하된다.
- 특히 동시 접속자가 많은 OLTP 시스템에서는 이러한 현상이 더욱 심각하다.
- **개선 방안**으로는 SQL에 **바인드 변수(:1, :cust_id 등)를 사용하는 방식으로 변경하거나**, `CURSOR_SHARING=FORCE` 설정을 통해 유사 SQL을 공유되도록 유도한다.

---

**5. [서술형]**

AWR 리포트에서 ‘log file sync’ 이벤트가 빈번하게 발생할 때, 가능한 원인과 해결 방안을 설명하시오.

> 풀이:
> 
- **log file sync**는 사용자가 **COMMIT**을 실행할 때, LGWR(Log Writer)가 **Redo Log Buffer의 내용을 리두 로그 파일로 쓰는 동안 대기**하는 이벤트이다.
- 이 이벤트가 많다는 것은 시스템에 **빈번한 커밋이 발생**하거나, **Redo 로그 디스크의 I/O 성능이 낮은 경우**가 많다는 것을 의미한다.\

**원인 예시:**

1. 커밋이 과도하게 자주 호출됨 (예: PL/SQL 루프마다 1건씩 커밋)
2. Redo 로그 파일이 **RAID 5와 같은 느린 디스크**에 존재하거나, DR 복제 환경에서 동기화 지연
3. Redo Log Buffer 크기 부족

**해결 방안:**

- 커밋을 **일괄 처리(batch commit)** 방식으로 조정
- Redo 로그 파일의 **디스크 성능 개선 (RAID 10 구성 등)**
- **Redo Log Buffer 크기 증가** (`LOG_BUFFER` 파라미터 조정)
- LGWR 활동 분석 및 시스템 전체적인 커밋 트래픽 최적화
