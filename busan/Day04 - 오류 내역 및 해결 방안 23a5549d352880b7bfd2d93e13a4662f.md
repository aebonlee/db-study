# Day04 - 오류 내역 및 해결 방안

## **(친절한 SQL 4,5,6장 예제 오류 포함)**

## ▶ 왜 생기는가 ▶ 증상이 어떻게 보이는가 ▶ 확실한 해결 절차

“문제가 실제로 어떻게 터지는지” → 제시된 교정책을 바로 적용해 “어떻게 사라지는지”를 체험해 보시면, 추후 실무 서버에서도 동일한 패턴을 훨씬 빠르게 진단하실 수 있습니다.

# 🗂 18 가지 오류 해설 및 예제

각각에 대해 **실제로 오류를 유발하는 짧은 SQL 예제**와 **즉석에서 교정하는 예제를** 통해 작은 데모 스키마 EMP/DEPT, 혹은 `dual` 테이블만 사용했으므로 별도 객체 생성 부담이 없이 실습이 가능합니다.

### 📌 Tip

- 위험한 시스템 파라미터 변경은 **테스트 인스턴스**에서만 시도하시고, 작업 후 반드시 원래 값으로 복구하세요.
- 대량 I/O·Temp 사용 예제는 Flash Recovery나 ASM 디스크 용량에 여유가 있는지 먼저 확인하세요.
- **DDL 과 병렬 작업** 은 낮은 부하 시간대에 실행하여 Library Latch, Redo 경합을 피하세요.
- 실습 시 **Flashback / 테스트 스키마** 를 별도로 두어 원본 데이터 손상을 방지합니다.
- 오류 원인을 빠르게 분류하려면 “**DB Time → Wait Class → 특정 Event**” 경로로 drill-down 하는 방법이 가장 효율적입니다.

### 1. 11g에서 `FETCH FIRST n ROWS ONLY` 사용-불가

- **문맥** : 교안 예제가 12c ANSI LIMIT 문법(`FETCH FIRST 2 ROWS ONLY`)을 씀.
- **원인** : 11g SQL 파서는 ANSI 2008 LIMIT 절을 알지 못한다.
- **증상** : `ORA-00933: SQL command not properly ended`(SQL*Plus) 또는 `ORA-00905: missing keyword`(툴에 따라).
- **해결** : 정렬을 인라인뷰로 감싼 뒤 바깥에서 `ROWNUM <= n` 을 필터하거나, 12c 이상 환경에서 실습.

🧪  11g에서 `FETCH FIRST n ROWS ONLY` **예제**

```sql
-- 오류 유발 (11g)
SELECT ename, sal
FROM   emp
ORDER  BY sal DESC
FETCH  FIRST 2 ROWS ONLY;   -- ORA-00933

-- 교정
SELECT *
FROM   ( SELECT ename, sal FROM emp ORDER BY sal DESC )
WHERE  ROWNUM <= 2;
```

### 2. `ORDER BY` 뒤에 바로 `ROWNUM` 필터를 걸었을 때 결과가 뒤틀림

- **문맥** : “상위 2건만 가져오되 급여순 정렬” 예제.
- **원인** : 옵티마이저는 먼저 ROWNUM을 붙이고 **그다음** 정렬한다. 즉 2건만 뽑아 놓고 뒤늦게 SORT.
- **증상** : 기대와 달리 급여가 가장 높은 두 명이 아님.
- **해결** : 먼저 서브쿼리에서 정렬하고 바깥쪽에서 ROWNUM을 자르는 전통 구조
    
    ```sql
    SELECT *
    FROM (SELECT * FROM emp ORDER BY sal DESC)
    WHERE ROWNUM <= 2;
    ```
    
    🧪  `ROWNUM` + `ORDER BY` 순서 뒤틀림 **예제**
    
    ```sql
    -- 잘못된 패턴 – 먼저 2행 잘라서 정렬 → 결과 뒤섞임
    SELECT ename, sal
    FROM   emp
    WHERE  ROWNUM <= 2
    ORDER  BY sal DESC;
    
    -- 올바른 패턴 – 정렬 후 외부에서 ROWNUM
    SELECT *
    FROM  (SELECT ename, sal FROM emp ORDER BY sal DESC)
    WHERE ROWNUM <= 2;
    ```

### 3. `ORA-00904: invalid identifier` — NL-Join 작성 중 컬럼 식별자 오류

- **문맥** : `USE_NL` 힌트로 강제 NL 조인을 시도하다가 컬럼 이름을 잘못 적은 경우.
- **원인** : 별칭을 줬는데 접두사를 빼먹었거나, 실제 존재하지 않는 컬럼·타이핑 오류.
- **증상** : 파서 단계에서 즉시 실패. 실행계획도 생성되지 않는다.
- **해결** :
    1. 각 테이블별 컬럼 목록을 다시 확인.
    2. 조인 조건에 항상 **별칭.컬럼** 형태를 사용.
    3. IDE(예: SQL Developer) 자동 완성으로 오타를 원천 차단.

🧪  식별자 오류 `ORA-00904` **예제**

```sql
-- 오타 : EMPID 는 존재하지 않는 컬럼
SELECT emp.empid, dept.dname
FROM   emp  JOIN dept ON emp.deptno = dept.deptno;
-- ORA-00904: "EMP"."EMPID": invalid identifier

-- 교정
SELECT emp.empno, dept.dname
FROM   emp  JOIN dept ON emp.deptno = dept.deptno;
```

### 4. Sort-Merge Join 강제 후 소트 과부하

- **문맥** : 대량 테이블 두 개를 SM 조인하려고 `USE_MERGE` 힌트를 줌.
- **원인** : SM 조인은 양쪽 집합을 모두 정렬해야 하는데, 조인 컬럼이 함수 호출(`NVL`, `SUBSTR` 등)이나 부등식(`!=`)이면 정렬 키를 만들 수 없어 대규모 TEMP 소트를 일으킨다.
- **증상** : 실행계획에 “MERGE JOIN”이 뜨지만 뒤에 “SORT JOIN” 스텝이 추가, SQL Mon에서 `direct path read temp` 폭증.
- **해결** :
    - 조인 컬럼을 순수한 단일 컬럼으로 맞추거나 함수-기반 인덱스 채택.
    - SM 조인이 반드시 유리한 조건(대량/이미 정렬된 입력)이 아니면 힌트 제거.

🧪  강제 Merge Join 후 과도한 소트 **예제**

```sql
-- 문제 : 함수 포함된 조인 컬럼 + 강제 USE_MERGE
SELECT /*+ USE_MERGE(e d) */
       e.empno, d.dname
FROM   emp e
JOIN   dept d
ON     SUBSTR(e.job,1,3) = SUBSTR(d.dname,1,3);  -- 대량 SORT 발생

-- 교정 : 조인 키를 순수 컬럼으로 맞추거나 FB 인덱스
ALTER TABLE emp  ADD job3 VARCHAR2(3)  GENERATED ALWAYS AS (SUBSTR(job,1,3));
ALTER TABLE dept ADD dname3 VARCHAR2(3) GENERATED ALWAYS AS (SUBSTR(dname,1,3));
CREATE INDEX emp_job3_ix  ON emp(job3);
CREATE INDEX dept_dn3_ix  ON dept(dname3);

SELECT /*+ USE_HASH(e d) */
       e.empno, d.dname
FROM   emp e
JOIN   dept d
ON     e.job3 = d.dname3;
```

### 5. `ORA-12801` — Hash Join 또는 병렬 쿼리 중 “error signaled in parallel query server”

- **문맥** : PQ(병렬) + 해시 조인을 돌리는데 슬레이브 한 곳에서 예외 발생.
- **주요 원인 2가지**
    1. PGA가 부족해 해시 테이블을 디스크로 반복 스필(over-allocation)하다 발생.
    2. TEMP 파일 부족.
- **증상** : 스택 트레이스에 `kghalo` PGA 할당 실패 메시지 또는 `ORA-1652` 동반.
- **해결** :
    - `PGA_AGGREGATE_TARGET` 및 TEMP 테이블스페이스 용량을 늘린다.
    - 병렬도(`PARALLEL` 힌트, `parallel_degree_policy`)를 한 단계 낮춘다.

🧪  `ORA-12801` (PGA 부족 / 해시 조인) **예제**

```sql
-- 극한 테스트 : 의도적으로 PGA를 줄여 오류 유발
ALTER SYSTEM SET pga_aggregate_target = 20M;  -- 실습 DB만!

SELECT /*+ PARALLEL(e 8) PARALLEL(d 8) USE_HASH(e d) */
       COUNT(*)
FROM   emp e CROSS JOIN dept d;   -- 대량 Hash → ORA-12801

-- 교정
ALTER SYSTEM SET pga_aggregate_target = 512M;
-- 또는 병렬도↓ / USE_MERGE 로 전환
```

### 6. 서브쿼리 반복 계산으로 실행계획 Cost 치솟음

- **문맥** : 다중 `IN (SELECT …)` 또는 동일 서브쿼리를 2번 이상 포함.
- **원인** : 옵티마이저가 뷰-머지(View Merge) 변환을 하지 못해 매번 서브쿼리를 풀 스캔.
- **증상** : A-라인에 같은 VIEW가 두 번, Buffer Gets 급증.
- **해결** :
    - 공통 서브쿼리를 `WITH t AS (…)` 구조로 빼고 `MATERIALIZE` 힌트를 줘 1회만 평가.
    - 서브쿼리를 실제 물리 뷰(또는 글로벌 임시 테이블)에 담아 재사용.

🧪  서브쿼리 반복 계산 **예제**

```sql
-- 동일 서브쿼리를 두 번 사용 → 2회 풀스캔
SELECT ename
FROM   emp
WHERE  empno IN (SELECT empno FROM emp WHERE sal > 1000)
   AND empno IN (SELECT empno FROM emp WHERE comm IS NOT NULL);

-- 교정 : WITH 절로 1회만 평가
WITH hi AS (
     SELECT /*+ MATERIALIZE */ empno
     FROM   emp
     WHERE  sal > 1000 AND comm IS NOT NULL
)
SELECT ename
FROM   emp
WHERE  empno IN (SELECT empno FROM hi);
```

### 7. `ORA-02248: invalid option for ALTER SESSION` — SORT_AREA_SIZE 직접 지정 시

- **문맥** : 실습 중 `ALTER SESSION SET SORT_AREA_SIZE=800M` 을 바로 실행.
- **원인** : 세션 정책이 기본값 `WORKAREA_SIZE_POLICY=AUTO` 인 상태에서 수동 파라미터를 지정했기 때문.
- **해결** :
    1. 먼저 `ALTER SESSION SET WORKAREA_SIZE_POLICY=MANUAL;`
    2. 그다음 `SORT_AREA_SIZE`, `HASH_AREA_SIZE` 등을 설정.
    3. 실습 후 반드시 `WORKAREA_SIZE_POLICY=AUTO`로 복구.

🧪  `ORA-02248`  (자동 워크에어리어 정책) **예제**

```sql
-- AUTO 상태에서 SORT_AREA_SIZE 직접 지정
ALTER SESSION SET sort_area_size = 800M;  -- ORA-02248

-- 교정
ALTER SESSION SET workarea_size_policy = MANUAL;
ALTER SESSION SET sort_area_size         = 800M;
-- 실습 끝나면
ALTER SESSION SET workarea_size_policy = AUTO;
```

### 8. 대규모 정렬 중 TEMP 부족 `ORA-1652: unable to extend temp segment`

- **문맥** : `ORDER BY`, `GROUP BY`, 병렬 해시 조인, CREATE INDEX 등.
- **원인** : PGA에서 처리 못한 정렬·해시 데이터가 TEMP 로 스필되는데 공간 고갈.
- **증상** : SQL이 중단되고 ORA-1652가 세션 로그에 찍힘.
- **해결** :
    - 가장 빠른 조치 : TEMPFILE 추가 `ALTER TABLESPACE temp ADD TEMPFILE ... SIZE 20G;`
    - 근본 : `PGA_AGGREGATE_TARGET` 상향·자동 정책 유지, 불필요한 대량 정렬 SQL 튜닝.

🧪  TEMP 부족 `ORA-1652` **예제**

```sql
-- PGA 의도적 축소 후 대량 정렬
ALTER SYSTEM SET pga_aggregate_target = 50M;   -- 실습 DB 한정
SELECT /*+ FULL(e) ORDERED */ *
FROM   emp e, emp e2, emp e3
ORDER  BY e.sal, e2.sal, e3.sal;               -- ORA-1652

-- 교정
ALTER SYSTEM SET pga_aggregate_target = 512M;
-- 또는 TEMPFILE 추가
ALTER TABLESPACE temp ADD TEMPFILE '/u01/app/temp02.dbf' SIZE 10G;
```

### 9. `ORA-01435: user does not exist` — `ALTER SESSION SET CURRENT_SCHEMA`

- **문맥** : 실습용 스키마 `DMLTEST` 를 아직 만들지 않았는데 CURRENT_SCHEMA 전환 시도.
- **원인** : Oracle은 CURRENT_SCHEMA 대상이 반드시 **사전에 존재** 해야 한다.
- **해결** :
    1. `CREATE USER DMLTEST IDENTIFIED BY ...` + 권한 부여.
    2. 이후 `ALTER SESSION SET CURRENT_SCHEMA=DMLTEST;`
    3. 권한 없이 전환해야 한다면 VIEW 또는 SYNONYM으로 우회 가능.

🧪  `ORA-01435` (CURRENT_SCHEMA 대상 없음) **예제**

```sql
ALTER SESSION SET current_schema = dmltest;   -- ORA-01435

-- 교정 : 스키마 먼저 생성
CREATE USER dmltest IDENTIFIED BY demo;
GRANT connect, resource TO dmltest;
ALTER SESSION SET current_schema = dmltest;
```

### 10. 파티션 DDL 오류 `ORA-00906: missing left parenthesis`

- **문맥** : 교안 예제처럼 컬럼 정의 없이 바로 `PARTITION BY RANGE` 절을 작성.
- **원인** : Oracle은 **컬럼 리스트** → `PARTITION BY` 순서를 요구한다.
- **해결** : 테이블 구조를 먼저 적고 그다음 파티션 절을 기술.
    
    ```sql
    CREATE TABLE big_sales (
       sale_id NUMBER, sale_dt DATE, ...
    )
    PARTITION BY RANGE (sale_dt) (
       PARTITION p_old  VALUES LESS THAN (DATE '2024-01-01'),
       PARTITION p2024 VALUES LESS THAN (DATE '2025-01-01'),
       PARTITION pmax  VALUES LESS THAN (MAXVALUE)
    );
    ```
    

🧪  파티션 DDL 왼쪽 괄호 누락 **예제**

```sql
-- 컬럼 정의 빠짐
CREATE TABLE x PARTITION BY RANGE (sale_dt) (
  PARTITION p1 VALUES LESS THAN (DATE '2025-01-01')
);   -- ORA-00906

-- 교정
CREATE TABLE x (
  sale_dt DATE,
  amt     NUMBER
)
PARTITION BY RANGE (sale_dt) (
  PARTITION p1 VALUES LESS THAN (DATE '2025-01-01')
);
```

### 11. `ORA-14001: partition bound must be specified`

- **문맥** : 파티션을 추가하면서 `VALUES LESS THAN` 절 누락, HIGH VALUE 미지정.
- **원인** : RANGE 파티션은 각 세그먼트마다 경계값을 의무적으로 가져야 한다.
- **해결** : 누락된 파티션 정의에 경계값·MAXVALUE 지정, 키 타입 일치 확인.

🧪  파티션 경계값 누락 `ORA-14001` **예제**

```sql
-- 경계값 미지정
ALTER TABLE x ADD PARTITION p2;  -- ORA-14001

-- 교정
ALTER TABLE x ADD PARTITION p2 VALUES LESS THAN (DATE'2026-01-01');
```

### 12. 병렬 DML 오류 `ORA-12838: cannot read/modify an object after modifying it in parallel`

- **문맥** : `INSERT /*+ APPEND PARALLEL */` 후 같은 세션에서 즉시 SELECT·UPDATE 시도.
- **원인** : 병렬 DML은 트랜잭션이 끝날 때까지 해당 오브젝트에 대해 “돌려막기”를 금지한다.
- **해결** : 병렬 DML 실행 → 바로 **COMMIT** → 이어서 다른 작업, 또는 별도 세션에서 조회.

🧪 병렬 DML 후 재조회 `ORA-12838` **예제**

```sql
INSERT /*+ APPEND PARALLEL(emp,4) */
INTO   emp
SELECT * FROM emp WHERE ROWNUM <= 1000;

SELECT COUNT(*) FROM emp;   -- ORA-12838 같은 세션

-- 교정
COMMIT;              -- 병렬 DML 종료
SELECT COUNT(*) FROM emp;   -- 정상
```

### 13. Direct-Path Insert 중 `ORA-00060: deadlock detected` (내부 DX 락)

- **문맥** : `INSERT /*+ APPEND */` 을 인덱스가 달린 큰 테이블에서 동시에 실행.
- **원인** : Append 모드는 세그먼트 단독락을 잡는데, 두 세션이 서로 상대 인덱스 블록을 먼저 잠그면서 교착.
- **해결** :
    - 대량 적재 전에 인덱스 Drop → 적재 후 Rebuild.
    - 인덱스를 유지해야 하면 APPEND 대신 NOAPPEND + 병렬도 축소.

🧪 Direct-Path Insert 데드락 **예제**

```sql
-- 두 세션에서 동시에 실행
INSERT /*+ APPEND */ INTO big_tab SELECT * FROM big_tab WHERE ROWNUM < 2;
-- 얼마 안가 ORA-00060 데드락 가능

-- 교정 : 인덱스 잠시 drop
ALTER TABLE big_tab DROP INDEX big_tab_pk;
-- INSERT 후
ALTER TABLE big_tab ADD CONSTRAINT big_tab_pk PRIMARY KEY(id);
```

### 14. `ORA-12827: insufficient parallel query slaves available`

- **문맥** : 병렬 힌트 수준이 인스턴스 한계(PARALLEL_MAX_SERVERS)보다 높음.
- **해결** :
    - DBA가 `PARALLEL_MAX_SERVERS`, `PARALLEL_MIN_SERVERS` 값을 업무 특성에 맞게 상향.
    - SQL 힌트에서 병렬도(`PARALLEL(x, 2)` 등) 조정.
    - GV$PX_PROCESS_SYSSTAT로 현재 슬레이브 사용량 확인.

🧪 병렬 슬레이브 부족 `ORA-12827` **예제**

```sql
-- 병렬도 64 힌트 (테스트 인스턴스는 8개 제한)
SELECT /*+ PARALLEL(emp 64) */ COUNT(*) FROM emp;
-- ORA-12827

-- 교정
SELECT /*+ PARALLEL(emp 4) */ COUNT(*) FROM emp;
-- 또는 DBA : ALTER SYSTEM SET parallel_max_servers = 64;
```

### 15. 로그 경합 `log file sync`, `log file parallel write`, `log buffer space`

- **문맥** : 실습 DB를 초당 수백 건 COMMIT 하도록 스크립트를 돌리면 관측 가능.
- **원인** :
    1. 자잘한 트랜잭션을 각각 COMMIT → 동기 I/O 폭주.
    2. Redo 로그 디스크가 RAID-5, NAS 등 느린 미디어.
    3. Redo 로그 파일이 작아 자주 Switch → `log file switch completion`.
- **해결** :
    - 논리적으로 묶을 수 있는 작업은 한 트랜잭션으로 합쳐 COMMIT 빈도 줄이기.
    - Redo 전용 SSD / RAID1+0로 이관.
    - 로그 파일 크기를 크게(예 2 GB 이상) 잡아 Switch 회수를 감소.

🧪 로그 경합 (log file sync · buffer space) **예제**

```sql
-- 초당 1,000건 커밋 스크립트
BEGIN
  FOR i IN 1..100000 LOOP
    INSERT INTO t_log VALUES (i, sysdate);
    COMMIT;  -- 잦은 commit
  END LOOP;
END;
/

-- 해결: 커밋 묶기
BEGIN
  FOR i IN 1..100000 LOOP
    INSERT INTO t_log VALUES (i, sysdate);
    IF MOD(i,1000)=0 THEN COMMIT; END IF;
  END LOOP;
  COMMIT;
END;
/
```

### 16. Buffer Cache 경합 `free buffer waits`, `db file parallel write`, `write complete waits`

- **문맥** : 체크포인트 주기가 짧거나(FAST_START_MTTR_TARGET 작게), DBWR 프로세스 부족, I/O 대역폭 협소.
- **증상** : 조회 세션에서 `free buffer waits`, DBWR 백그라운드에서 `db file parallel write`가 비정상적으로 길다.
- **해결** :
    1. `FAST_START_MTTR_TARGET` 값을 완화(예 30 → 120초).
    2. `DB_WRITER_PROCESSES` 를 CPU/8 수준으로 늘려 DBWR 동시 write 실선 확보.
    3. 데이터파일을 Stripe 된 Fast Storage 로 옮겨 write bandwidth 확보.

🧪 DBWR 지연에 따른 `free buffer waits` **예제**

```sql
-- 과도한 Checkpoint 주기
ALTER SYSTEM SET fast_start_mttr_target = 5; -- 아주 짧게!
-- 대량 업데이트
UPDATE emp SET sal = sal*1.01 WHERE 1=1;

-- 해결
ALTER SYSTEM SET fast_start_mttr_target = 120;
```

### 17. Library Cache 경합 `library cache lock` / `library cache: mutex X`

- **문맥** : 동시 접속 수천 세션이 Literal SQL 쏟아내고, 그 와중에 대용량 패키지를 컴파일.
- **원인** :
    1. 각 Literal SQL이 새로운 Child Cursor 생성 → Pin 경합.
    2. DDL(ALTER PACKAGE, CREATE INDEX) 은 Library Cache 객체를 Exclusive로 잠금.
- **해결** :
    - 운영 SQL은 항상 바인드 변수 사용; 불가피할 때 `CURSOR_SHARING=FORCE` 임시 활용.
    - 운영 시간대에 DDL 실행 금지·릴리즈 패키지는 오프 피크 배포.
    - Shared Pool 여유분 확보 (`V$SHARED_POOL_ADVICE` 권고치 반영).

🧪 Library Cache 경합 (Literal SQL & DDL) **예제**

```sql
-- Literal 폭주
BEGIN
  FOR i IN 1..10000 LOOP
    EXECUTE IMMEDIATE
      'SELECT /*literal*/ count(*) FROM emp WHERE empno = '||i;
  END LOOP;
END;
/

-- 해결책 예시
ALTER SYSTEM SET cursor_sharing = FORCE;   -- 임시
-- 또는 코드를 바인드 변수 형태로 수정
```

### 18. 대용량 해시 조인/정렬 중 `direct path read temp`

- **문맥** : 실습 스크립트에서 `SORT_AREA_SIZE`를 100 KB 로 인위적으로 작게 두고 20만 행 정렬.
- **원인** : 작업 집합이 PGA를 꽉 채워 TEMP로 스필.
- **해결** :
    - 실환경은 AUTO 정책 유지 + PGA 타깃을 여유 있게.
    - 정렬 대상이 정말 필요한 컬럼·행인지 WHERE 절·프로젝션을 먼저 다이어트

🧪 `direct path read temp` 폭증 (PGA 부족) **예제**

```sql
ALTER SYSTEM SET pga_aggregate_target = 20M;  -- 고의 축소
-- 대량 정렬
CREATE TABLE t_big AS
SELECT /*+ PARALLEL(4) */
       LEVEL AS n, LPAD('x',4000,'x') AS pad
FROM   dual CONNECT BY LEVEL <= 100000;  -- temp 스필 발생

-- 교정
ALTER SYSTEM SET pga_aggregate_target = 512M;
DROP TABLE t_big PURGE;
```

## ✅ 4장. 조인 튜닝 관련 추가 예외 사항

| 항목 | 발생 가능 오류 | 원인 | 해결 방안 |
| --- | --- | --- | --- |
| **NL 조인 
(Nested Loop Join)** | `ORA-00904: invalid identifier` | 조인 컬럼 오타 또는 스키마 미지정 | `SELECT ... FROM A a JOIN B b ON a.col1 = b.col1` 식으로 fully qualified 사용 권장 |
| **SM 조인 
(Sort Merge Join)** | 성능 저하 또는 소트 발생 | 조인 조건이 정렬 불가능한 경우 
(예: `LIKE`, `!=`) | 조인 컬럼에 인덱스 존재 여부 확인 및 힌트 사용(`USE_MERGE`) |
| **해시 조인 오류** | `ORA-12801: error signaled in parallel query server` | 데이터량 초과로 TEMP 공간 부족 | TEMP TABLESPACE 크기 증가 또는 병렬 조정 (`PARALLEL` 수준 낮추기) |
| **서브쿼리 조인 비효율** | Cost 증가 | 인라인 뷰로 매번 서브쿼리 평가됨 | 서브쿼리를 뷰로 변환하거나 `WITH` 절 사용, `MATERIALIZE` 힌트 활용 |

## ✅ 5장. 소트 튜닝 관련 추가 예외 사항

| 항목 | 발생 가능 오류 | 원인 | 해결 방안 |
| --- | --- | --- | --- |

## ✅ 6장. 병렬 처리 및 Direct Path 관련 추가 예외 사항

| 항목 | 발생 가능 오류 | 원인 | 해결 방안 |
| --- | --- | --- | --- |
| **병렬 DML 실패** | `ORA-12838: cannot read/modify an object after modifying it in parallel` | 병렬 처리 후 같은 트랜잭션 내에 수정 시도 | 병렬 DML 수행 후 COMMIT 또는 트랜잭션 분리 필요 |
| **Direct Path Insert 실패** | `ORA-00060: deadlock detected` | Append 힌트와 동시에 인덱스 존재 시 경합 | 인덱스 제거 후 Insert 수행 또는 NOLOGGING + APPEND 전략 |
| **병렬 작업 불가** | `ORA-12827: insufficient parallel query slaves available` | 병렬 프로세스 자원 부족 | `PARALLEL_MAX_SERVERS`, `PARALLEL_MIN_SERVERS` 파라미터 확인 |
| **Partition 관련 오류** | `ORA-14001: partition bound must be specified` | 파티션 정의 누락 또는 부정확 | 반드시 `VALUES LESS THAN (...)` 구문과 함께 컬럼 및 범위 정확히 지정 |
