### AWR Report를 해석할 때 *Load Profile*과 *Top Wait Event*를 반드시 동시에 확인해야 합니다.

- “Wait Event와 Load Profile의 결합 분석” 절에서, 두 지표를 **함께 보아야 정확한 병목 원인에 접근할 수 있다**.

- **CPU 사용률 · Wait 비율 · Load Profile을 결합해 분석해야 합니다.**

### 튜닝 액션 플랜

| 구분 | 세부 조치 | 목적 |
| --- | --- | --- |
| SQL | **ORDER_ITEMS.SUPPLIER_ID** 복합 B-tree Index 생성, 힌트 제거 | Random I/O → Index Access |
| 옵티마이저 | 테이블 통계 재수집, **cursor_sharing = force** | Hard Parse·계획 흔들림 방지 |
| 메모리 | **PGA_AGGREGATE_TARGET** 2 GB→4 GB 상향 | 한 패스·멀티패스 Sort 제거 |
| 트랜잭션 | Batch App Commit 주기 100→1 000 행 | Redo / log file sync 감축 |
| 모니터링 | 15 분 간격 AWR 자동 스냅 + ADDM 검토 | 사후 검증 및 회귀 감시 |

[Buffer_Cache.txt](Buffer_Cache.txt)

```sql
/* BUFFER CACHE ADVICE 정보 확인 */
SELECT * FROM V$DB_CACHE_ADVICE;

/* SGA내의 COMPONENT별 메모리 크기 조회 */
SELECT * FROM V$SGAINFO;

/* V$BH 조회. SYSDBA에서 수행 할 것 */
SELECT * FROM V$BH;

/* BUFFER CACHE내에 올라가 있는 Object 들의 정보와 크기 그리고 block 상태 조회
   SYSDBA 에서 수행 할 것 */
SELECT o.object_name, bh.status, COUNT(*) number_of_blocks, COUNT(*)*8000/1024168 SIZE_MB
FROM DBA_OBJECTS o, V$BH bh
WHERE o.data_object_id = bh.OBJD
AND o.owner != 'SYS'
GROUP BY o.object_Name, bh.status
ORDER BY 3 DESC;

/* Buffer Cache내 올라가 있는 Object 조회를 위한 테스트. SOE_10G에서 수행할 것 */

SELECT /*+ USE_NL(A B) */ A.SALES_REP_ID, COUNT(A.ORDER_TOTAL), SUM(B.QUANTITY)
FROM ORDERS A, ORDER_ITEMS B WHERE A.ORDER_ID = B.ORDER_ID 
AND A.SALES_REP_ID = 400 GROUP BY SALES_REP_ID;

SELECT tt.ORDER_TOTAL,
          tt.SALES_REP_ID,
          tt.ORDER_DATE,
          customers.CUST_FIRST_NAME,
          customers.CUST_LAST_NAME
        FROM
          (SELECT orders.ORDER_TOTAL,
            orders.SALES_REP_ID,
            orders.ORDER_DATE,
            orders.customer_id,
            rank() Over (Order By orders.ORDER_TOTAL DESC) sal_rank
          FROM orders
          WHERE orders.SALES_REP_ID = 500
          ) tt,
          customers
        WHERE tt.sal_rank        <= 100
        AND customers.customer_id = tt.customer_id;

SELECT /*+ FULL(A) */ count(*) CNT FROM ORDER_ITEMS A 
WHERE PRODUCT_ID = 100 AND UNIT_PRICE > 1000;

/* 아래는 SYSDBA 에서 수행 할것. */
ALTER SYSTEM FLUSH BUFFER_CACHE;
```

## 2. AWR 스냅샷 생성 & 부하 구간 지정

```sql
-- (1) 튜닝 전 스냅
EXEC DBMS_WORKLOAD_REPOSITORY.create_snapshot;

-- 10~15 분 동안 실제 업무 부하 실행 또는 재현

-- (2) 튜닝 후 스냅
EXEC DBMS_WORKLOAD_REPOSITORY.create_snapshot;
```

> TIP : 스냅 간격은 15 분이 가장 다루기 쉽습니다. 장기 분석용이라면 60 분 주기로 AUTO SNAPSHOT(기본값)만으로도 충분합니다.

## 7. 결과 해석 예시

아래는 샘플 리포트(303→304 vs 307→308)를 diff 모드로 봤을 때의 핵심 포인트입니다.

| 항목 | 튜닝 전 | 튜닝 후 | 변화 |
| --- | --- | --- | --- |
| DB Time | 93 s | 14 s | **-85 %** |
| CPU/DB Time 비중 | 65 % | 82 % | CPU 집중도로 개선 |
| Top Wait | `db file sequential read` | 미미 | 랜덤 I/O 병목 해소 |
| Top SQL | `ORDER_ITEMS` Full Scan | 인덱스 Scan | 실행 시간 12 → 0.8 s |

> 분석 팁 : Load Profile + Top 5 Timed Event를 함께 봐야 “부하 규모”와 “병목 성격”을 동시에 파악할 수 있습니다.
> 

### 결과

| 지표 | 튜닝 전 → 후 |
| --- | --- |
| 사용자 체감 응답 | 1 .8 s → 0 .7 s (NewOrder) |
| Top SQL Elapsed | 380 s → 27 s |
| I/O Wait 비중 | 32 % → 6 % |
| CPU Util | 90 % (Usr 63 %) → 55 % |

## 8. 자동화 예시 (리포트 + 스냅 + 분석 한방에)

```bash
#!/bin/bash
sqlplus / as sysdba <<EOF
ALTER SESSION SET nls_date_format='yyyy-mm-dd hh24:mi:ss';

EXEC dbms_workload_repository.create_snapshot;
-- 10분 대기 (업무 부하)
EXEC dbms_lock.sleep(600);

EXEC dbms_workload_repository.create_snapshot;

-- HTML 리포트
DEFINE inst_num=1
DEFINE begin_snap=&1   -- 첫 스냅 번호 자동 치환
DEFINE end_snap=&2     -- 두 번째 스냅 번호 자동 치환
@?/rdbms/admin/awrrpt.sql

-- Diff
@?/rdbms/admin/awrddrpt.sql

-- Top SQL 추출
@dba_hist_script '20250724' '20250725'
EXIT
EOF
```

`create_noncdb.sh` 또는 VM-init 스크립트 이후에 붙여서 실행하면 **신규 서버->부하 유발->AWR 분석**까지 자동화됩니다.

### 마무리 : AWR 예제 체크리스트

1. **SYSAUX 공간** 확보 후 스냅샷 성공 여부 확인.
2. **스냅 간격**은 문제 원인에 맞춰 15 분 내외로 조정.
3. **awrrpt.sql + awrddrpt.sql** 두 종류 모두 생성해 변동 폭을 수치로 제시.
4. **DBA_HIST 스크립트**로 리포트 외 특정 지표만 빠르게 확인.
5. AWR 결과-> Buffer Cache·PGA·SQL 계획 등 튜닝 액션을 연결하고, **재측정**으로 효과를 검증.

### AWR Report - “좋은 도출 결과 얻기 위한” 팁

| 단계 | Best Practice |
| --- | --- |
| 스냅샷 전략 | 부하 변동 구간마다 최소 15 분 간격 스냅샷. 장기 분석용으론 1 시간 간격. |
| Report 옵션 | `awrrpt.sql` → HTML, 또 `awrddrpt.sql`로 **Diff Report**까지 생성 |
| 해석 순서 | ① Report Summary → ② Load Profile → ③ Top 5 Timed Events → ④ SQL Stats. OS Stat과 교차 확인 |
| 자동 진단 | ADDM 결과로 튜닝 우선순위 참고 |
| 보존 정책 | 중요한 Change 전후 스냅샷은 **DBMS_WORKLOAD_REPOSITORY.modify_snapshot_settings**로 보존기간 연장 |

이처럼 **수치-근거 기반 AWR 분석 → 정밀 SQL/인덱스/메모리 튜닝 → 재측정**의 사이클을 돌리면, 실 서비스에서도 확실한 성능 개선 효과를 얻을 수 있습니다.

### 사례에서 배울 점

1. **Load Profile + Top Wait Event** 두 시각을 결합하면 “양(Load) vs. 병목(Resource)”를 동시에 볼 수 있다.
2. **Top SQL→실행계획→객체 통계** 로 이어지는 drill-down이 가장 빠른 문제해결 루트.
3. AWR 리포트는 **“비교”**가 핵심이다.
    - 같은 요일·시간대 스냅샷 두 개를 만든 후 **Report Diff** 모드로 보면 변동치를 바로 확인할 수 있다.
4. 튜닝 후 반드시 **후속 스냅샷**을 찍어 개선 효과를 수치로 검증해야 한다.

# AWR 리포트 한 방에 끝내는 실습 가이드

### 1. 실습 목표

> “데이터베이스 성능을 사진 찍듯 기록(AWR 스냅샷) → 전·후 비교 보고서(Diff) → 병목을 찾아 튜닝 → 효과를 숫자로 검증” 하는 전 과정을 한 번에 체험합니다.
> 

---

### 2. 환경 준비

1. **SYSAUX 테이블스페이스 확장** — AWR 데이터가 저장될 공간입니다.
    
    ```sql
    /* 최소 1 GB 권장 */
    ALTER DATABASE DATAFILE '.../sysaux01.dbf' RESIZE 1G;
    -- 또는 AUTOEXTEND ON
    ```
    
    SYSAUX가 좁으면 스냅샷이 실패하므로 먼저 용량을 늘려야 합니다.
    
2. **접속 권한**
    - `sysdba` 권한으로 SQL*Plus / SQL Developer에 로그인하십시오.

---

### 3. 부하 재현 & 스냅샷 찍기

| 단계 | 명령 | 설명 |
| --- | --- | --- |
| 튜닝 전 스냅 | `EXEC DBMS_WORKLOAD_REPOSITORY.create_snapshot;` | 첫 사진 |

> 10~15 분 동안 애플리케이션을 사용하거나 테스트 스크립트로 일부러 부하를 줍니다.
> 

| 튜닝 후 스냅 | `EXEC DBMS_WORKLOAD_REPOSITORY.create_snapshot;` | 두 번째 사진 |

스냅 간격은 `15 분`이 다루기 쉽습니다. 장기 분석은 1 시간 간격의 자동 스냅샷으로도 충분합니다.

### 6. 리포트 읽는 법 (가장 중요한 두 칸만 먼저 본다)

1. **Load Profile** : 초당 논리 I/O, 물리 I/O, 트랜잭션 수 등 ‘**양(Load)**’을 보여 줍니다.
2. **Top 5 Timed Events** : CPU·I/O 대기가 차지한 시간 비율로 ‘**어디서 기다리나(병목)**’를 알려 줍니다.

두 지표를 **같이** 봐야 Random I/O인지 Scan I/O인지, CPU인지 확실히 판단할 수 있습니다.오라클강의_PDF_교재_v1_0, 오라클강의_PDF_교재_v1_0

> 예) CPU 20 % + db file sequential read 30 %이면 랜덤 I/O 병목이 의심됩니다.
