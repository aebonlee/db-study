# 8교시 - AWR 리포트를 활용한 SQL 튜닝 효과 분석 및 병목 진단 실습

### 🎯 학습 목표

- AWR(Auto Workload Repository)의 구성과 분석 항목을 이해하고, 수집 시기 및 방법을 설명할 수 있다.
- SQL 튜닝 전후의 성능 차이를 AWR 리포트 기반으로 정량적으로 비교할 수 있다.
- 주요 Wait Event, Top SQL, CPU/Elapsed Time 등의 항목을 해석하고 병목 원인을 진단할 수 있다.
- 실습을 통해 AWR 리포트 내 SQL ID, 실행 횟수, Buffer Gets 등의 항목을 직접 분석한다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “SQL 튜닝의 궁극적인 목적은 성능 개선입니다. 그런데, 그 성능이 개선되었는지 아닌지를 **어떻게 증명할 수 있을까요?**
    
    단순히 속도가 빨라진 것 같다고 말하는 것이 아니라, 수치로 보여주고, 리포트로 정리할 수 있어야 합니다.
    
    오늘은 Oracle에서 제공하는 가장 강력한 진단 도구인 **AWR 리포트**를 통해 SQL 튜닝의 결과를 확인하는 방법을 배웁니다.”
    

---

1. **[AWR 리포트 개요 및 생성 방식 – 10분]**
    - AWR(Auto Workload Repository)란?
        - Oracle 내부의 성능 데이터를 주기적으로 스냅샷(Snapshot) 형태로 저장
        - 시스템 전반의 Wait Event, Top SQL, CPU 사용률, I/O 발생 등 다양한 항목 포함
    - AWR 보고서 생성 방법 (DBA 권한 필요)
        
        ```sql
        
        -- 스냅샷 생성
        EXEC DBMS_WORKLOAD_REPOSITORY.CREATE_SNAPSHOT();
        
        -- 보고서 생성 (SQL*Plus)
        @$ORACLE_HOME/rdbms/admin/awrrpt.sql
        ```
        
    - 생성 시점: 튜닝 전/후로 Snapshot을 구분하여 비교 분석
    - Oracle XE에서는 AWR 대신 Statspack 또는 `V$` 뷰 실습 가능

---

1. **[AWR 주요 항목 설명 – 10분]**
    
    
    | 항목 | 설명 |
    | --- | --- |
    | **Top 5 Timed Events** | 대기 시간이 높은 주요 Wait Event |
    | **SQL ordered by Elapsed Time** | 총 수행 시간이 긴 SQL |
    | **SQL ordered by Buffer Gets** | 메모리 접근량이 높은 SQL |
    | **SQL ordered by CPU Time** | CPU 점유가 높은 SQL |
    | **Instance Activity Stats** | Parse 횟수, Hard Parse, Soft Parse, Executions 등 |
    | **Segment Statistics** | I/O가 많이 발생한 테이블 및 인덱스 객체 |
    
    ➤ 분석 목적: 병목 지점 파악, 자원 낭비 원인 규명, 튜닝 전후 성능 개선 검증
    

---

1. **[실습 – SQL 튜닝 전/후 AWR 수치 비교] – 15분**
    
    🔸 실습 1: 튜닝 전 Snapshot 생성
    
    ```sql
    
    EXEC DBMS_WORKLOAD_REPOSITORY.CREATE_SNAPSHOT();
    ```
    
    🔸 성능이슈 SQL 실행 (예: Full Scan SQL)
    
    ```sql
    
    SELECT * FROM SALES WHERE TO_CHAR(SALE_DATE, 'YYYY') = '2023';
    ```
    
    🔸 튜닝 SQL 적용 (인덱스 유도 구조)
    
    ```sql
    
    SELECT * FROM SALES WHERE SALE_DATE BETWEEN TO_DATE('2023-01-01','YYYY-MM-DD') AND TO_DATE('2023-12-31','YYYY-MM-DD');
    ```
    
    🔸 실습 2: 튜닝 후 Snapshot 생성 → AWR 보고서 생성 및 분석
    
    ```sql
    
    EXEC DBMS_WORKLOAD_REPOSITORY.CREATE_SNAPSHOT();
    -- awrrpt.sql 또는 V$SQL에서 수치 비교
    ```
    
    ➤ SQL ID 확인 후 `V$SQL` 또는 `DBA_HIST_SQLSTAT` 통해 실행 횟수, 버퍼 접근량, 물리 I/O 등 비교
    

---

1. **[SQL 튜닝 효과 수치로 정리하기 – 5분]**
    
    
    | 지표 | 튜닝 전 | 튜닝 후 | 개선률 |
    | --- | --- | --- | --- |
    | Elapsed Time | 13.5s | 0.8s | 94% 감소 |
    | Buffer Gets | 120,000 | 8,000 | 93% 감소 |
    | CPU Time | 6.8s | 0.6s | 91% 감소 |
    - 이러한 수치를 AWR 보고서 또는 `V$SQL` 통계 기반으로 문서화하면 성능 개선 효과 입증 가능

---

1. **[마무리 정리 – 5분]**
    - 튜닝은 ‘감’이 아니라 ‘수치’로 증명되어야 한다
    - AWR은 성능 문제의 원인과 결과를 동시에 보여주는 보고서이며,
        
        → 진단 + 개선 + 결과보고까지 전 과정에서 활용된다
        
    - “SQL 튜닝은 기술이 아니라 분석이다.
        
        AWR은 그 분석을 수치로 완성하는 증거자료다.”

### 📝 과제

1. 튜닝 대상 SQL을 하나 선택하고, `DBMS_XPLAN.DISPLAY` 및 `V$SQL`로 실행 전후 COST 비교
2. 실행계획상의 예상 Row 수, Buffer Gets, Elapsed Time을 비교하여 개선율 산정
3. AWR(또는 Statspack) Report를 활용해 ‘Top SQL’ 영역에서 튜닝 전후 SQL이 차지한 비중 비교
