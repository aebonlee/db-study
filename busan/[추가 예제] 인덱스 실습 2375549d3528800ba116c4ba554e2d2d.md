# [추가 예제] 인덱스 실습

**데이터 양이 적은 상태에서의 인덱스 실습**은 대부분 `TABLE ACCESS FULL`로 처리되기 때문에

**인덱스 튜닝 효과를 체감하기 어렵습니다.**

그래서 이번에 실습용 데이터를 **양적으로 늘리고**,

**인덱스를 사용하는 쿼리와 사용하지 않는 쿼리의 차이를 분명히 체감할 수 있도록 해보겠습니다.**

## ✅ 인덱스 실습용 예제

### 🎯 목표

- `emp_large` 테이블에 **1만 건 이상의 데이터를 삽입**
- `job` 컬럼 기준으로 **단일 인덱스와 복합 인덱스 비교**
- `deptno + job` 복합 인덱스 실습
- `AUTOTRACE` / `EXPLAIN PLAN`을 통해 **명확한 I/O 차이 확인**

---

## 🧱 1. 실습 테이블 생성

```sql
CREATE TABLE emp_large (
  empno NUMBER,
  ename VARCHAR2(20),
  job VARCHAR2(10),
  sal NUMBER,
  deptno NUMBER
);
```

---

## 📥 2. 대용량 데이터 삽입 (10,000건)

```sql
BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO emp_large
    VALUES (
      i,
      'EMP_' || TO_CHAR(i),
      CASE MOD(i, 4)
        WHEN 0 THEN 'CLERK'
        WHEN 1 THEN 'MANAGER'
        WHEN 2 THEN 'SALESMAN'
        ELSE 'ANALYST'
      END,
      MOD(i, 5000) + 1000,
      MOD(i, 4)*10 + 10
    );
  END LOOP;
  COMMIT;
END;
/
```

---

## 🧪 3. 인덱스 실습 시나리오

### ✅ 3-1. 인덱스 없이 조회

```sql

SET AUTOTRACE ON
SELECT * FROM emp_large WHERE job = 'MANAGER';
```

→ `TABLE ACCESS FULL` 이 뜨고, 블록 수 많음

---

### ✅ 3-2. 단일 인덱스 생성

```sql
CREATE INDEX idx_emp_large_job ON emp_large(job);7
```

다시 쿼리 실행:

```sql
SELECT * FROM emp_large WHERE job = 'MANAGER';
```

→ `INDEX RANGE SCAN` 이 뜨고, 블록 수 크게 감소

---

### ✅ 3-3. 복합 인덱스 비교

```sql
CREATE INDEX idx_emp_large_dept_job ON emp_large(deptno, job);
```

비교 쿼리:

```sql
SELECT * FROM emp_large WHERE deptno = 10 AND job = 'CLERK';
```

→ `INDEX RANGE SCAN` 또는 `SKIP SCAN` 확인

---

### ✅ 3-4. 통계 비교용 쿼리

```sql
SELECT job, COUNT(*) FROM emp_large GROUP BY job;
SELECT deptno, COUNT(*) FROM emp_large GROUP BY deptno;
```

→ 각 컬럼의 selectivity (선택도)를 확인

---

## 📌 4. 관찰 포인트

| 항목 | 인덱스 없음 | 인덱스 있음 |
| --- | --- | --- |
| 실행계획 | TABLE ACCESS FULL | INDEX RANGE SCAN |
| 논리적 읽기 | 수천 블록 이상 | 수십 블록 이하 |
| 물리적 읽기 | 증가 가능 | 캐시에서 해결 |
| 속도 체감 | 눈에 띄게 느려짐 | 확실히 빨라짐 |

> 💬 **Aebon’s Tip**
> 

“이제 보시는 것처럼, 인덱스 없을 땐 Full Scan으로 전 테이블을 읽지만, 인덱스를 만들고 나면 필요한 부분만 조회합니다. 블록 수와 실행시간 차이를 직접 체감해 보세요.”

## 🧩 추가 심화 과제 (선택)

- `LIKE`, `BETWEEN`, `IN` 조건으로 인덱스 활용 유무 비교
- 복합 인덱스 선두 컬럼을 빠뜨렸을 때 인덱스 미사용 확인

```sql
-- 복합 인덱스 존재 시
SELECT * FROM emp_large WHERE job = 'CLERK';  -- 선두컬럼 없으므로 인덱스 미사용
```

## ✅ 마무리 요약

- **데이터 양이 늘어나야 튜닝의 효과를 정확히 체감할 수 있다**
- 실습용 `emp_large` 테이블로 인덱스의 성능 개선 효과를 **수치와 실행계획**으로 분석
- **블록 읽기 차이, COST, ROWS, ACCESS PATH 변화** 중심으로 설명

```sql
-- 1. 실습용 테이블 생성
CREATE TABLE emp_large (
  empno NUMBER,
  ename VARCHAR2(20),
  job VARCHAR2(10),
  sal NUMBER,
  deptno NUMBER
);

-- 2. 대량 데이터 삽입 (1만 건)
BEGIN
  FOR i IN 1..10000 LOOP
    INSERT INTO emp_large
    VALUES (
      i,
      'EMP_' || TO_CHAR(i),
      CASE MOD(i, 4)
        WHEN 0 THEN 'CLERK'
        WHEN 1 THEN 'MANAGER'
        WHEN 2 THEN 'SALESMAN'
        ELSE 'ANALYST'
      END,
      MOD(i, 5000) + 1000,
      MOD(i, 4)*10 + 10
    );
  END LOOP;
  COMMIT;
END;
/

-- 3. 인덱스 없이 조회
SET AUTOTRACE ON
SELECT * FROM emp_large WHERE job = 'MANAGER';

-- 4. 단일 인덱스 생성 후 실행
CREATE INDEX idx_emp_large_job ON emp_large(job);
SELECT * FROM emp_large WHERE job = 'MANAGER';

-- 5. 복합 인덱스 생성 후 실행
CREATE INDEX idx_emp_large_dept_job ON emp_large(deptno, job);
SELECT * FROM emp_large WHERE deptno = 10 AND job = 'CLERK';

-- 6. 선택도 분석
SELECT job, COUNT(*) FROM emp_large GROUP BY job;
SELECT deptno, COUNT(*) FROM emp_large GROUP BY deptno;

-- 7. 인덱스 미사용 예시 (선두 컬럼 빠짐)
SELECT * FROM emp_large WHERE job = 'CLERK';
```
