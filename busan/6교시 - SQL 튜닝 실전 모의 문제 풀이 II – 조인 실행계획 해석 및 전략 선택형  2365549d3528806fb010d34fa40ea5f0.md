### 2. [문제 풀이 및 해설 – 40분]

### ✅ 문제 1. 조인 방식 판단 (객관식)

**문제:** 다음 SQL에서 `EMP` 테이블이 10만 건, `DEPT` 테이블이 5건일 경우, 옵티마이저가 기본적으로 선택할 조인 방식은?

```sql
SELECT E.ENAME, D.DNAME
FROM EMP E JOIN DEPT D ON E.DEPTNO = D.DEPTNO;
```

A. Nested Loop Join

B. Hash Join

C. Merge Join

D. Star Transformation

**정답:** A

**해설:** 조인 대상이 적은 쪽이 Driving Table이 되며, 상대 테이블이 크면 옵티마이저는 NLJ를 우선 고려함.

---

### ✅ 문제 2. 실행계획 판단

**문제:** 다음 실행계획에서 병목 발생 가능성이 가장 높은 연산 단계는?

```
Id | Operation                   | Rows | Cost
---|-----------------------------|------|------
0  | SELECT STATEMENT            |      | 540
1  | NESTED LOOPS                |      | 540
2  |   TABLE ACCESS FULL EMP     | 100K | 220
3  |   TABLE ACCESS FULL DEPT    | 100K | 320
```

**답변:** Id 3 – DEPT Full Scan

**해설:** 100K × 100K Join → Cartesian Product 또는 잘못된 Join 조건/순서

---

### ✅ 문제 3. 힌트 적용 후 예상 실행계획 변화

**문제:** 다음 SQL에 `/*+ USE_HASH(B) */` 힌트를 적용했을 때, 예상되는 실행계획 변화는?

```sql
SELECT * FROM A JOIN B ON A.KEY = B.KEY;
```

**정답:**

- 옵티마이저는 B를 **Build 입력**으로 보고 Hash Join 구조를 채택
- B가 큰 경우 적절치 않음
- Full Scan + Hash Join 구조 발생 가능성 ↑

---

### ✅ 문제 4. 통계 변경 시 옵티마이저 판단 변화

**문제:** 옵티마이저가 통계 수집 후 **실행계획을 바꾸는 이유**로 옳지 않은 것은?

A. NDV가 줄어들어 Selectivity가 높아졌다

B. 행 수가 줄어들어 Join 순서가 바뀌었다

C. 히스토그램이 삭제되었다

D. CURSOR_SHARING 파라미터가 FORCE로 설정되었다

**정답:** D

**해설:** CURSOR_SHARING은 파싱/공유에 영향, 실행계획 결정은 통계 기반

→ A/B/C는 옵티마이저 결정 요소

---

### ✅ 문제 5. 실행계획 수치 비교

**문제:** 다음 두 SQL 실행계획 중 더 효율적인 것은?

| SQL A | SQL B |
| --- | --- |
| Cost: 180, Rows: 10K | Cost: 90, Rows: 200K |

**정답:**

**해석 필요** – A는 고비용이지만 소량 데이터, B는 저비용이지만 다량 처리

→ **목표가 소량 고속 처리**면 A, **전체 집계 처리**면 B

※ 상황 기반 판단이 필요

### 3. [정리 및 마무리 – 5분]

- 조인은 튜닝에서 가장 많은 병목이 발생하는 영역 → 순서, 방식, 조건 판단이 핵심
- 실행계획은 단지 구조가 아니라 **수치를 통해 병목을 드러내는 지도**
- “좋은 튜닝이란, 실행계획을 보고 ‘왜 이렇게 나왔는지’를 말할 수 있는 것”
