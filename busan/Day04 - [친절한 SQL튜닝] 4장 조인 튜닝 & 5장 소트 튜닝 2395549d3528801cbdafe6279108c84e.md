## 0. 실습용 기본 테이블·데이터 준비

```sql
-- 학생(Students) 테이블
CREATE TABLE students (
  student_id  NUMBER PRIMARY KEY,
  name        VARCHAR2(50)
);

-- 성적(Scores) 테이블
CREATE TABLE scores (
  score_id    NUMBER PRIMARY KEY,
  student_id  NUMBER,
  score       NUMBER
);

-- 예시 데이터
INSERT INTO students VALUES (1, '영희');
INSERT INTO students VALUES (2, '철수');
INSERT INTO students VALUES (3, '민수');

INSERT INTO scores VALUES (101, 1, 95);
INSERT INTO scores VALUES (102, 2, 88);
INSERT INTO scores VALUES (103, 3, 72);
COMMIT;

-- 조인 효율을 높이기 위한 인덱스
CREATE INDEX idx_scores_student_id ON scores(student_id);
```

# 4장. 조인 튜닝

### 4.1 NL 조인 (Nested Loop Join)

- 마치 교실에 있는 학생 명단을 한 명씩 보며 각각의 학생이 다른 반 학생 중 누구와 친구인지 확인하는 방식.
- 찾는 사람이 적고, 반이 작으면 빠르지만, 찾을 인원이 많고 반이 크면 느려짐
- **튜닝 포인트** : “먼저 작은 반부터 살펴보자” 라는 전략을 세우면 빠르게 조인 할 수 있음.

NL 조인(Nested Loop Join)은 조인을 수행할 때 하나의 테이블에서 데이터를 하나씩 읽어오고, 그에 대해 다른 테이블에서 조건에 맞는 데이터를 찾는 방식입니다.

예를 들어, 학생 명단이 적힌 A표와 성적이 적힌 B표가 있을 때, A표의 학생 한 명을 선택하고, B표를 처음부터 끝까지 훑어보며 그 학생의 성적을 찾는 과정을 반복합니다.

이러한 방식은 적은 데이터에는 효율적이지만, 두 테이블 모두 데이터가 많을 경우 시간이 오래 걸릴 수 있습니다.

### **📌** 개념 설명

> 두 개의 표에서 하나씩 꺼내서 서로 비교해보는 방식입니다. 
예를 들어 A반과 B반 친구들 명단이 있을 때, A반의 한 명을 기준으로 B반 전부를 돌아보며 친구를 찾는 방식입니다.
> 

### **📌** NL 조인 수행 과정 분석

NL 조인은 아래와 같은 순서로 동작합니다.

1. 외부 테이블(A)에서 한 건의 데이터를 선택
2. 내부 테이블(B)에서 조인 조건에 맞는 데이터를 탐색
3. 조건에 맞는 데이터를 결과로 출력
4. 외부 테이블의 다음 데이터를 선택하고 위 과정을 반복

즉, "밖의 테이블 하나 → 안쪽 전체 검사 → 다시 밖의 다음 하나 → 다시 안쪽 전체" 이런 반복 구조입니다.

### **📌** NL 조인 튜닝 포인트

- 외부 테이블은 데이터 양이 적은 쪽으로 선택해야 합니다.
- 내부 테이블은 인덱스가 걸려 있어 빠르게 찾을 수 있어야 합니다.
- 바인드 변수를 사용하여 하드 파싱을 피하는 것이 중요합니다.

### **📌** NL 조인 특징 요약

- **장점**: 소량 데이터 조인에 유리하며, 인덱스를 활용할 수 있음
- **단점**: 반복적인 탐색으로 인해 데이터 양이 많아지면 성능 저하 발생
- **적합한 경우**: 외부 테이블이 작고 내부 테이블에 인덱스가 있을 때

### **📌** NL 조인 확장 메커니즘

Oracle은 **Nested Loop Join with Batching**과 같은 개선된 NL 조인 방식도 제공합니다.

이는 한 번에 여러 건의 RowID를 미리 가져와서 내부 테이블에서 묶음(batch)으로 처리하는 방식으로, I/O를 줄여 성능을 향상시킬 수 있습니다.

| 소목차 | 핵심 정리 |
| --- | --- |
| **4.1.1 기본 메커니즘** | 한 행씩 꺼내어 안쪽 테이블을 반복 탐색. “외부 1명 → 내부 전체”를 계속 반복합니다. |
| **4.1.2 실행계획 제어** | `USE_NL( )` 힌트를 지정하면 옵티마이저가 NL 조인을 선택합니다. |
| **4.1.3 수행 과정** | 외부 테이블에서 N건을 읽을 때마다 내부 테이블에 대한 인덱스 탐색이 N번 일어납니다. |
| **4.1.4 튜닝 포인트** | (1) 외부 테이블을 작은 쪽으로, (2) 내부 테이블에 인덱스를 갖추기, (3) 바인드 변수 사용. |
| **4.1.5 특징 요약** | 소량 데이터에는 빠르나 대량 데이터에는 느립니다. |
| **4.1.6 실습** | 아래 예제를 실행해 보고 `EXPLAIN PLAN`으로 NL 조인을 확인합니다. |
| **4.1.7 확장 메커니즘** | “배치 NL”처럼 여러 건을 한꺼번에 내부 테이블에서 찾아 I/O를 줄이는 기법도 존재합니다. |

```sql
/* NL 조인 실습 */
EXPLAIN PLAN FOR
SELECT /*+ USE_NL(s sc) */ s.name, sc.score
FROM   students s
JOIN   scores   sc
ON     s.student_id = sc.student_id;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### 4.2 소트 머지 조인 (Sort Merge Join)

- 양쪽 명단이 가나다순으로 미리 정리해두고, 동시에 비교를 해가면서 짝을 찾는 방식입니다.
- 특징 : 이름이 순서대로 정리된 명단이 있다면, 아주 많은 데이터에서 사람을 찾더라도 빠르게 비교하며 찾을 수 있습니다.
- **SGA vs PGA : 데이터를 일시 저장하는 공간 (SGA는 공동책상, PGA는 개인책상)이 어느 쪽에서 쓰이냐가 중요합니다.**

### **📌** 개념 설명

> 두 명단을 미리 가나다순으로 정리해두고, 위에서부터 동시에 비교하면서 짝을 찾는 방식이에요. 정렬이 되어 있어서 비교가 아주 빠르게 됩니다.
> 

### **📌 빠르게 이해하는 한 줄 정리**

> **소트 머지 조인**은 “먼저 정렬, 그 다음 나란히 걸으며 비교” 방식 입니다.
데이터가 많고 인덱스가 없을 때 특히 강력하며, 메모리(PGA)를 충분히 주면 속도가 더욱 좋아집니다.
> 

| 소목차 | 핵심 정리 |
| --- | --- |
| **4.2.1 SGA vs PGA** | 정렬에 필요한 임시 공간이 공유 메모리(SGA)인지, 세션 전용 메모리(PGA)인지가 **속도에 영향**.

소트 머지 조인은 두 테이블을 **정렬**한 뒤 병합합니다. 이때 정렬용 임시 공간은 세션 전용 메모리(**PGA**)를 먼저 쓰고, 부족하면 디스크(Temp Tablespace)를 이용합니다. 공유 메모리(**SGA**)는 주로 데이터를 읽어 들일 때만 사용합니다. |
| **4.2.2 기본 메커니즘** | 두 테이블을 각각 **정렬**한 뒤, 동시에 앞에서부터 맞대어 가며 짝을 찾습니다.

① 두 테이블을 모두 조인 열 기준으로 오름차순 정렬 → ② 양쪽을 **왼쪽에서 오른쪽으로 한 칸씩** 움직이며 같은 값을 만나면 결과에 추가 → ③ 끝까지 도달할 때까지 반복합니다.정렬된 전화번호부 두 권을 나란히 놓고, 한 장씩 넘기며 같은 이름을 찾는 모습과 유사합니다. |
| **4.2.3 빠른 이유** | 미리 정렬되어 있으면 “한 번씩만 앞으로” 이동하므로 큰 테이블도 선형 시간에 처리됩니다.

이미 정렬된 데이터끼리는 다시 뒤로 돌아갈 필요가 없습니다. 따라서 대량 데이터라도 **선형(한 번씩만)** 스캔하면 되므로 효율적입니다. |
| **4.2.4 주용도** | 대량 집합 간 조인 또는 이미 정렬 인덱스를 가진 열끼리 조인할 때 유리합니다.
• 두 테이블 모두 **대량**이고 조인 열에 인덱스가 없거나 범위가 넓어 인덱스 이점이 적을 때 혹은 양쪽이 **이미 정렬**(예: 파티션 키)되어 있는 경우 |
| **4.2.5 제어** | `USE_MERGE( )` 힌트로 소트 머지 조인을 지정할 수 있습니다.
옵티마이저 힌트 `USE_MERGE(A B)` 또는 `MERGE_JOIN`을 주면 소트 머지 조인을 강제할 수 있습니다. |
| **4.2.6 특징 요약** | 정렬 비용이 크지만, 한 번 정렬 후에는 풀스캔식 조인보다 훨씬 빠름.
• 정렬 비용이 선행되지만, 한 번 정렬 후에는 빠르게 병합메모리가 넉넉하면 Temp I/O가 줄어듦
• 정렬을 건너뛸 방법(인덱스 정렬)이나 메모리를 충분히 확보하는 것이 핵심 |

```sql
/* 소트 머지 조인 실습 */
EXPLAIN PLAN FOR
SELECT /*+ USE_MERGE(s sc) */ s.name, sc.score
FROM   students s
JOIN   scores   sc
ON     s.student_id = sc.student_id
ORDER  BY s.student_id;   -- 정렬 발생 확인

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

```sql
/* 1) 옵티마이저에게 소트 머지 조인을 사용하라고 지시 */
EXPLAIN PLAN FOR
SELECT /*+ USE_MERGE(s sc) */ 
       s.name,
       sc.score
FROM   students s
JOIN   scores   sc
ON     s.student_id = sc.student_id
ORDER  BY s.student_id;      -- 정렬이 실제로 수행되는지 확인

/* 2) 실행계획 출력 */
SELECT * 
FROM   TABLE(DBMS_XPLAN.DISPLAY);
```

### 예제 해설

1. **힌트** `/*+ USE_MERGE(s sc) */` 가 두 테이블을 소트 머지 조인으로 묶습니다.
2. `ORDER BY` 절은 결과가 정렬되어 있는지 확인하기 위한 단순 추가이며, 조인 자체에도 정렬 과정이 포함됩니다.
3. `DBMS_XPLAN.DISPLAY` 결과에
    
    ```sql
    MERGE JOIN
      SORT JOIN
      SORT JOIN
    ```
    

       과 같은 단계가 보이면 소트 머지 조인이 성공적으로 선택된 것입니다.

1. 실제 업무 환경에서는 정렬 단계가 PGA에서 메모리만으로 끝나는지, Temp Tablespace로 내려갔는지(Wait Event: `direct path write temp`)를 함께 관찰하여 튜닝합니다.

### 4.3 해시 조인 (Hash Join)

- 명단을 ‘해시함수’ 라는 특별한 방법으로 요약해서 찾는 방식 . 사물함 번호처럼 하나의 키로 빠르게 찾는 방법입니다.
- 아주 큰 명단(수만 명)도 빠르게 비교 가능, 단, 해시 공간을 만들 공간이 필요합니다.

### **📌**  개념 설명

> 명단을 사물함처럼 정리해서 찾는 방식이에요. 한쪽 표를 미리 ‘요약 정리’(해시)해두고, 다른 쪽 표에서 값을 던져 넣어 바로 찾아냅니다.
> 

### **📌**  빠르게 이해하는 한 줄 정리

> 해시 조인은 “한쪽을 해시 표로 만들어 두고, 다른 쪽을 열쇠로 톡톡 두드려 찾는 방식” 입니다.
양쪽 테이블이 모두 크거나 인덱스가 부족할 때 가장 강력하며, 메모리를 넉넉히 주면 더욱 빨라집니다.
> 

| 소목차 | 핵심 정리 |
| --- | --- |
| **4.3.1 기본 메커니즘** | 한 테이블을 메모리(또는 임시 파일)에 **해시 테이블**로 만든 뒤, 다른 테이블의 키 값을 넣어 매칭합니다.
하나의 테이블을 메모리(PGA)에 올려 **해시 테이블**로 변환( Build ) → 다른 테이블의 각 행을 해시 함수에 넣어 값이 같은지 검사( Probe ) → 일치하면 결과로 출력. |
| **4.3.2 빠른 이유** | 해시 함수가 거의 O(1)에 가까운 속도로 일치 여부를 확인해 줍니다.
해시 함수는 “주소록에서 사람 이름 첫 글자를 알파벳 숫자로 바로 치고 들어가는” 것처럼 **O(1)** 수준의 탐색 속도를 제공합니다. 그래서 두 테이블이 모두 대량이어도 탐색 횟수가 크게 늘어나지 않습니다. |
| **4.3.3 대용량 Build Input** | 메모리가 부족하면 해시 파티션을 나눠서 반복 처리합니다(멀티 패스).
Build 단계에서 테이블이 메모리에 다 담기지 않을 경우, Oracle은 내부적으로 **해시 파티션**을 나누고 여러 번(멀티 패스) 반복하여 처리합니다. |
| **4.3.4 실행계획 제어** | `USE_HASH( )` 힌트를 이용합니다.
힌트 `USE_HASH(A B)` 또는 `HASH_JOIN`을 지정하면 옵티마이저가 해시 조인을 사용합니다. |
| **4.3.5 조인 메소드 선택 기준** | (1) 테이블 크기, (2) 인덱스 유무, (3) 가용 메모리량이 결정 요소입니다.
 ‣ **테이블 크기**: 둘 다 크면 해시 조인이 유리.
 ‣ **인덱스 유무**: 인덱스가 없다면 NL 보다 해시 유리. 
‣ **가용 메모리(PGA)**: Build용 테이블이 메모리에 들어가야 가장 빠릅니다. |

```sql
/* 해시 조인 실습 */
EXPLAIN PLAN FOR
SELECT /*+ USE_HASH(s sc) */ s.name, sc.score
FROM   students s
JOIN   scores   sc
ON     s.student_id = sc.student_id;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

### 예제 해설

1. `USE_HASH(s sc)` 힌트가 `students`(s)·`scores`(sc)를 해시 조인 방식으로 묶도록 지시합니다.
2. `DBMS_XPLAN.DISPLAY` 결과에
    
    ```sql
    HASH JOIN
      TABLE ACCESS...
      TABLE ACCESS...
    ```
    
    구문이 나타나면 해시 조인이 적용된 것입니다.
    
3. 결과 옆에 `Hash Cond`(조인 조건)와 **Build**·**Probe** 역할이 표기되어 Build 단계가 어느 테이블에 할당되었는지 알 수 있습니다.
4. 대량 데이터 환경에서 속도를 더 높이려면 PGA 크기를 늘려 Build 테이블이 메모리에 완전히 적재되도록 설정합니다(`PGA_AGGREGATE_TARGET`).

### 4.4 서브쿼리 조인

- 표 안에 있는 또 다른 작은 질문(서브쿼리)을 먼저 풀고, 그 결과를 다시 원래 질문과 합치는 방식입니다.
- 예 : “시험을 잘 본 사람들만 따로 모아서, 그 사람들의 출결 정보를 다시 확인” 같은 두 단계 요청에 따라 처리 합니다.

### **📌** 개념 설명

> “성적이 90점 넘는 학생만 골라서 그들의 이름을 알려줘” 처럼 두 번에 걸쳐서 정보를 꺼내 오는 방식입니다.
> 

### **📌**  빠르게 이해하는 한 줄 정리

> 서브쿼리도 결국 “조인으로 풀어 써야” 빠르다. 
옵티마이저가 이를 자동으로 해 주도록 EXISTS·IN·뷰 머지를 적극 활용하고, 스칼라 서브쿼리는 인덱스로 보강한다.
> 

| 소목차 | 핵심 정리 | 짧은 실습 예제 |
| --- | --- | --- |
| **4.4.1 변환 필요성** | 서브쿼리를 미리 펼치면 옵티마이저가 더 좋은 조인 계획을 세울 수 있습니다.

옵티마이저가 서브쿼리를 **조인 형태**로 바꾸면 실행계획이 단순해지고, 불필요한 반복 검색이 줄어 속도가 향상됩니다. | sql -- 90점 이상 학생이 있는지 검사 (EXISTS 변환) 
SELECT name FROM students s WHERE EXISTS ( SELECT 1 FROM scores sc WHERE sc.student_id = s.student_id AND sc.score >= 90 ); |
| **4.4.2 서브쿼리와 조인** | • `IN (서브쿼리)` → **반정규(ANTI) 조인**으로 변환 가능. 
• `EXISTS` → **세미 조인**으로 변환 가능. 각각 “포함 여부”만 확인하므로, 결과 집합을 줄이는 데 효과적입니다. | sql -- IN 서브쿼리 : 60점 미만 학생 추출 
SELECT name FROM students WHERE student_id IN ( SELECT student_id FROM scores WHERE score < 60 ); |
| **4.4.3 뷰(View)와 조인** | 뷰는 일종의 **포장지**일 뿐입니다. 옵티마이저는 필요할 때 뷰 내부로 들어가(뷰 머지) 원본 테이블과 직접 조인하여 더 빠른 계획을 세웁니다. | sql -- 단순 뷰 생성 및 조인 
CREATE OR REPLACE VIEW high_score_v AS SELECT student_id FROM scores WHERE score >= 90; SELECT [s.name](http://s.name/) FROM students s, high_score_v v WHERE s.student_id = v.student_id; /* 실행계획에서 VIEW MERGE 확인 */ |
| **4.4.4 스칼라 서브쿼리 조인** | 서브쿼리가 하나의 값만 반환할 때, NL과 유사하게 반복 계산될 수 있으므로 주의합니다.

스칼라 서브쿼리는 **한 행(한 값)** 만 반환합니다. NL 조인처럼 바깥 행마다 서브쿼리가 반복 실행될 수 있으므로, 결과가 크지 않을 때만 사용하거나 인덱스를 준비해야 합니다. | sql -- 각 학생의 최고 점수(스칼라 서브쿼리) 
SELECT name, ( SELECT MAX(score) FROM scores sc WHERE sc.student_id = s.student_id ) AS max_score FROM students s; |

```sql
/* 서브쿼리 조인 실습: 90점 이상 학생 찾기 */
SELECT name
FROM   students
WHERE  student_id IN (
        SELECT student_id
        FROM   scores
        WHERE  score >= 90
      );
```

### **📌** 해설 & 권장 튜닝 팁

1. **가능하면 EXISTS / IN** 이 옵티마이저가 가장 쉽게 조인 형태로 바꾸어 주는 서브쿼리 구조입니다.
2. **뷰는 성능을 느리게 하지 않습니다.** 다만 뷰 안에 불필요한 정렬·그룹화가 들어 있으면 뷰 머지 시에도 그대로 남으므로, 뷰 정의를 단순하게 유지하십시오.
3. **스칼라 서브쿼리는 인덱스 필수**입니다. 내부 테이블(`scores`)의 `student_id` 인덱스가 없다면 바깥 학생 수(예: 수천 명) × 점수 테이블 재검색이 일어나 속도가 급격히 저하됩니다.
4. 성능이 의심될 때는 `EXPLAIN PLAN`으로 **SEMI JOIN / ANTI JOIN / VIEW MERGE** 문구가 실제로 나타나는지 확인하고, 없다면 힌트(`MERGE`, `SEMIJOIN` 등)를 검토하십시오.

# 5장. 소트 튜닝

### **📌**  전체 요약 문장

> 정렬은 비싼 작업이다. 정렬을 없애거나(5.2), 미리 순서를 맞추고(5.3), 남은 데이터를 작게 만들어(5.4) 메모리 안에서 끝내는 것이 소트 튜닝의 핵심입니다.
> 

### 5.1 소트 연산에 대한 이해

- **설명** : '키 큰 순서', '이름 순서' 처럼 뭔가를 순서대로 정리하는 일. 엑셀에서 '정렬' 누르는 거랑 똑같습니다.

| 소목차 | 핵심 정리 |
| --- | --- |
| **5.1.1 소트 수행 과정** | 메모리(PGA)에 올려 정렬 → 메모리 초과 시 Temp Tablespace로 내려쓰기. |
| **5.1.2 소트 오퍼레이션** | `ORDER BY`, `GROUP BY`, `DISTINCT`, 윈도우 함수( =분석 함수)등에서 발생합니다. |
1. **소트 수행 흐름**
    1. 우선 서버 프로세스는 자신의 작업-메모리(PGA)에 데이터를 올려 **메모리 정렬**을 시도합니다.
    2. 행이 많아 PGA가 모자라면, 임시 디스크 공간(Temp Tablespace)에 내려보내고 다시 합칩니다.
    3. 최종적으로 정렬이 끝난 결과만 클라이언트에게 전달됩니다.
2. **실행계획에서 보이는 소트 연산자**
    - `SORT ORDER BY` → `ORDER BY` 절 수행
    - `SORT GROUP BY` → `GROUP BY` 절 수행
    - `SORT UNIQUE - NOSORT` 등 → 중복값 제거

> 핵심 : 정렬은 메모리를 많이 먹고, 부족하면 디스크 I/O(읽기·쓰기)가 급격히 늘어난다.
> 

```sql
/* 단순 소트 예제 */
SELECT name, score
FROM   students s JOIN scores sc ON s.student_id = sc.student_id
ORDER  BY score DESC;
```

### 5.2 소트가 발생하지 않도록 SQL 작성

**📌 개념 설명**

> UNION은 중복 제거를 해서 시간이 오래 걸리지만, UNION ALL은 중복 제거를 안 해서 빠릅니다.
> 

| 소목차 | 핵심 정리 |
| --- | --- |
| **5.2.1 Union vs. Union All** | `UNION`은 중복 제거를 위해 추가 정렬, `UNION ALL`은 정렬 없음. |
| **5.2.2 Exists 활용** | `EXISTS`는 조건 만족 여부만 확인하므로 불필요한 정렬·중복 제거가 없습니다. |
1. **UNION ALL 사용**
    - `UNION`은 두 결과를 합친 뒤 **중복을 제거**하기 때문에 정렬이 필수입니다.
    - `UNION ALL`은 단순 합치기이므로 정렬 단계를 건너뜁니다.
2. **EXISTS / IN 최적 사용**
    - `EXISTS`는 “찾았다!” 순간 바로 멈추므로 추가 정렬이 생길 가능성이 낮습니다.
    - `IN (서브쿼리)` 도 옵티마이저가 내부적으로 **세미 조인**으로 바꾸면 정렬을 피할 수 있습니다.

> 핵심 : 처음부터 정렬을 요구하지 않는 SQL 구조가 가장 빠르다.
> 

```sql
/* UNION ALL 실습: 중복 제거 없이 빠르게 합치기 */
SELECT student_id FROM scores WHERE score > 90
UNION ALL
SELECT student_id FROM scores WHERE score < 60;
```

### 5.3 인덱스를 이용한 소트 연산 생략

**📌 개념 설명**

> 이미 정렬되어 있는 명단(=인덱스)이 있다면 다시 정렬할 필요가 없습니다.
> 

| 소목차 | 핵심 정리 |
| --- | --- |
| **5.3.1 Sort Order By 생략** | 정렬이 필요한 열이 **인덱스의 선두 컬럼**이면 정렬을 건너뜁니다. |
| **5.3.2 Top N 쿼리** | “앞에서 몇 줄만” 요구하면 인덱스로 바로 가져올 수 있습니다. |
| **5.3.3~5.3.5** | 최소·최대값, 이력 조회 등에서도 인덱스로 소트를 줄일 수 있습니다. |
1. **ORDER BY와 인덱스 일치**
    - 정렬 컬럼과 순서가 **B-Tree 인덱스**와 동일하다면, DB는 인덱스 순서대로 읽고 끝냅니다.
        
        예) `ORDER BY student_id` → `CREATE INDEX idx ON students(student_id)`.
        
2. **Top N 쿼리(상위 N건)**
    - `FETCH FIRST 10 ROWS ONLY` 또는 Oracle의 `ROWNUM <= 10` 과 선행 인덱스를 결합하면,
        
        전 행을 정렬하지 않고 10행까지만 탐색합니다.
        
3. **최소·최대값 즉시 검색**
    - 인덱스 선두 컬럼이면 `MIN`·`MAX` 함수가 첫 / 끝 리프 블록만 조회하므로 정렬이 없습니다.
4. **최근 일시 DESC 인덱스**
    - 내림차순( `DESC` ) 컬럼으로 인덱스를 만들면 최신 데이터 조회 시 따로 `ORDER BY … DESC`를 실행 계획에 올리지 않습니다.
5. **GROUP BY 인덱스 생략**
    - 카디널리티(동일값 분포)가 낮은 컬럼은 **인덱스 스킵 스캔**으로 바로 그룹핑이 가능할 때가 있습니다.
    

> 핵심 : “이미 정렬된 길” — 인덱스 — 를 깔아 두면 SORT 단계가 사라진다.
> 

```sql
/* 인덱스로 Top N 소트 생략 실습 */
-- (앞서 만든 idx_scores_student_id는 student_id용이므로 score 인덱스를 따로 생성)
CREATE INDEX idx_scores_score ON scores(score);

SELECT /*+ INDEX(sc idx_scores_score) */ score
FROM   scores sc
ORDER  BY score DESC
FETCH  FIRST 2 ROWS ONLY;
```

### 5.4 Sort Area를 적게 사용하도록 SQL 작성

**📌 개념 설명**

> 메모리가 부족하면 정렬 중에 잠깐 데이터를 바닥에 내려놓고 다시 불러야 합니다. 
그러면 느려지니까 미리 정렬 데이터를 줄이거나 메모리를 늘려야 합니다.
> 

| 소목차 | 핵심 정리 |
| --- | --- |
| **5.4.1 소트 데이터 줄이기** | WHERE 절로 대상 행을 먼저 줄이면 정렬 메모리를 덜 씁니다. |
| **5.4.2 Top N 소트 부하 경감** | `FETCH FIRST N ROWS` 구문으로 정렬 범위를 제한합니다. |
| **5.4.4 분석 함수 Top N** | 윈도 함수 내에서도 파티션을 잘 나누면 필요 메모리가 줄어듭니다. |
1. **행 수와 컬럼 수 최소화**
    - 먼저 `WHERE` 절로 범위를 좁히고 필요 없는 컬럼은 SELECT 목록에서 제거한 뒤 정렬합니다.
    - 같은 1 만 행이라도 두 컬럼만 가져오면 네 컬럼보다 쓰는 메모리가 절반입니다.
2. **Top N 정렬 원리**
    - 대부분의 DB 엔진은 상위 N건만 필요하다고 알면, 정렬 버퍼를 N건만큼만 유지하다 넘치는 행을 바로 버립니다.
3. **Top N이 아닐 때의 부하**
    - `ORDER BY`만 있고 제한 절이 없으면, 엔진은 **모든 행**을 한꺼번에 정렬해야 합니다 → Temp I/O 급증.
4. **분석 함수(윈도 함수) 최적화**
    - `ROW_NUMBER() OVER (PARTITION BY dept ORDER BY score DESC)` 같이 파티션 범위를 좁히면,
        
        부서별 소량 정렬로 전체 정렬 부하를 분산할 수 있습니다.
        

> **핵심:** 정렬 대상 데이터를 최소화하고, 필요하면 상위 N 건으로 제한하여 메모리 사용을 절감하라.
> 

```sql
/* WHERE로 정렬 대상 축소 실습 */
SELECT name, score
FROM   students s
JOIN   scores   sc ON s.student_id = sc.student_id
WHERE  score >= 90
ORDER  BY score DESC;
```

## **✅** 마무리

- **조인 튜닝**은 “어떤 두 테이블을 어떤 순서·방식으로 이어 붙이면 가장 빠른가”를,
- **소트 튜닝**은 “필요한 정렬만 최소 비용으로 수행하려면 어떻게 작성해야 하는가”를 다룹니다.
