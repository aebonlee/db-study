# Day04 - AWR 분석 보고서 예시

## 💼 실습 주제: 비효율 SQL 튜닝 및 AWR 리포트 분석

### 1. 실습 개요

- **목표**: 비효율적인 SQL 실행을 유도하고, AWR 스냅샷 비교를 통해 튜닝 전후의 성능 차이를 분석한다.
- **환경**: Oracle 23c Free Edition / SQL Developer

---

### 2. 실험 SQL

```sql
SELECT * FROM orders WHERE TO_CHAR(order_date, 'YYYY-MM') = '2020-05';
```

- 문제점: `TO_CHAR()` 함수로 인해 **인덱스 사용 불가**, Full Scan 유발
- 데이터량: 100만 건 이상
- AWR 스냅샷 시점:
    - Snapshot ID: 231 → 232 (튜닝 전)
    - Snapshot ID: 233 → 234 (튜닝 후)

## DBMS_XPLAN 출력 스냅샷

```sql
-- 튜닝 전
EXPLAIN PLAN FOR
SELECT * FROM orders WHERE TO_CHAR(order_date, 'YYYY-MM') = '2020-05';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 튜닝 후
EXPLAIN PLAN FOR
SELECT * FROM orders
WHERE order_date BETWEEN TO_DATE('2020-05-01', 'YYYY-MM-DD')
                     AND TO_DATE('2020-05-31', 'YYYY-MM-DD');
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

예상 결과 스냅샷은 다음과 같습니다:

```
| Id | Operation                   | Name                | Rows  | Cost  |
|----|-----------------------------|---------------------|-------|-------|
|  0 | SELECT STATEMENT            |                     |       | 3000  |
|  1 | TABLE ACCESS FULL           | ORDERS              | 1M    | 3000  |
```

vs.

```
| Id | Operation                   | Name                | Rows  | Cost  |
|----|-----------------------------|---------------------|-------|-------|
|  0 | SELECT STATEMENT            |                     |       | 50    |
|  1 | TABLE ACCESS BY ROWID       | ORDERS              | 10000 | 50    |
|  2 | INDEX RANGE SCAN            | IDX_ORDERS_DATE     | 10000 | 20    |
```

---

### 3. AWR 리포트 주요 항목 비교

| 항목 | 튜닝 전 (231~232) | 튜닝 후 (233~234) | 개선 효과 |
| --- | --- | --- | --- |
| Elapsed Time (Top SQL) | 345초 | 37초 | 약 89% 개선 |
| Logical Reads | 120만 | 210만 | 증가 (버퍼캐시 사용 향상) |
| Physical Reads | 103만 | 9천 | 급감 |
| Top Wait Event | `db file scattered read` | `db file sequential read` | 랜덤 I/O 감소 |
| Buffer Cache Hit Ratio | 78% | 96% | 향상됨 |

---

### 4. 튜닝 SQL

```sql
SELECT * FROM orders
WHERE order_date BETWEEN TO_DATE('2020-05-01', 'YYYY-MM-DD')
                     AND TO_DATE('2020-05-31', 'YYYY-MM-DD');
```

- 인덱스 활용 가능 (`order_date` 인덱스 생성)
- 실행계획: INDEX RANGE SCAN → TABLE ACCESS BY ROWID

---

### 5. 종합 분석

- ✅ 비효율 SQL에서 발생하던 Full Scan은 물리적 I/O 병목 원인
- ✅ 클러스터링 팩터가 개선되어 Random I/O도 크게 감소함
- ✅ AWR 리포트의 Top Timed Event, Logical/Physical Reads, Top SQL 등을 종합적으로 비교해야 튜닝 효과가 정량적으로 입증됨

---

### 6. 개선 제안

| 항목 | 제안 내용 |
| --- | --- |
| SQL 작성 가이드 | WHERE 절에 함수 사용 금지, 인덱스 컬럼 그대로 비교 |
| 모니터링 | AWR + ADDM + V$SQLSTATS 등 통합적 활용 |
| 튜닝 자동화 | SQL Plan Baseline 또는 SQL Tuning Advisor 사용 고려 |

## ✅ [0] 실습 준비 – 테이블 생성 및 데이터 삽입

```sql
DROP TABLE orders PURGE;

CREATE TABLE orders AS
SELECT level AS order_id,
       MOD(level, 1000) AS customer_id,
       TO_DATE('2020-01-01', 'YYYY-MM-DD') + MOD(level, 365) AS order_date,
       DBMS_RANDOM.VALUE(100, 5000) AS amount
FROM dual CONNECT BY level <= 1000000;

-- 인덱스는 후반에 생성
```

## ✅ [2] 비효율 SQL 다중 실행

```sql
-- TO_CHAR 사용으로 인덱스 무시

  FOR i IN 1..10000 LOOP
    EXECUTE IMMEDIATE q'[SELECT * FROM orders WHERE TO_CHAR(order_date, 'YYYY-MM') = '2020-05']';
  END LOOP;
END;

```

## 📌 참고사항

- AWR이 작동하려면 `STATISTICS_LEVEL = TYPICAL` 이상이어야 하며, Oracle EE에서만 사용 가능
- SQL 실행 시 실제 성능 차이가 발생하도록 충분한 데이터량과 반복횟수 필요
- `Elapsed Time`, `Disk Reads`, `Buffer Gets`, `Wait Event` 분석 필요
