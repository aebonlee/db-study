# 3교시 - 비효율 SQL 탐지 전략 – V$SQL 및 AWR 기반 상위 부하 SQL 분석

### 🎯 학습 목표

- 성능 저하의 주요 원인이 되는 **비효율 SQL**을 Oracle 성능 뷰에서 추출하는 방법을 이해한다.
- `V$SQL`, `V$SQLAREA`, `V$ACTIVE_SESSION_HISTORY`, `AWR` 등을 통해 **Top SQL**을 탐지할 수 있다.
- SQL별 **Elapsed Time, CPU Time, Buffer Gets, Disk Reads, Executions** 등 주요 성능 지표를 분석하여 병목 원인을 식별할 수 있다.
- 수치 기반 진단을 통해 SQL 튜닝 우선순위를 설정할 수 있다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “실무에서 SQL 튜닝을 요청받았을 때, 어디서부터 시작하시겠습니까?
    
    튜닝은 실행계획을 보는 것보다 먼저, **무엇이 문제인지 진단하는 과정**부터 시작해야 합니다.
    
    오늘은 V$SQL, V$SQLAREA, 그리고 AWR 리포트를 활용하여 **가장 자원을 많이 사용하는 SQL**을 찾아내고,
    
    해당 SQL의 어떤 부분이 병목인지를 진단하는 방법을 배웁니다.”
    

---

1. **[비효율 SQL 탐지 기준 설명 – 10분]**
    
    ▶ 대표 성능 지표:
    
    | 항목 | 설명 | 의미 |
    | --- | --- | --- |
    | **ELAPSED_TIME** | 총 수행 시간 | 성능 문제 SQL 여부 |
    | **CPU_TIME** | CPU 자원 사용량 | 비효율 로직 여부 |
    | **BUFFER_GETS** | 메모리 접근량 | 캐시 효율성 |
    | **DISK_READS** | 물리적 디스크 접근 | I/O 병목 여부 |
    | **EXECUTIONS** | 실행 횟수 | 반복 발생 여부 |
    | **ROWS_PROCESSED** | 결과 행 수 | 효율 판단 근거 |
    
    ➤ 위 지표들을 종합적으로 비교하면, ‘가장 부담이 큰 SQL’을 찾을 수 있음
    

---

1. **[실습 – V$SQL, V$SQLAREA 활용한 Top SQL 조회 – 15분]**
    
    🔸 Top 10 SQL (수행시간 기준)
    
    ```sql
    SELECT *
    FROM (
      SELECT sql_id, executions, elapsed_time, sql_text
      FROM v$sql
      ORDER BY elapsed_time DESC
    )
    WHERE ROWNUM <= 10;
    ```
    
    🔸 CPU 자원 기준
    
    ```sql
    SELECT sql_id, cpu_time, buffer_gets, sql_text
    FROM v$sql
    WHERE ROWNUM <= 10
    ORDER BY cpu_time DESC;
    ```
    
    🔸 공유 실패 여부 확인
    
    ```sql
    SELECT sql_id, child_number, parse_calls, executions
    FROM v$sql
    WHERE parse_calls > executions;
    ```
    
    ➤ SQL_ID로 `DBMS_XPLAN.DISPLAY_CURSOR(sql_id)` 조회하여 실행계획 확인
    

---

1. **[AWR 리포트를 통한 Top SQL 분석 – 10분]**
    - AWR 리포트에서 확인 가능한 SQL 관련 항목:
        
        
        | 항목 | 의미 |
        | --- | --- |
        | SQL ordered by Elapsed Time | 총 수행 시간 높은 SQL |
        | SQL ordered by CPU Time | CPU 사용량 높은 SQL |
        | SQL ordered by Gets | 메모리 읽기량 높은 SQL |
        | SQL ordered by Executions | 자주 호출된 SQL |
        | SQL ordered by Disk Reads | 디스크 I/O 많은 SQL |
    - 📌 SQL_ID 확인 후 아래 쿼리로 직접 비교
        
        ```sql
        SELECT * FROM table(DBMS_XPLAN.DISPLAY_AWR('SQL_ID'));
        ```
        
    
    ➤ SQL이 자원을 많이 쓰는 원인이 무엇인지 수치 기반으로 판단 가능
    

---

1. **[튜닝 우선순위 설정 전략 – 5분]**
    - 총 수행 시간이 길고, 실행 횟수가 적으면 **쿼리 구조가 비효율적**
    - 실행 횟수가 많고, 총 수행 시간은 짧으면 **재사용 및 공유 실패 가능성**
    - Buffer Gets가 많고 Disk Reads가 적으면 **메모리 낭비**
    - Disk Reads가 많으면 **인덱스 부재 또는 통계 미정확성** 가능성
    
    ➤ 가장 튜닝 효과가 클 SQL부터 선별하여 우선 대응
    

---

1. **[정리 및 마무리 – 5분]**
    - 모든 튜닝은 ‘증상 기반’이 아닌 ‘데이터 기반’으로 접근해야 한다
    - 비효율 SQL은 증상보다 숫자로 먼저 확인해야 한다
    - “SQL 튜닝은 해석이 아니라 탐지에서 시작된다.
        
        튜닝은 실행계획이 아니라, **자원 소비 패턴 분석**에서 출발한다.”
        

---

### 📝 과제

1. V$SQL에서 ELAPSED_TIME 기준 상위 3개 SQL을 추출하고 실행계획 확인
2. BUFFER_GETS, DISK_READS 수치 기준으로 비효율이 예상되는 SQL을 식별
3. AWR 또는 V$SQL로 수집한 SQL 성능 지표를 기준으로 튜닝 우선순위표 작성
