# 6교시 - 서브쿼리 튜닝 전략 – IN / EXISTS / 인라인 뷰 / 스칼라 서브쿼리

### 🎯 학습 목표

- 서브쿼리 유형별(스칼라, 인라인, IN, EXISTS) 동작 방식을 구분할 수 있다.
- 옵티마이저가 서브쿼리를 실행계획에서 어떻게 변환(Transformation)하는지 이해한다.
- 비효율적인 서브쿼리 구조를 튜닝하고, EXISTS 또는 JOIN 방식으로 전환할 수 있다.
- 서브쿼리가 튜닝 포인트가 되는 이유와 성능 영향을 분석할 수 있다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “서브쿼리는 SQL을 간결하게 표현하는 데 매우 유용하지만, 그 구조에 따라 성능 차이가 극단적으로 벌어질 수 있습니다.
    
    특히 스칼라 서브쿼리나 IN절 안에서 중첩된 서브쿼리는 반복 수행되는 경우가 많아 CPU 부하를 일으키기도 합니다.
    
    오늘은 옵티마이저가 서브쿼리를 어떻게 해석하고, 우리가 어떤 방식으로 튜닝할 수 있는지를 실습을 통해 확인해 보겠습니다.”
    

---

1. **[서브쿼리 유형별 특징 설명 – 10분]**
    
    
    | 유형 | 설명 | 성능 특징 |
    | --- | --- | --- |
    | **IN 서브쿼리** | 결과 목록 중 포함 여부 확인 | 중첩 루프 시 느림 |
    | **EXISTS 서브쿼리** | 존재 여부 확인 | 조건 만족 시 중단 → 빠름 |
    | **스칼라 서브쿼리** | SELECT 절 또는 WHERE 절에 단일 값 반환 | 반복 호출 시 병목 |
    | **인라인 뷰** | FROM 절에 SELECT 삽입 | Materialized vs Merged 처리 |
    
    ➤ 옵티마이저는 내부적으로 `IN → JOIN`, `EXISTS → SEMI JOIN`, `스칼라 → 반복 LOOP`로 전환할 수 있음
    

---

1. **[실습 – IN vs EXISTS 성능 비교] – 15분**
    
    🔸 예제 SQL 1 (IN 사용)
    
    ```sql
    SELECT ENAME
    FROM EMP
    WHERE DEPTNO IN (SELECT DEPTNO FROM DEPT WHERE LOC = 'DALLAS');
    ```
    
    🔸 예제 SQL 2 (EXISTS 사용)
    
    ```sql
    SELECT E.ENAME
    FROM EMP E
    WHERE EXISTS (
      SELECT 1 FROM DEPT D WHERE D.DEPTNO = E.DEPTNO AND D.LOC = 'DALLAS'
    );
    ```
    
    🔸 실행계획 비교
    
    ```sql
    EXPLAIN PLAN FOR ... → DBMS_XPLAN.DISPLAY;
    ```
    
    ➤ 소량 서브쿼리는 IN이 더 빠른 경우도 있음
    
    ➤ 다수 Row가 관련된 경우 EXISTS가 효율적 (중단 수행 가능)
    

---

1. **[스칼라 서브쿼리 성능 이슈와 대체 전략 – 10분]**
    
    🔸 예제 SQL (비효율 구조)
    
    ```sql
    SELECT ENAME,
      (SELECT DNAME FROM DEPT WHERE DEPTNO = E.DEPTNO) AS DNAME
    FROM EMP E;
    ```
    
    - EMP가 10,000건이면, DEPT 테이블을 10,000번 접근 → **N+1 문제 발생**
    
    🔸 튜닝 SQL (JOIN으로 변경)
    
    ```sql
    SELECT E.ENAME, D.DNAME
    FROM EMP E JOIN DEPT D ON E.DEPTNO = D.DEPTNO;
    ```
    
    ➤ 실행계획에서 HASH JOIN 또는 NESTED LOOP로 병렬 처리 가능
    
    ➤ 단일 Row 반환 스칼라 서브쿼리는 JOIN으로 구조 변경하는 것이 일반적
    

---

1. **[인라인 뷰 튜닝 전략 – 5분]**
    - 인라인 뷰는 **FROM 절에 SELECT 절이 포함된 구조**
    - 옵티마이저는 뷰를 **머지(Merged)** 하거나 **물리적으로 Materialize(고정)** 처리할 수 있음
    - 불필요한 중첩을 피하고, 조건절을 최대한 바깥으로 끌어내는 것이 튜닝 포인트
    
    🔸 예시:
    
    ```sql
    SELECT *
    FROM (SELECT * FROM EMP WHERE SAL > 2000) E
    WHERE E.JOB = 'MANAGER';
    ```
    
    🔸 튜닝: 조건을 뷰 바깥으로 이동 가능
    

---

1. **[정리 및 마무리 – 5분]**
    - 서브쿼리는 작성 방식에 따라 실행 횟수에 큰 차이를 만든다
    - 옵티마이저의 변환 전략을 이해하면 불필요한 반복 수행을 피할 수 있다
    - “튜닝은 성능을 끌어올리는 것이 아니라, 낭비를 제거하는 것이다.
        
        특히 서브쿼리에서의 반복 I/O는 반드시 줄여야 할 낭비다.”

1. IN과 EXISTS를 사용하는 동일 SQL을 작성하여 실행계획과 COST 차이 비교

3. 인라인 뷰에 조건을 적용할 때 머지 전략과 머티리얼라이즈 전략이 성능에 미치는 영향 비교
