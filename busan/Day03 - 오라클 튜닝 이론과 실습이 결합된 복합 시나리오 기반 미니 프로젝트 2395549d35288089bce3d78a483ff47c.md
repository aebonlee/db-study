## ✅ 테이블 및 컬럼 정의 vs SQL문 사용 항목 비교

실습 문제에서 사용하는 컬럼들(`order_id`, `customer_id`, `order_date` 등)이 **모두 테이블 정의에 포함되어 있는지** 정확히 확인

### 1. `customer` 테이블

```sql
CREATE TABLE customer (
  id NUMBER PRIMARY KEY,               -- ✅ 사용됨 → customer_id로 JOIN
  name VARCHAR2(100),
  city VARCHAR2(50),
  join_date DATE
);
```

🔎 `customer_id`는 실제 테이블에서는 `id` → JOIN 시 `c.id = o.customer_id` 형식으로 처리됨

✅ **문제 없음**

---

### 2. `product` 테이블

```sql
CREATE TABLE product (
  id NUMBER PRIMARY KEY,               -- ✅ 사용됨 → product_id로 JOIN
  name VARCHAR2(100),
  price NUMBER                         -- ✅ 사용됨 (문제 2에서 조건: price > 10000)
);
```

✅ **문제 없음**

---

### 3. `orders` 테이블

```sql
CREATE TABLE orders (
  id NUMBER PRIMARY KEY,              -- ✅ `order_id`로 사용됨 → alias 필요
  customer_id NUMBER,                 -- ✅ 사용됨
  product_id NUMBER,                  -- ✅ 사용됨
  order_date DATE,                    -- ✅ 사용됨 (정렬 및 필터 조건)
  quantity NUMBER,                    -- ✅ 사용됨
  status VARCHAR2(20)
);
```

🔎 ✅ `id` 컬럼을 `order_id`로 SELECT에 쓰고 있음:

```sql
SELECT order_id, customer_id, order_date
FROM orders
ORDER BY order_date DESC;
```

→ **여기서는 다음과 같이 alias로 지정해야 정확합니다**:

```sql
SELECT id AS order_id, customer_id, order_date
FROM orders
```

✅ 따라서 문제 자체는 의미가 있지만, **컬럼명이 id인데 order_id로 착각하지 않도록 alias 명시가 필요**합니다.

---

### 4. `delivery` 테이블

```sql
CREATE TABLE delivery (
  id NUMBER PRIMARY KEY,
  order_id NUMBER,                     -- ✅ 사용됨
  delivery_date DATE,
  delivery_type VARCHAR2(20)
);
```

✅ `order_id` 존재함

## 🔍 결론: 잠재적 혼동 포인트

| 문제 | 문제점 | 수정 제안 |
| --- | --- | --- |
| orders 테이블 `order_id` 사용 | 실제는 `id` 컬럼 → `order_id` 아님 | `SELECT id AS order_id` 또는 문제에 컬럼명을 명확히 고지 |
| customer 테이블 `customer_id` | 실제는 `id` → JOIN 시 혼동 방지 필요 | `c.id AS customer_id`로 명시 추천 |

## 🛠 수정이 필요한 예시

```sql
-- 기존 (잠재적으로 혼동 유발)
SELECT order_id, customer_id, order_date FROM orders;

-- 수정된 정석 버전
SELECT id AS order_id, customer_id, order_date FROM orders;
```

## 🔍 실습 과제 예시 (난이도 증가 순)

### [Step 1] 느린 SQL 확인 및 분석

```sql
SELECT name FROM customer WHERE city = '서울';
```

- 실행계획 확인
- 인덱스 유무 확인
- 클러스터링 팩터 설명

### [Step 2] 인덱스 설계 및 힌트 테스트

```sql
CREATE INDEX idx_customer_city ON customer(city);
-- 힌트 실습
SELECT /*+ FULL(customer) */ * FROM customer WHERE city = '서울';
SELECT /*+ INDEX(customer idx_customer_city) */ * FROM customer WHERE city = '서울';
```

### [Step 3] 복합 Join 쿼리 성능 개선

```sql
SELECT c.name, p.name, o.quantity
FROM customer c
JOIN orders o ON c.id = o.customer_id
JOIN product p ON o.product_id = p.id
WHERE o.order_date BETWEEN TO_DATE('2023-01-01','YYYY-MM-DD') AND TO_DATE('2023-12-31','YYYY-MM-DD')
AND p.price > 10000;
```

- 실행 계획 비교: Nested Loop vs Hash Join
- 통계 수집 전/후 실행 계획 비교

### [Step 4] AWR 없이 V$SQL로 튜닝 SQL 찾기

```sql
SELECT sql_id, executions, buffer_gets, sql_text
FROM v$sql
WHERE buffer_gets > 10000
ORDER BY buffer_gets DESC;
```

-- 아래 쿼리에 힌트를 적용하여 성능 차이를 확인하고, 그 이유를 설명하시오.

SELECT c.name, o.order_date, p.name, o.quantity

## ✅ 미니 프로젝트 풀이

### 🔍 문제 1: 도시(city) 기준 고객 검색 성능 분석

### 도시 기준 검색 성능 비교

- `SELECT name FROM customer WHERE city = '서울';`
- 목표: 인덱스 유무에 따른 성능 차이 확인
- 수행 과제:
    - 실행계획 확인
    - 인덱스 생성 후 힌트 실험
    - 통계 수집 여부 테스트

**문제 설명:**

다음 SQL은 특정 도시(예: '서울')에 거주하는 고객 목록을 조회하는 쿼리입니다.

```sql
SELECT name FROM customer WHERE city = '서울';
```

**문제점:**

해당 쿼리는 `city` 컬럼에 인덱스가 존재하지 않으면 `TABLE ACCESS FULL` 방식으로 전체 테이블을 스캔하게 됩니다. 이로 인해 레코드 수가 많을 경우 성능이 급격히 저하됩니다.

**해결 방법:**

1. `city` 컬럼에 인덱스를 생성합니다.
    
    ```sql
    CREATE INDEX idx_customer_city ON customer(city);
    ```
    
2. 실행 계획을 다시 확인하여 `INDEX RANGE SCAN`으로 변경되었는지 확인합니다.
3. 힌트를 통해 명시적으로 인덱스를 사용하도록 지정해 테스트할 수 있습니다.
    
    ```sql
    SELECT /*+ INDEX(customer idx_customer_city) */ * FROM customer WHERE city = '서울';
    ```
    
4. 클러스터링 팩터가 나쁘면 인덱스를 사용하더라도 효과가 떨어질 수 있으며, 이 경우 테이블 재정렬이나 히스토그램 활용도 고려합니다.

### 🔍 문제 2: Join 쿼리 성능 개선

### 다중 테이블 조인 쿼리 튜닝

- 고객 + 주문 + 상품 조인 쿼리
- 목표: 조인 방식 및 순서 힌트 적용 성능 비교
- 수행 과제:
    - `USE_HASH`, `USE_NL`, `LEADING` 힌트 적용
    - 통계 수집 후 실행 계획 변화 분석
    - 인덱스 생성 전/후 성능 비교
    

**문제 설명:**

다음 SQL은 고객, 주문, 상품 테이블을 조인하여 2023년 동안 고가 상품을 구매한 내역을 조회하는 쿼리입니다.

```sql
SELECT c.name, p.name, o.quantity
FROM customer c
JOIN orders o ON c.id = o.customer_id
JOIN product p ON o.product_id = p.id
WHERE o.order_date BETWEEN TO_DATE('2023-01-01','YYYY-MM-DD') AND TO_DATE('2023-12-31','YYYY-MM-DD')
AND p.price > 10000;

```

**문제점:**

옵티마이저가 테이블의 통계 정보 없이 조인 순서를 잘못 판단하거나 비효율적인 접근 방법을 선택할 수 있습니다. 예를 들어 대용량 테이블에서 `NESTED LOOP JOIN`을 사용하는 경우 성능이 저하됩니다.

**해결 방법:**

1. 각 테이블에 대해 통계 수집을 먼저 수행합니다.
    
    ```sql
    EXEC DBMS_STATS.GATHER_TABLE_STATS('SCOTT', 'CUSTOMER');
    ```
    
2. 힌트를 사용하여 조인 순서 및 조인 방식을 제어합니다.
    
    예시: `/*+ LEADING(p o c) USE_HASH(o c) */`
    
3. `orders.order_date`, `product.price` 컬럼에 인덱스가 필요한지 확인하고, 분포도 등을 고려해 생성 여부를 결정합니다.
4. 실행 계획을 비교하면서 `HASH JOIN` 또는 `MERGE JOIN`으로 변경해 성능을 개선합니다.

### 🔍 문제 3: 고비용 SQL 식별 (V$SQL 활용)

### 고비용 SQL 식별 및 분석

- `V$SQL`, `DBMS_XPLAN.DISPLAY_CURSOR`, `BUFFER_GETS` 활용
- 수행 과제:
    - 상위 buffer_gets SQL 추출
    - 실행 횟수/응답 시간 분석
    - 고비용 SQL 개선 제안서 작성
    

**문제 설명:**

다음 SQL은 오라클의 동적 성능 뷰인 `V$SQL`을 활용하여, 버퍼 접근 횟수(`buffer_gets`)가 높은 SQL 문장을 식별하는 용도입니다.

```sql
SELECT sql_id, executions, buffer_gets, sql_text
FROM v$sql
WHERE buffer_gets > 10000
ORDER BY buffer_gets DESC;
```

**문제점:**

DB에 비효율적인 SQL이 있을 경우 전체 시스템 성능 저하로 이어지며, 이들을 정확히 식별해야 튜닝 포인트를 찾을 수 있습니다.

**해설 및 활용:**

- `buffer_gets`는 논리적 I/O를 의미하며, SQL이 수행되면서 얼마나 많은 블록을 읽었는지를 나타냅니다.
- `executions` 대비 `buffer_gets`가 과도하게 높은 경우 비효율적인 SQL입니다.
- 튜닝 대상 SQL을 파악한 뒤, 해당 SQL의 실행계획(`V$SQL_PLAN`)을 함께 조회하여 개선 방향을 탐색할 수 있습니다.

예를 들어 다음과 같이 특정 SQL의 실행 계획도 확인할 수 있습니다.

```sql
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR('sql_id값', NULL, 'ALLSTATS LAST'));
```
