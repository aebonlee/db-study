# 2교시 - 옵티마이저 통계정보의 역할과 수집 전략 실습

### 🎯 학습 목표

- 옵티마이저가 통계정보를 통해 실행계획을 결정하는 원리를 이해한다.
- 테이블, 인덱스, 컬럼 통계정보의 구성 요소를 구분할 수 있다.
- 통계정보가 부정확할 때 실행계획이 왜곡되는 현상을 실습을 통해 파악할 수 있다.
- `DBMS_STATS` 패키지를 활용하여 통계정보를 수집, 갱신, 확인하는 방법을 익힌다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “우리가 지금까지 실행계획과 조인 방식, 인덱스 전략 등을 배워왔지만, 사실 그 모든 판단의 기반은 바로 ‘통계정보’입니다.
    
    옵티마이저는 마법사가 아닙니다. 테이블에 대한 크기, 컬럼 값 분포, 인덱스 유무 등 수치 기반의 정보를 토대로 판단합니다.
    
    오늘은 그 핵심인 통계정보에 대해 학습하고, 이를 정확하게 수집하고 관리하는 방법을 실습합니다.”
    

---

1. **[통계정보의 개념과 구성 설명 – 10분]**
    - 옵티마이저는 Cost-Based 방식으로 동작 → 모든 판단은 통계에 기반
    - 통계정보 구성:
        - **테이블 통계**: NUM_ROWS, BLOCKS, AVG_ROW_LEN 등
        - **컬럼 통계**: NDV(고유값 수), 히스토그램(Histogram)
        - **인덱스 통계**: LEAF_BLOCKS, CLUSTERING_FACTOR 등
    - 통계정보가 오래되거나 부정확하면?
        - 옵티마이저가 잘못된 실행계획 선택
        - 예: NLJ 대신 HASH JOIN 선택, 인덱스 무시 등

---

1. **[실습 – 통계정보 확인 및 수집 – 20분]**
    
    🔸 통계 확인
    
    ```sql
    
    SELECT table_name, num_rows, blocks, avg_row_len
    FROM user_tables
    WHERE table_name = 'SALES';
    
    SELECT column_name, num_distinct, density
    FROM user_tab_col_statistics
    WHERE table_name = 'SALES';
    
    SELECT index_name, clustering_factor
    FROM user_indexes
    WHERE table_name = 'SALES';
    ```
    
    🔸 통계 수집
    
    ```sql
    
    BEGIN
      DBMS_STATS.GATHER_TABLE_STATS(
        ownname => USER,
        tabname => 'SALES',
        estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
        cascade => TRUE
      );
    END;
    /
    ```
    
    🔸 수집 후 실행계획 비교
    
    ```sql
    
    EXPLAIN PLAN FOR SELECT * FROM SALES WHERE CUST_ID = 100;
    SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
    ```
    
    ➤ 수집 전/후 실행계획의 COST 또는 INDEX 사용 여부 비교
    

---

1. **[히스토그램의 개념과 사용 전략 – 5분]**
    - 컬럼 값 분포가 불균형할 경우 히스토그램 생성 필요
    - NDV(고유값 수)가 적고, 특정 값의 비율이 높은 컬럼 → 히스토그램 유용
    - 히스토그램이 없으면 ‘평균 분포’ 가정 → 실행계획 오류 발생 가능
    
    ```sql
    
    BEGIN
      DBMS_STATS.GATHER_TABLE_STATS(
        ownname => USER,
        tabname => 'SALES',
        method_opt => 'FOR COLUMNS SIZE 10 REGION'
      );
    END;
    /
    ```
    

---

1. **[실행계획 왜곡 사례 소개 – 5분]**
    - 통계정보가 오래되었거나 존재하지 않을 때:
        - INDEX가 있음에도 Full Scan 발생
        - Join 순서가 비효율적으로 선택됨
        - 실행계획이 지속적으로 변경됨 → Cursor Sharing 실패
    
    ➤ 실제 업무 시스템에서는 **정기적인 통계 수집 작업(Job 또는 Scheduler)**이 중요
    

---

1. **[정리 및 마무리 – 5분]**
    - 통계는 옵티마이저의 눈이다. 눈이 흐리면 판단도 흐려진다.
    - 튜닝 이전에 통계정보부터 먼저 확인하는 습관이 필요하다.
    - “SQL 튜닝은 추론이 아니라 계산이다.
        
        옵티마이저가 계산할 수 있도록 데이터를 정확히 제공하는 것이 우리의 역할이다.”

### 📝 과제

1. SALES 테이블의 통계 수집 전후 실행계획 비교 리포트 작성
2. REGION 컬럼에 히스토그램을 생성하고 실행계획의 변화 관찰
3. 다음 SQL에 대해 통계 수집 없이 실행했을 때와 수집 후의 계획 차이 확인
    
    ```sql
    SELECT * FROM SALES WHERE REGION = '제주';
    ```
