# 2교시 - 인덱스 튜닝 심화 – 다중 컬럼 인덱스, 함수 기반 인덱스, 스킵 스캔 분석

### 🎯 학습 목표

- 다중 컬럼 인덱스의 구성 방식과 컬럼 순서가 성능에 미치는 영향을 이해한다.
- 함수 기반 인덱스를 사용해야 하는 경우와 사용 방법을 설명할 수 있다.
- 실행계획에서 Skip Scan 방식이 발생하는 조건을 확인하고, 이를 활용 또는 회피하는 전략을 실습한다.
- 실습을 통해 다양한 인덱스 유형이 SQL 성능에 미치는 영향을 체감할 수 있다.

---

### 🧑‍🏫 강의 흐름 및 설명 내용

1. **[도입 – 5분]**
    
    “여러분은 지금까지 인덱스를 단순히 하나의 컬럼에 걸어두는 구조로 이해해 오셨을 수 있습니다.
    
    그러나 실무에서는 대부분 ‘다중 컬럼 인덱스’를 사용하며, 조건절에 함수가 들어가거나, 일부 컬럼만 검색하는 상황도 빈번히 발생합니다.
    
    이번 시간에는 인덱스를 어떻게 설계하고 활용하느냐에 따라 SQL 성능이 얼마나 달라지는지를 구체적으로 실습하며 확인하겠습니다.”
    

---

1. **[다중 컬럼 인덱스의 순서 중요성 설명 – 10분]**
    - 인덱스는 ‘선형 구조’ → **선행 컬럼부터 조건이 걸려야 효율적 사용 가능**
        - 예: `CREATE INDEX IDX_EMP_JOB_DEPT ON EMP(JOB, DEPTNO);`
        - `WHERE JOB = 'CLERK'` → 사용됨
        - `WHERE DEPTNO = 10` → 사용되지 않음
        - `WHERE DEPTNO = 10 AND JOB = 'CLERK'` → 효율적 사용 (선행 컬럼 포함 시)
    - 옵티마이저는 가능할 경우 Skip Scan을 시도하지만, 성능 저하 위험 있음

---

1. **[함수 기반 인덱스(Function-Based Index) 설명 – 10분]**
    - 조건절에 함수가 포함되면 일반 인덱스는 무시됨
        
        예: `WHERE TO_CHAR(HIREDATE, 'YYYY') = '1981'`
        
    - 해결 방법:
        
        ```sql
        
        CREATE INDEX IDX_EMP_HIREDATE_YR ON EMP(TO_CHAR(HIREDATE, 'YYYY'));
        ```
        
    - 함수 기반 인덱스는 다음과 같은 상황에서 사용:
        - 날짜 형식 변환
        - 대소문자 통일 (`UPPER(NAME)`)
        - 수식이 포함된 조건절
    - 주의: 함수 기반 인덱스 사용 시 옵티마이저가 Index 사용을 인식해야 하므로 통계 수집 필요

---

1. **[Skip Scan 개념 및 실행계획 분석 – 10분]**
    - **Index Skip Scan**: 다중 컬럼 인덱스에서 선행 컬럼 값 없이 후행 컬럼으로 조건이 걸렸을 때 발생
        - 예: 인덱스 (JOB, DEPTNO)인데 `WHERE DEPTNO = 10`일 경우
    - Oracle은 JOB 컬럼의 **NDV(고유값 수)** 만큼 각 JOB별로 반복 스캔 발생
    - 효율 낮고, Random I/O 증가 → 주로 통계 기반으로 발생 여부 결정됨
    - 실행계획에서 다음과 같이 확인 가능:
        
        ```
        INDEX SKIP SCAN IDX_EMP_JOB_DEPT
        ```
        

---

1. **[실습 – 다양한 인덱스 유형 비교 및 실행계획 해석 – 15분]**
    
    🔸 Step 1: 인덱스 생성
    
    ```sql
    CREATE INDEX IDX_EMP_JOB_DEPT ON EMP(JOB, DEPTNO);
    CREATE INDEX IDX_EMP_HIREDATE_YR ON EMP(TO_CHAR(HIREDATE, 'YYYY'));
    ```
    
    🔸 Step 2: 실행계획 비교
    
    ```sql
    EXPLAIN PLAN FOR SELECT * FROM EMP WHERE TO_CHAR(HIREDATE, 'YYYY') = '1981';
    EXPLAIN PLAN FOR SELECT * FROM EMP WHERE DEPTNO = 10;
    EXPLAIN PLAN FOR SELECT * FROM EMP WHERE JOB = 'MANAGER';
    ```
    
    ➤ `INDEX RANGE SCAN`, `INDEX SKIP SCAN`, `FULL TABLE SCAN` 차이 해석
    
    ➤ 통계정보 미수집 시 옵티마이저 판단이 달라질 수 있음
    

---

1. **[정리 및 마무리 – 5분]**
    - 다중 컬럼 인덱스는 컬럼 순서가 가장 중요하다
    - 함수 기반 인덱스는 '함수 사용 자체'가 문제라기보다, 인덱스와 통계가 없을 때 문제가 된다
    - Index Skip Scan은 옵티마이저의 ‘임시 조치’로 성능이 불리할 수 있음
    - “좋은 인덱스란, 잘 쓰일 수 있는 구조를 가진 인덱스다.
        
        튜닝은 인덱스를 만드는 일이 아니라, **읽히게 만드는 일**이다.”

### 📝 과제

1. EMP 테이블에서 다중 컬럼 인덱스를 생성하고, 조건 순서를 바꿨을 때의 실행계획을 비교해 보기
2. TO_CHAR, UPPER 등 함수가 포함된 SQL에서 일반 인덱스와 함수 기반 인덱스를 비교
3. Skip Scan이 발생하는 조건을 일부러 만들어보고, 통계 수집 후 발생 여부 변화 확인
