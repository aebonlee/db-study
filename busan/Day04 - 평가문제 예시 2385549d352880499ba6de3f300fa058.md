## ✅ 객관식 문제 정답 및 해설

| 문항 | 정답 | 해설 |
| --- | --- | --- |
| 1 | ③ | B-Tree 인덱스는 Leaf Block에 인덱스 키값 + Rowid가 정렬된 형태로 저장됨. |
| 2 | ③ | AWR의 Top SQL 항목에는 일반적으로 **Elapsed Time, CPU Time, Buffer Gets, Disk Reads** 등이 포함되며 **Active Session Count**는 직접적인 항목이 아님. |
| 3 | ③ | 클러스터링 팩터가 높으면 인덱스를 이용해도 테이블에서 랜덤 I/O가 많이 발생해 오히려 Full Scan보다 비효율적일 수 있음. |
| 4 | ③ | `TO_CHAR(date_col, 'YYYYMM')`은 함수가 인덱스 컬럼에 적용되어 **인덱스를 사용할 수 없음**, Full Scan 유도. |
| 5 | ① | `db file sequential read`는 **인덱스를 통해 Random Access 하는 I/O 대기 이벤트**로, Disk I/O 병목 시 대표적으로 나타남. |

## ✅ 주관식 문제 예시 정답

**6. SQL 개선문**

```sql
-- 원래 SQL (비효율)
SELECT * FROM orders WHERE TO_CHAR(order_date, 'YYYY-MM') = '2020-05';

-- 개선 SQL
SELECT * FROM orders
WHERE order_date BETWEEN TO_DATE('2020-05-01', 'YYYY-MM-DD')
                     AND TO_DATE('2020-05-31', 'YYYY-MM-DD');
```

> ✅ 해설: 함수 제거 → 인덱스 사용 가능 → Full Scan 방지, Index Range Scan 유도
> 

---

**7. 클러스터링 팩터 설명**

> 클러스터링 팩터는 인덱스의 순서와 테이블에 실제 저장된 Row의 순서가 얼마나 일치하는지를 나타내는 지표입니다.
> 
> 
> 값이 작을수록 테이블이 인덱스 순서에 가깝게 정렬되어 있어서 Random I/O가 줄어들고 인덱스 성능이 좋아집니다.
> 

---

**8. LIKE 쿼리 분석**

```sql
SELECT * FROM customer WHERE name LIKE '%김%';
```

> 이 SQL은 % 와일드카드가 앞에 위치하기 때문에 Index를 사용할 수 없고 Full Scan이 발생합니다.
> 
> 
> 또한 문자열 검색의 특성상 결과셋이 클 경우 I/O 부하가 큽니다.
> 

---

**9. 파티셔닝의 성능 개선 효과**

1. **Partition Pruning**: WHERE 절 조건으로 불필요한 파티션을 제외 → I/O 감소
    
    예: `birth_year BETWEEN 1990 AND 1999`이면 해당 Range 파티션만 스캔함
    
2. **병렬 처리 가능**: 각 파티션 단위로 병렬 쿼리 실행이 가능하여 대용량 테이블 처리 성능 향상
    
    예: 파티션이 8개면 8개의 Parallel Process에서 동시에 처리 가능
    

---

**10. AWR 리포트 분석**

| 이벤트 | 의미 |
| --- | --- |
| `db file scattered read` | Full Table Scan이 자주 발생함을 의미. 원인은 인덱스 미사용 또는 범위 조건 미사용 |
| `db file sequential read` | Index Range Scan에 의한 Random I/O로, 튜닝되지 않은 SQL 다수 존재 가능 |
| `buffer busy waits` | 동일 블록을 여러 세션이 동시에 Access → Hot Block 발생 가능성 |
| `log file sync` | 잦은 COMMIT으로 인해 Redo Log File 기록 지연 발생 가능성 |

> ✅ 종합 개선 방향:
> 
- WHERE 절 함수 제거 및 인덱스 최적화
- 데이터 정렬 재조정 → 클러스터링 팩터 개선
- Hot Block 회피 → Hash Partition 도입 고려
- COMMIT 빈도 조절 또는 배치 처리
